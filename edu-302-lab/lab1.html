LAB 1

Scenario: Safemarch is deploying Zscaler services and needs organisation-specific departments and users provisioned in ZIdentity so that role-based policies can be enforced across ZIA, ZPA, and ZDX. You must also enable Zscaler Support Remote Assistance so the support team can observe your tenant during troubleshooting.

Time: ~30 minutes
Portal: ZIdentity Admin
💭 THINK FIRST · IDENTITY FOUNDATION

ZIdentity is the single control plane for authentication across ZIA, ZPA, and ZDX — departments and users here directly drive which policies fire and which access decisions are made.

  • Why does Zscaler require users to be provisioned in ZIdentity rather than relying solely on an external IdP like Azure AD or Okta?
    Q1 — MODEL ANSWER
    ZIdentity acts as the authoritative source of truth for Zscaler-specific attributes such as department membership, service entitlements, and admin roles. External IdPs authenticate the user identity, but ZIdentity maps that identity to Zscaler-specific policy groups. Without provisioned users, ZIA and ZPA cannot resolve which department-based policies to apply at the point of access.
  • What is the risk of enabling "Prompt for Password Change After Initial Log In" when setting passwords by administrator for multiple lab users?
    Q2 — MODEL ANSWER
    In a lab environment where the session password is shared, a forced password change after first login would immediately break all subsequent logins for that account — the lab steps that try to authenticate as finance-ptcb83@domain.com would fail with "wrong password." You disable the prompt so the lab password remains stable. In production, this setting exists for security; in a time-boxed lab pod it creates friction that prevents you from completing the exercise.
  • What would break in the overall Zscaler deployment if users were added to the wrong department — for example, if the marketing user was placed in IT Security?
    Q3 — MODEL ANSWER
    Department membership is the key attribute used to scope ZIA URL filtering rules, ZPA access policies, and DLP policies. A marketing user placed in IT Security would inherit IT Security department exemptions — such as access to eicar.org and SSH firewall allow rules. Conversely, they would lose the marketing-only exemption for LinkedIn posting. The misconfiguration would silently give over-permissive access without triggering any error; policies would still evaluate, just against the wrong department context.
Task 1.1 Create Departments and Users in ZIdentity ▾

1 Create the Safemarch Departments

  1. Log in to the ZIdentity Admin portal using your ZIA Admin account: ptcb83-ADMIN@thezerotrustexchange.com and Session Password <Session Password>.
  2. Navigate to Directory > Departments.
  3. Create each of the following departments:

    • Finance

Task 1.2 Configure Zscaler Support Remote Assistance ▾

1 Enable Remote Assistance

  1. In ZIdentity Admin, navigate to Settings > Remote Assistance.
  2. Enable Zscaler Support Remote Assistance with the following configuration:
    • Access Type: View-Only
    • Duration: 90 days
  3. Click Save.

Note: View-only access allows Zscaler support engineers to observe your portal's status and data during troubleshooting sessions without the ability to make configuration changes.

Verification: Confirm that the Remote Assistance status shows Active with your selected expiry date 90 days from today.
