Lab 3: Configure ZIA Policy

LAB 3

Scenario: Configure a comprehensive set of ZIA policies for Safemarch's initial deployment — SSL inspection, URL filtering, file type controls, Office 365 tenant isolation, and firewall rules. Rule order matters throughout: the first matching rule wins, so department-specific exceptions must always sit above the broader catch-all rules.

Time: ~60 minutes
Portal: ZIA Admin (https://admin.zscalerthree.net)
💭 THINK FIRST · ZIA POLICY DESIGN

A single misconfigured rule order can bypass your entire security posture — always think about rule precedence before placement.

  • Why must the SSL inspection rule for MS Login Services be placed HIGHER in order than the "Office 365 One Click" rule?
    Q1 — MODEL ANSWER
    Rules are evaluated top-to-bottom. The MS Login rule applies the Tenant Profile that restricts O365 logins to only the corporate Safemarch tenant. If the broader O365 One Click rule fires first, it bypasses SSL inspection for all Office 365 traffic — including the login flow — making it impossible for Zscaler to inject the Restrict-Access-To-Tenants HTTP header that enforces tenant isolation. The MS Login rule must run first to intercept the login traffic before the broader exemption takes over.
  • What is the difference between a "Caution" action and a "Block" action for executable file downloads — and when would you use Caution for IT Security but Block for everyone else?
    Q2 — MODEL ANSWER
    Caution displays a warning page the user can bypass after acknowledging the risk. Block denies access entirely. IT Security legitimately downloads executables (patch files, security tools, malware samples) — a hard Block would prevent their work. Caution warns them to be deliberate but allows it. For all other departments, executables represent an unacceptable malware delivery risk, so Block is appropriate. The IT Security rule must be higher in order than the Block rule so it fires first for that department.
  • What breaks if you forget to create the custom URL category for Safemarch-specific blocked URLs before referencing it in the URL filtering policy?
    Q3 — MODEL ANSWER
    ZIA will not allow you to save a URL filtering rule that references a URL category that doesn't exist. Create the custom category first — even with a placeholder URL — then reference it in the filtering rule. Later, the Safemarch security team can add off-limits URLs without changing the policy structure.

LAB 3 PROGRESS
QUICK REFERENCE ▲
Rule Order
First matching rule wins. Department exceptions must always sit above catch-all rules.
SSL Inspection
Zscaler decrypts HTTPS to inspect it. Certificate-pinned apps must be exempted or they break.
Tenant Restriction
Injects HTTP headers during O365 login to allow only the corporate tenant. Requires SSL inspection on the login flow — hence the MS Login rule above O365 One Click.
SafeSearch
Forces Google/Bing/Yahoo to return only safe results. Set via URL Filtering → Advanced Policy Settings.
Firewall vs URL Filter
URL filtering handles HTTP/HTTPS. SSH is layer-4 and needs a Firewall rule, not a URL filter.
