Lab 5: Configure ZPA Policy

LAB 5

Scenario: Provision the ZPA App Connector on the Safemarch corporate network, define Application Segments for all internal resources, and create access policies with department-level controls and idle timeout rules so only the right users reach the right apps.

Time: ~60 min
Portals: ZPA Admin Portal + Corp: App Connector VM
💭 THINK FIRST · ZPA POLICY DESIGN

ZPA works on a deny-by-default model — if there is no matching access policy rule, the user cannot reach the app. Every application you want accessible must have an explicit rule.

  • Why does Safemarch use a Client Forwarding Policy (bypass rule) for apps like erp.safemarch.com and crm.safemarch.com, and what does this mean for their security posture?
    Q1 — MODEL ANSWER
    A Client Forwarding Policy tells ZCC to NOT send traffic for those destinations through ZPA — the client sends it directly (or via ZIA). This is used for apps that: (1) have their own security controls and don't need ZPA's micro-segmentation, (2) are accessible only from inside the corporate network anyway, or (3) use protocols incompatible with ZPA. The security implication: these apps do NOT benefit from ZPA's per-request authentication and zero-trust policy. They rely on whatever perimeter security is in front of them.
  • Why is device posture required for SIEM server SSH access specifically, but not for Intranet or File Share access?
    Q2 — MODEL ANSWER
    The SIEM server contains security logs and forensic data — the highest-sensitivity target in the environment. Domain-joined device posture ensures only corporate-managed machines (not personal or contractor laptops) can SSH in. Intranet and file share are general-purpose resources where the risk of a non-domain-joined device is lower. Device posture is an additional verification layer on top of user identity — "who you are" + "what device you're using" = higher trust.
  • What happens if the idle timeout policy for HVAC (15 minutes) is not configured, and why does it matter specifically for that application?
    Q3 — MODEL ANSWER
    Without idle timeout, a ZPA connection to hvac.patraining.safemarch.com stays open indefinitely after the user stops interacting. HVAC/OT systems are frequently left connected by maintenance staff who then walk away from their workstations. An open, unattended ZPA connection to an industrial control system is a lateral movement risk — if the user's machine is compromised, the attacker inherits the live session. The 15-minute idle close directly limits this window.

Configuration Requirements

Work through all nine requirements. Check each one off as you complete it.

App Connector · ZPA Admin Portal → Infrastructure → App Connectors
  1. Log in to ZPA Admin Portal
  2. Navigate to Infrastructure → App Connectors → Connector Groups → Add App Connector Group — name it (e.g. "Safemarch-On-Prem"), click Save
  3. Navigate to Infrastructure → App Connectors → Connector Keys → Add Provisioning Key
  4. Select the connector group you just created, click Save — copy the full key value immediately
  5. On the App Connector VM (10.0.0.3), place the key: cp <keyfile> /opt/zscaler/var/provision_key
  6. Start the connector service: /opt/zscaler/bin/zsproxy start
  7. Return to ZPA Admin Portal → Infrastructure → App Connectors and confirm the connector shows status Connected before proceeding
Application Segments · ZPA Admin Portal → Application Management → Application Segments
  1. Navigate to Applications → App Segments → Add Application Segment
  2. Name: Intranet
  3. Domain/Address: intranet.patraining.safemarch.com
  4. Ports: TCP 80, TCP 443
  5. Assign App Connector Groups: select the group created in req 1
  6. Click Save
  1. Navigate to Applications → App Segments → Add Application Segment
  2. Name: File Share
  3. Domain/Address: shares.patraining.safemarch.com
  4. Ports: TCP 139, TCP 445
  5. Assign App Connector Groups: select the group from req 1 — click Save
  1. Navigate to Applications → App Segments → Add Application Segment
  2. Name: HVAC
  3. Domain/Address: hvac.patraining.safemarch.com
  4. Ports: TCP 80
  5. Assign App Connector Groups: select the group from req 1
  6. Click Save — the Maintenance dept restriction and 15-minute idle close are configured in the Access Policy (req 7)
  1. Navigate to Applications → App Segments → Add Application Segment
  2. Name: SIEM
  3. Domain/Address: 10.0.0.2
  4. Ports: TCP 22
  5. Assign App Connector Groups: select the group from req 1 — click Save
  6. The domain-joined device posture requirement is enforced in the Access Policy (req 7)
  1. Navigate to Applications → App Segments → Add Application Segment
  2. Name: RDP
  3. Domain/Address: rdp.patraining.safemarch.com
  4. Ports: TCP 3389, UDP 3389
  5. Assign App Connector Groups: select the group from req 1 — click Save
  6. Administrator dept restriction and 10-minute re-auth are configured in the Access Policy and Timeout Policy (req 7)
Access, Timeout & Forwarding Policy · ZPA Admin Portal → Policy
  1. Navigate to Policy → Access Policy → Add Rule
  2. Rule 1 — All Users / Intranet + File Share: Criteria: Any authenticated user. App Segments: Intranet, File Share. No additional posture. Action: Allow.
  3. Rule 2 — Maintenance / HVAC: Criteria: Department = Maintenance. App Segments: HVAC. Idle Network Timeout: 15 minutes. Action: Allow.
  4. Rule 3 — Administrator / SIEM (domain-joined): Criteria: Department = Administrator AND Posture = Domain Joined. App Segments: SIEM. Action: Allow.
  5. Rule 4 — Administrator / RDP: Criteria: Department = Administrator. App Segments: RDP. Action: Allow.
  6. Navigate to Policy → Timeout Policy — set RDP session re-auth to 10 minutes.
  7. Ensure department-specific rules are ordered above the default Deny rule.
  1. Navigate to Policy → Client Forwarding Policy → Add Rule
  2. Name the rule (e.g. "Bypass-Safemarch-Internal")
  3. Add the four destinations: erp.safemarch.com, crm.safemarch.com, contractors.safemarch.com, b2b.safemarch.com
  4. Action: Bypass — this tells ZCC to not tunnel these destinations through ZPA
  5. Click Save
  1. In the ZPA Admin Portal, navigate to Applications → Application Discovery
  2. Ensure the App Connector from req 1 is enabled for discovery scanning
  3. Review the discovered applications list — ZPA automatically detects traffic patterns through the connector
  4. Select any relevant discovered applications and promote them to managed App Segments if required

Testing Requirements

  1. From the Corp: Client PC (192.168.1.1), browse to http://intranet.patraining.safemarch.com — confirm it loads via ZPA.
  2. From the Client PC, access the file share at \\shares.patraining.safemarch.com — confirm it mounts via ZPA.
  3. Log in as a maintenance user and browse to http://hvac.patraining.safemarch.com — access should succeed. Leave the session idle for 15+ minutes and confirm ZPA terminates it.
  4. Log in as an admin user from a domain-joined VM and SSH to 10.0.0.2 — confirm it succeeds via ZPA. Attempt from a non-domain-joined device — confirm it is denied.
  5. As an admin user, RDP to rdp.patraining.safemarch.com (Admin / Admin-123!) — confirm ZPA policy matches and re-auth fires after ~10 min.
  6. Attempt to reach erp.safemarch.com — confirm ZPA does NOT tunnel it (Client Forwarding bypass applied).

QUICK REFERENCE ▲
App Connector
On-prem agent that connects internal resources to the Zscaler cloud — users never connect directly to it.
Application Segment
Defines an internal app's address, ports, and connector group — the unit ZPA policy acts on.
Idle Timeout
Automatically terminates a ZPA session after N minutes of inactivity — reduces risk from unattended sessions.
Client Forwarding Bypass
Excludes specific traffic from the ZPA tunnel — needed for local network access and connector management.
