Lab 10: Apply Zero Trust Automation: Adding URL Filtering Policy

LAB 10

Scenario: Configure a ZIdentity OneAPI client, authenticate a Postman session against it, and use the ZIA REST API to programmatically create a URL filtering rule that blocks gambling websites for all users.

Time: ~30 min
Portals: ZIdentity Admin Portal · ZIA Admin Portal · Postman (Corp: Win Svr VM)
💭 THINK FIRST · ZERO TRUST AUTOMATION

The OneAPI is Zscaler's unified REST API layer — it lets you do everything you can do in the portal, programmatically, enabling Infrastructure-as-Code approaches to security policy.

  • Why would a security team prefer to create a URL filtering rule via API rather than the ZIA Admin portal — what does this enable that the portal cannot?
    Q1 — MODEL ANSWER
    API-based rule creation enables: (1) Version control — the JSON payload can be committed to Git, reviewed, and rolled back. (2) Automation — the same API call can be triggered by a SIEM alert, a SOAR playbook, or a CI/CD pipeline (e.g. auto-block gambling domains when a finance audit kicks off). (3) Scale — one API call can be looped across dozens of tenant environments simultaneously, which is essential for MSPs managing multiple customers. The portal is single-tenant, single-session, and human-speed.
  • What is the role of the OneAPI client in ZIdentity, and how does it differ from a regular admin user account?
    Q2 — MODEL ANSWER
    A OneAPI client is a machine identity — a non-human account that authenticates via client credentials (client ID + secret) rather than username/password/MFA. It's designed for API-to-API authentication, not human login. Unlike admin user accounts (which have session-based authentication tied to a human), the OneAPI client generates short-lived OAuth tokens that expire after a defined period. This follows the principle of least privilege: the API client gets only the API permissions it needs, cannot log into the portal, and its token lifecycle is tightly controlled.
  • What would happen if the JSON payload for the URL filtering rule omits the "protocols": ["ANY_RULE"] parameter?
    Q3 — MODEL ANSWER
    The API call would either fail validation (the ZIA API requires Protocol Criteria to be defined for URL filtering rules) or create a rule with no protocol scope, which would not match any traffic. ZIA URL filtering rules require an explicit protocol match — ANY_RULE covers all protocols (HTTP, HTTPS, FTP). Without it, the gambling block rule would be created in the portal but would never fire, because no protocol criteria is defined to trigger evaluation. This is a common API pitfall: the portal often fills in sensible defaults, but the API is strict about explicit parameters.

Configuration Requirements

Work through these four requirements across ZIdentity, Postman, and the ZIA API. Check each one off as you complete it.

OneAPI Client · ZIdentity Admin Portal → OneAPI Clients
  1. Log in to ZIdentity Admin Portal at the URL shown in Lab Configuration above
  2. Navigate to Administration → Integration → API Clients → + Add API Client
  3. In the Client tab: Name: "ZIA-API-Client", Access Token Lifetime: 90 minutes, Validation Type: Secret. Click + Add to generate the client secret — copy it immediately, it will not be shown again
  4. In the Resources tab: under ZIA, assign the Super Admin scope. Click Save
  5. Note the auto-generated Client ID alongside the copied Client Secret — you need both in Postman
Postman · Corp: Win Svr VM
  1. Open Postman on the Corp: Win Svr VM — File → Import → select the Zscaler OneAPI Postman collection JSON (available from automate.zscaler.com → Tools → Postman)
  2. In the Collections panel, click the collection root (Updated OneAPI Collection) and open the Authorization tab
  3. Set Access Token URL to your ZIdentity URL with suffix /oauth2/v1/token (e.g. https://zs4803safemarch.zslogin.net/oauth2/v1/token)
  4. Enter your Client ID and Client Secret from requirement 1
  5. Scroll down and click Get New Access Token — when the popup appears, click Use Token
  6. All requests in the collection now automatically include the Bearer token in the Authorization header
ZIA REST API · ZIA API (https://zsapi.zscalerthree.net)
  1. In the Postman collection, navigate to ZIA → URL Filtering Policy → find the POST (create rule) endpoint
  2. Click the Body tab — select raw / JSON
  3. Enter the rule payload: {"name":"Block-Gambling","order":1,"action":"BLOCK","urlCategories":["GAMBLING"],"protocols":["ANY_RULE"],"state":"ENABLED"}
  4. Click Send — confirm the response is HTTP 200 or 201 and the body contains the new rule's ID
  5. Activate the change: in ZIA Admin Portal, click Activate (API calls do not auto-activate)
  1. Log in to ZIA Admin Portal at https://admin.zscalerthree.net
  2. Navigate to Policy → URL Filtering
  3. Confirm the new rule appears in the list with the correct action and URL category
  4. From the Corp: Client PC VM, browse to a gambling website — confirm it is blocked
  5. Check Analytics → Web Insights to confirm the block event is logged

Testing Requirements

  • After the API call succeeds (HTTP 200/201), open ZIA Admin Portal → Policy → URL Filtering
  • Confirm the new URL filtering rule appears in the list with the correct settings
  • Browse to a gambling site from the Corp: Client PC VM — confirm it is blocked
  • Check ZIA Web Insights logs to confirm the block event is recorded

MENTAL NOTE
API-driven policy has three steps: (1) authentication — get a token from ZIdentity using OneAPI client credentials, (2) the API call — POST to ZIA with the token as Bearer auth and the rule JSON in the body (don't forget "protocols": ["ANY_RULE"]), (3) verification — confirm in the ZIA portal that the rule was created correctly. The OneAPI client in ZIdentity is the bridge between your script and the ZIA API.
