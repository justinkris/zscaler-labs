Lab 2: Configure ZIA and ZPA Traffic Forwarding

LAB 2

Scenario: Safemarch needs all ZCC users to have every byte of traffic tunnelled to the Zero Trust Exchange, regardless of port or protocol. You will configure a forwarding profile, a Client Connector profile, and ZPA re-authentication settings so security policy applies to every user, everywhere.

Time: ~45 minutes
Portals: ZIA Admin · Client Connector · ZPA Admin
💭 THINK FIRST · TRAFFIC FORWARDING

The forwarding profile is the policy engine's gatekeeper — if traffic doesn't reach the Zscaler cloud, no policy can inspect or control it.

  • What is the difference between a ZIA Forwarding Profile and a ZCC Client Connector Profile, and why do both need to be configured?
    Q1 — MODEL ANSWER
    A ZIA Forwarding Profile defines HOW traffic is sent to Zscaler — tunnel type, ports, protocols. A ZCC Client Connector Profile defines WHO gets which forwarding profile applied, plus client-side settings like log mode and SSL cert auto-install. You need both: the forwarding policy is meaningless if the client isn't configured to use it, and the client profile is useless if there's no forwarding profile telling it how to tunnel.
  • Why would you set ZPA Re-authentication time to 10 minutes, and what is the security tradeoff?
    Q2 — MODEL ANSWER
    A 10-minute window limits how long a compromised ZPA session token can be used — at most 10 minutes before the user must re-authenticate. The tradeoff is friction: for long-running sessions like RDP or file transfers, the user is interrupted frequently. Organisations balance this based on app sensitivity.
  • What happens to traffic on non-standard ports if "All ports and protocols" is NOT enabled in the forwarding profile?
    Q3 — MODEL ANSWER
    Traffic on non-standard ports bypasses the Zscaler tunnel entirely — uninspected, unlogged, and outside all ZIA policy. An attacker or data exfiltration tool using a non-standard port would be completely invisible to your security controls.

Configuration Requirements

Work through these six requirements in the ZIA Admin and ZPA portals. Check each one off as you complete it.

  1. Log in to ZIA Admin Portal at https://admin.zscalerthree.net
  2. Navigate to Infrastructure → Client Connector → Forwarding Profiles
  3. Click Add Forwarding Profile — name it (e.g. "Safemarch-AllPorts")
  4. Set Windows Driver Selection to Packet Filter Based
  5. Under Forwarding Profile Action for ZIA — set On-Trusted Network, VPN Trusted Network, and Off-Trusted Network all to Tunnel, Tunnel version: Z-Tunnel 2.0
  6. Under Forwarding Profile Action for ZPA — set all network states to Tunnel
  7. Click Save
  1. Navigate to Infrastructure → Client Connector → ZCC Profiles
  2. Click Add ZCC Profile — name it (e.g. "Safemarch-Default")
  3. Under ZIA settings, set Disable ZIA to Require Password
  4. Under ZPA settings, set Disable ZPA to Require Password
  5. Note: also complete requirements 3, 4, and 5 in this same profile before saving
  6. Click Save
  1. In the same ZCC Profile (Infrastructure → Client Connector → ZCC Profiles)
  2. Locate the Log Mode field
  3. Set it to Warn
  4. Click Save
  1. In the same ZCC Profile
  2. Enable the Auto-install Zscaler SSL Certificate toggle
  3. This causes the Zscaler root CA to be installed automatically on the endpoint at enrollment
  4. Click Save
  1. In the same ZCC Profile, find the Forwarding Profile dropdown
  2. Select the forwarding profile you created in requirement 1 (e.g. "Safemarch-AllPorts")
  3. This attaches the all-ports forwarding policy to every device enrolled in this profile
  4. Click Save — this saves requirements 2, 3, 4, and 5 together
  1. Log in to the ZPA Admin Portal
  2. Navigate to Policy → Re-authentication Policy
  3. Set the re-authentication timeout to 10 minutes
  4. Click Save

MENTAL NOTE
Two separate things must be configured: (1) a Forwarding Profile specifying all-port/protocol tunnelling, and (2) a ZCC Profile that attaches the forwarding profile and sets client behaviour. ZPA re-auth is set independently in the ZPA portal. All three must be in place before traffic inspection begins.

QUICK REFERENCE ▲
Forwarding Profile
Defines how ZCC tunnels traffic — ports, protocols, and tunnel type (Zscaler Tunnel 2.0).
ZCC Profile
Client-side settings: log level, SSL cert install, disable password, and which forwarding profile to apply.
ZPA Re-auth
How often users must re-authenticate to ZPA. 10 min is aggressive — balance security against friction for long-running sessions.
Log Mode (Warn)
ZCC logs warnings and errors only. Reduces noise in production; use Debug only when troubleshooting.
