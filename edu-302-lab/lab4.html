Lab 4: Configure ZIA Logging & Reporting

LAB 4

Scenario: Safemarch needs blocked web traffic events to stream in real time from ZIA to the on-premises SIEM for threat analysis and compliance reporting. You will add the NSS VM, configure an NSS Feed, and verify log records arrive at the SIEM.

Time: ~30 min
Portal: ZIA Admin (https://admin.zscalerthree.net)
💭 THINK FIRST · ZIA LOG STREAMING

NSS is Zscaler's bridge between cloud-generated logs and your on-premises SIEM — without it, your security team is flying blind during an incident.

  • Why would you filter the NSS feed to stream only "blocked" web transactions rather than all traffic — what is the operational benefit and what do you lose?
    Q1 — MODEL ANSWER
    Streaming only blocked transactions dramatically reduces log volume — a 1,000-user organisation might generate millions of allowed transactions per day but only thousands of blocked ones. This makes SIEM storage costs manageable and reduces analyst fatigue (less noise). What you lose: you cannot do retrospective analysis of allowed traffic for threat hunting. A more complete deployment streams all transactions but applies SIEM-side filters. For this lab, blocked-only is the baseline requirement.
  • The NSS Server VM is at 10.0.0.4. What would happen if you configure the Management Interface and Service Interface with the same IP address?
    Q2 — MODEL ANSWER
    The NSS VM uses two interfaces for a reason: the Management Interface (10.0.0.4/24) handles admin access and ZIA cloud connectivity for pulling logs, while the Service Interface (10.0.0.5/24) handles the actual log forwarding to the SIEM over TCP 514. Using the same IP for both would create routing conflicts — the VM's OS cannot simultaneously serve management traffic and high-throughput syslog forwarding on the same interface without careful QoS. Zscaler's NSS deployment guide requires separate IPs.
  • What is the purpose of using WinSCP to transfer the NSS certificate to the NSS Server VM rather than generating it on the VM?
    Q3 — MODEL ANSWER
    The NSS certificate is issued by ZIA when you provision the NSS server in the ZIA portal. The VM itself cannot generate this certificate — it comes from Zscaler's cloud authority. WinSCP (pre-installed on Corp: Win Svr) provides a GUI SFTP client that lets you transfer the downloaded certificate from the Windows Server to the Linux NSS VM without needing to set up SSH key auth or use command-line SCP. It's the path of least resistance in the lab environment.

Configuration Requirements

Work through all four requirements. Check each one off as you complete it.

NSS Server · ZIA Admin Portal → Administration → Nanolog Streaming Service → NSS Servers
  1. Navigate to Administration → Nanolog Streaming Service → NSS Servers
  2. Click Add NSS Server
  3. Set Type to NSS for Web
  4. Enter IP addresses: 10.0.0.4 and 10.0.0.5 (the NSS VM in Skytap)
  5. Ensure the NSS VM is running in Skytap before proceeding
  6. Click Save
NSS Feed · ZIA Admin Portal → Administration → Nanolog Streaming Service → NSS Feeds
  1. Navigate to Administration → Nanolog Streaming Service → NSS Feeds
  2. Click Add NSS Feed
  3. Set NSS Server to the server added in req 1
  4. Set SIEM IP to 10.0.0.2 and port to 514 (TCP)
  5. Set Format to CSV and Log Type to Web
  6. Add a filter to include only blocked traffic (Action = Block) — click Save
SIEM Server Setup · Corp: SIEM Server (10.0.0.2) — Ubuntu Linux
⚠ Check your VM first: SSH into the SIEM Server (10.0.0.2), NOT the NSS Server (10.0.0.4).
Credentials: username zscaler / password Zscaler123!
  1. Update package lists and install rsyslog
    sudo apt-get update sudo apt-get install rsyslog -y
  2. Open rsyslog.conf for editing
    sudo nano /etc/rsyslog.conf
  3. Enable UDP and TCP input on port 514 — find and uncomment (or add) all four lines in the MODULES section:
    /etc/rsyslog.conf — uncomment or add (MODULES section) module(load="imudp") input(type="imudp" port="514") # provides TCP syslog reception module(load="imtcp") input(type="imtcp" port="514") $template RemInputLogs, "/var/log/remotelogs/%FROMHOST-IP%/%PROGRAMNAME%.log" *.* ?RemInputLogs
  4. Add log routing rule — paste this at the very bottom of the file:
    /etc/rsyslog.conf — add at bottom *.* /var/log/syslog
    💡 Save with Ctrl+O ↵, then exit with Ctrl+X.
  5. Validate the config file syntax — catch errors before restarting:
    rsyslogd -f /etc/rsyslog.conf -N1
    💡 Output should end with rsyslogd: End of config validation run. — fix any errors before proceeding.
  6. Create the syslog file and set permissions
    sudo touch /var/log/syslog sudo chmod 640 /var/log/syslog sudo chown syslog:adm /var/log/syslog
  7. Create the remotelogs directory
    sudo mkdir -p /var/log/remotelogs sudo chown -R syslog:syslog /var/log/remotelogs sudo chmod -R 755 /var/log/remotelogs
  8. Allow port 514 through the firewall
    sudo ufw allow 514/tcp sudo ufw allow 514/udp
  9. Start, enable, and restart rsyslog
    sudo systemctl start rsyslog sudo systemctl enable rsyslog sudo systemctl restart rsyslog
  10. Verify status and confirm port 514 is listening
    sudo systemctl status rsyslog
    ✅ Look for Active: active (running) before continuing.
    ss -antpl | grep 514
    ✅ You should see rsyslogd listed, confirming it is bound to port 514.
Verification · Corp: SIEM Server (10.0.0.2)
  1. From Corp: Client PC (192.168.1.1), browse to a blocked URL category (e.g. www.freeslots-machines.com) to generate a blocked event.
  2. Check the syslog file exists on the SIEM
    ls -la /var/log/syslog
  3. Watch for incoming log records in real time
    sudo tail -f /var/log/syslog
    ✅ You should see comma-separated log entries appear within seconds of the blocked browse event.
    💡 Press Ctrl+C to exit tail when done.

MENTAL NOTE
Three things must align for logs to flow: (1) NSS Server — the VM registered in ZIA (10.0.0.4/10.0.0.5), (2) NSS Feed — defines CSV format, block-only filter, SIEM dest 10.0.0.2:514, (3) rsyslog on the SIEM — must be listening on TCP 514 before the feed delivers records. All three are required; missing any one and logs silently never arrive.
