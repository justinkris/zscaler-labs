Lab 4: Configure ZIA Logging & Reporting

LAB 4

Scenario: Safemarch needs blocked web traffic events to stream in real time from ZIA to the on-premises SIEM for threat analysis and compliance reporting. You will configure the full pipeline: ZIA portal → NSS VM → SIEM running rsyslog, streaming only blocked-transaction records in CSV format.

Time: ~45–60 min
Products: ZIA, NSS VM, rsyslog
Prerequisite: Lab 3 policies active (generates blocked traffic)
💭 THINK FIRST · NSS Architecture

Before touching the portal, make sure you can reason through how the three components trust each other. The most common exam mistake is treating NSS as a simple syslog forwarder — it's a cloud-connected log decoder with a specific certificate-based trust model.

  • If NSS doesn't need to register an IP address in the ZIA portal, what mechanism does Zscaler use to authenticate which NSS VM belongs to which tenant?
    Q1 — MODEL ANSWER
    Authentication is entirely certificate-based. When you add an NSS Server object in ZIA, the portal generates a tenant-specific certificate (.pem). This certificate embeds your tenant identity and the correct Zscaler cloud endpoint. When NSS starts and reads that certificate, it knows which cloud cluster to connect to and presents the cert as its credential. Zscaler validates the cert — no IP registration is involved at all. This is why using another student's certificate will permanently show Disconnected.
  • NSS has two network interfaces — Management (eth0) and Service (eth1). Which one connects to Zscaler cloud, and which one sends logs to your SIEM? Why does the separation matter?
    Q2 — MODEL ANSWER
    Management (eth0 / 10.0.0.4) pulls nanolog data from Zscaler cloud over HTTPS port 443. Service (eth1 / 10.0.0.5) pushes decoded syslog to the SIEM on TCP 514. Separating them means your SIEM firewall rules only need to allow traffic from the service IP — not the management IP. When verifying log receipt on the SIEM, you filter by 10.0.0.5, not 10.0.0.4. Troubleshooting is also cleaner: no cloud connectivity = eth0 problem; no logs at SIEM = eth1 or feed problem.
  • The NSS feed is configured to stream only blocked transactions. If no logs appear on the SIEM, what is the first thing you should rule out before blaming the network or rsyslog?
    Q3 — MODEL ANSWER
    Rule out the filter first. The feed only streams records that match "blocked" policy action. If you haven't generated any blocked traffic from the client PC, there is nothing for NSS to send — even if the entire pipeline is working perfectly. Always generate a test block (e.g. browse to a gambling/gaming site) before concluding the SIEM isn't receiving logs.

Part 1 ZIA Admin Portal

Task 4.1 Add the NSS Server & Download Certificate ▾

1 · Navigate to NSS Servers

ZIA Admin Portal› Administration› Nanolog Streaming Service› NSS Servers
  1. Click Add NSS Server (top right).

2 · Configure NSS Server Fields

FieldValueNotes
Name requiredSafemarch-NSSAny descriptive name
Type requiredNSS for WebUse NSS for Firewall only for firewall logs
StatusEnabledMust be enabled for the feed to stream

No IP address is entered here. The ZIA portal does not register the NSS VM's IP. The certificate downloaded in the next step handles all identity and routing.

3 · Save and Download Certificate

  1. Click Save, then click Activate Changes.
  2. Click on the NSS Server entry you just created.
  3. Click Download Certificate — this produces a .pem file unique to your tenant.
  4. Save the certificate to Corp: Win Svr (10.0.0.8) — you will use WinSCP from there to transfer it to the NSS VM in Task 4.4.

Exam trap: This certificate is unique to your tenant. Using a certificate from another student's pod causes permanent authentication failure — NSS will show Disconnected with no error message.

Task 4.2 Configure the NSS Feed ▾

1 · Navigate to NSS Feeds

ZIA Admin Portal› Administration› Nanolog Streaming Service› NSS Feeds
  1. Click Add NSS Feed.

2 · Configure Feed Fields

FieldValueWhy
Feed NameSafemarch-WebLog-BlockedDescriptive — easier to audit later
NSS ServerSafemarch-NSSLinks this feed to the NSS VM you registered
SIEM IP Address required10.0.0.2Corp: SIEM Server — destination for log stream
SIEM TCP Port required514Standard syslog TCP port — must match rsyslog config
Feed Output TypeWeb LogLab requires web transaction logs
Feed Output FormatCSVLab requirement — human-readable, SIEM-parseable
User ObfuscationDisabledKeep usernames visible for SIEM correlation
StatusEnabled

3 · Set the Log Filter — Blocked Only

  1. Scroll to Log Filters within the feed configuration.
  2. Under Filter by Policy Action, select: Blocked
  3. Leave all other filters unset — only transactions where a block policy fired will stream.

Leaving the filter unset streams ALL transactions. This floods the SIEM with every web request. The lab — and the exam — specifically require blocked-only. An examiner will check this setting.

4 · Save and Activate

  1. Click Save → Click Activate Changes.
  2. The feed will not stream until the NSS VM is connected (Task 4.4). Continue to Part 2.

Task 4.3 Configure NSS VM Network Interfaces ▾

1 · Access the NSS VM Console

  1. Open the NSS Server VM console in Skytap.
  2. Log in: username zsroot / password zsroot
  3. You will see the NSS appliance configuration menu.

2 · Set Management Interface (eth0)

SettingValue
Interfaceeth0 (Management)
IP Address required10.0.0.4
Subnet Mask255.255.255.0 (/24)

This interface pulls nanolog data from Zscaler cloud outbound over HTTPS port 443.

3 · Set Service Interface (eth1)

SettingValue
Interfaceeth1 (Service)
IP Address required10.0.0.5
Subnet Mask255.255.255.0 (/24)

This interface sends decoded syslog to the SIEM on TCP 514. Logs arriving at the SIEM will show source IP 10.0.0.5, not 10.0.0.4.

4 · Set Gateway and DNS

SettingValueWhy critical
Default Gateway required10.0.0.254NSS must route outbound to Zscaler cloud
Nameserver (DNS) required10.0.0.254NSS must resolve Zscaler cloud FQDNs

If gateway or DNS is wrong, NSS will never connect. The portal shows Disconnected permanently with no specific error — it silently fails to resolve the cloud endpoint.

5 · Verify Outbound Connectivity

From the NSS VM shell, confirm network is reachable before installing the certificate:

NSS VM — connectivity checks
ping -c 4 8.8.8.8
# Expect: replies — confirms gateway is working

nslookup nss.zscaler.net
# Expect: valid IP returned — confirms DNS is working

Do not proceed to certificate installation until both succeed. A certificate installed on a VM that cannot reach Zscaler cloud will appear to install fine but NSS will never come up.

Task 4.4 Transfer & Install the NSS Certificate ▾

1 · Transfer Certificate via WinSCP

  1. On Corp: Win Svr (10.0.0.8), open WinSCP (pre-installed).
  2. Create a new session to the NSS VM:
    • Protocol: SCP or SFTP
    • Host: 10.0.0.4
    • Username: zsroot / Password: zsroot
  3. Connect and copy the .pem certificate file to the NSS VM home directory: /home/zsroot/

2 · Install the Certificate on the NSS VM

  1. On the NSS VM console, from the appliance configuration menu, select Install Certificate (or equivalent option depending on your firmware version).
  2. When prompted, enter the path to the certificate: /home/zsroot/<certfile>.pem
  3. Confirm the installation. NSS will restart its connection service.

After certificate installation, NSS typically takes 1–2 minutes to authenticate with Zscaler cloud. Monitor the status in ZIA portal → NSS Servers — it should change from Disconnected to Connected.

3 · Verify NSS is Connected in ZIA Portal

ZIA Admin Portal› Administration› Nanolog Streaming Service› NSS Servers
  1. Confirm the NSS Server row shows status: Connected.
  2. If still Disconnected after 3 minutes, verify gateway/DNS settings from Task 4.3 and confirm the correct .pem file was transferred.

Part 3 SIEM Server (10.0.0.2)

Task 4.5 Configure rsyslog on the SIEM Server ▾
⚠ Check your VM: SSH into SIEM Server (10.0.0.2), NOT the NSS Server (10.0.0.4).
Credentials: username zscaler / password Zscaler123!
  1. Update package lists and install rsyslog
    sudo apt-get update sudo apt-get install rsyslog -y
  2. Open rsyslog.conf for editing
    sudo nano /etc/rsyslog.conf
  3. Enable UDP and TCP input on port 514 — find and uncomment (or add) all four lines in the MODULES section:
    /etc/rsyslog.conf — uncomment or add (MODULES section) module(load="imudp") input(type="imudp" port="514") # provides TCP syslog reception module(load="imtcp") input(type="imtcp" port="514") $template RemInputLogs, "/var/log/remotelogs/%FROMHOST-IP%/%PROGRAMNAME%.log" *.* ?RemInputLogs
  4. Add log routing rule — paste this at the very bottom of the file:
    /etc/rsyslog.conf — add at bottom *.* /var/log/syslog
    💡 Save with Ctrl+O ↵, then exit with Ctrl+X.
  5. Validate the config file syntax
    rsyslogd -f /etc/rsyslog.conf -N1
    💡 Output should end with rsyslogd: End of config validation run. — fix any errors before proceeding.
  6. Create the syslog file and set permissions
    sudo touch /var/log/syslog sudo chmod 640 /var/log/syslog sudo chown syslog:adm /var/log/syslog
  7. Create the remotelogs directory
    sudo mkdir -p /var/log/remotelogs sudo chown -R syslog:syslog /var/log/remotelogs sudo chmod -R 755 /var/log/remotelogs
  8. Allow port 514 through the firewall
    sudo ufw allow 514/tcp sudo ufw allow 514/udp
  9. Start, enable, and restart rsyslog
    sudo systemctl start rsyslog sudo systemctl enable rsyslog sudo systemctl restart rsyslog
  10. Verify status and confirm port 514 is listening
    sudo systemctl status rsyslog
    ✅ Look for Active: active (running) before continuing.
    ss -antpl | grep 514
    ✅ rsyslogd should be listed, confirming it is bound to port 514.

Validate Test the Full Pipeline

Testing Requirements

  1. In ZIA Portal → NSS Servers, confirm the NSS Server shows Connected.
  2. On Corp: SIEM Server (10.0.0.2), run sudo systemctl status rsyslog — confirm active (running).
  3. From Corp: Client PC (192.168.1.1), browse to a blocked site (e.g. www.freeslots-machines.com) to generate a blocked event.
  4. On the SIEM, run sudo tail -f /var/log/syslog — confirm CSV log records appear within seconds.
  5. In ZIA Admin Portal → Analytics → Web Insights, confirm the same block event is visible with the correct action and URL category.

Troubleshoot Common Issues

SymptomMost likely causeFix
NSS shows Disconnected Wrong certificate, bad gateway, or DNS not resolving Zscaler cloud Verify ping 8.8.8.8 and nslookup nss.zscaler.net succeed from NSS VM. Re-download and reinstall the certificate from ZIA portal.
NSS shows Connected but no logs at SIEM No blocked traffic generated, or TCP 514 blocked by UFW on SIEM Browse to a gambling/gaming site from Client PC, then check sudo ufw status — confirm port 514 is allowed.
rsyslog starts but port 514 not listed in ss output imtcp module not uncommented in rsyslog.conf Re-open /etc/rsyslog.conf, confirm module(load="imtcp") and input(type="imtcp" port="514") lines are uncommented. Run rsyslogd -N1 to validate config.
rsyslog fails to start / errors in status Syntax error in rsyslog.conf Run rsyslogd -f /etc/rsyslog.conf -N1 to identify the error line. Fix and restart.
Logs arrive at SIEM but feed shows wrong format Feed output format not set to CSV In ZIA portal → NSS Feeds, edit the feed and confirm Feed Output Format = CSV. Save and Activate.

MENTAL NOTE
Four things must align for logs to flow: (1) NSS Server — registered in ZIA, certificate installed (10.0.0.4/10.0.0.5); (2) NSS Feed — CSV format, block-only filter, SIEM dest 10.0.0.2:514; (3) rsyslog on SIEM — listening on TCP 514 with UFW open; (4) blocked traffic — the feed only streams what the filter matches. Missing any one and logs silently never arrive.
