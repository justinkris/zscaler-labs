Lab 8: Adding Custom Application and Creating Custom Probes

LAB 8

Scenario: Add a custom application to the ZDX Admin Portal, then create a Web Probe and a Cloud Path Probe to continuously measure performance and reachability for that application.

Time: ~30 min
Portal: ZDX Admin Portal
💭 THINK FIRST · ZDX MONITORING

ZDX probes run from the user's device — not from the cloud — which means probe data reflects the actual experience of the end user, including ISP conditions, local network congestion, and ZCC path routing.

  • What is the difference between a Web Probe and a Cloud Path Probe, and what different failure scenarios does each one surface?
    Q1 — MODEL ANSWER
    A Web Probe sends an HTTP request to a destination URL and measures page fetch time, HTTP response code, and availability. It surfaces application-layer failures — the app is slow, returning 5xx errors, or down. A Cloud Path Probe uses ICMP to measure network path quality (packet loss, latency, hops) from the device through the Zscaler cloud to the destination. It surfaces network-layer failures — high latency on a specific cloud path, packet loss, or routing anomalies. Together they tell you whether a slow app is caused by the application itself or by the network underneath it.
  • Why is "Follow Web Probe: Test_Web" set on the Cloud Path Probe — what does this relationship mean operationally?
    Q2 — MODEL ANSWER
    The "Follow Web Probe" setting tells ZDX to run the Cloud Path Probe only when the Web Probe detects an issue. Instead of running ICMP probes constantly (consuming device battery and network resources), the Cloud Path Probe activates as a diagnostic follow-up when the Web Probe reports degraded performance or failure. This event-driven architecture reduces overhead and makes the Cloud Path data immediately contextual — you see the network path state at the exact moment the application was experiencing problems.
  • What happens to ZDX probe data if the device running ZCC goes offline — and how does this affect the application health score?
    Q3 — MODEL ANSWER
    Probes require an active ZCC session to execute — if the device is offline, no probes run from that device. ZDX distinguishes between "no data" (device offline) and "bad score" (probe running but results are poor). Offline devices do not contribute to the application's ZDX Score. If many devices go offline simultaneously (e.g. during an outage), the number of contributing probe sources drops, which can actually make the aggregate ZDX Score appear better than it is. For alerts, you need minimum 1 active device for the alert trigger logic to work correctly.

Configuration Requirements

Work through these three requirements in the ZDX Admin Portal. Check each one off as you complete it.

Custom Application · ZDX Admin Portal → Apps
  1. Log in to ZDX Admin Portal
  2. Navigate to Apps → Add App
  3. Set Name to "Test Application", Type to Custom, Status to Enabled
  4. Click Save
Probes · ZDX Admin Portal → Apps → Test Application → Probes
  1. Open the Test Application in ZDX
  2. Navigate to the Probes tab → Add Probe → Web Probe
  3. Name: Test_Web. Frequency: 5 minutes
  4. Success codes: 200–299 and 304. Retry attempts: 2. Retry interval: 60 seconds
  5. Enable Follow redirects. Set max redirects to 5
  6. Click Save
  1. On the same Test Application, Add Probe → Cloud Path Probe
  2. Name: Test_CloudPath. Protocol: ICMP
  3. Associate with the Test_Web web probe
  4. Frequency: 5 minutes. Packets: 5. Interval between packets: 1000ms. Timeout: 1000ms
  5. Click Save

Testing Requirements

  • Allow 10–15 minutes for initial probe data to collect after saving
  • Open ZDX Admin Portal → Apps → Test Application and confirm Web Probe results appear
  • Confirm Cloud Path Probe results and hop data appear
  • Verify the ZDX score populates for the application

MENTAL NOTE
ZDX monitoring is built in two layers: (1) the Application entity (the thing being monitored — Test Application), (2) the Probes attached to it. Web Probe = synthetic HTTP check (measures page fetch time, success rate). Cloud Path Probe = ICMP hop-by-hop measurement (identifies where latency occurs in the network path). Both are needed for a complete ZDX score.
