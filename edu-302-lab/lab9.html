Lab 9: Creating a Custom Alert

LAB 9

Scenario: Create a custom ZDX alert for the Test Application that triggers when Page Fetch Time exceeds 800ms AND ZDX Score drops below 10 for Europe-geolocated devices, firing 3 times in a row before sending an email notification.

Time: ~20 min
Portal: ZDX Admin Portal
💭 THINK FIRST · ZDX ALERTING

Alert thresholds need careful tuning — too sensitive and you get alert fatigue, too loose and you miss real incidents. The "Alert Only if Repeated" setting is the primary tool for eliminating noise.

  • Why is "Alert Only if Repeated: 3 times in a Row" a better default than alerting on the first occurrence of a threshold breach?
    Q1 — MODEL ANSWER
    A single Page Fetch Time spike of 800ms can be caused by a momentary network blip, browser caching behaviour, or a one-off server GC pause — none of which indicate a real problem requiring intervention. Alerting on three consecutive occurrences means the probe must detect the threshold breach three times in a row (across 3 probe cycles = 15 minutes at 5-minute frequency) before firing. This eliminates transient false positives while ensuring that sustained degradation — which real users are experiencing — does generate an alert. It's the minimum viable noise filter for a production ZDX deployment.
  • Why is the alert scoped to "Geolocation: Europe" rather than all devices — what operational scenario does this reflect?
    Q2 — MODEL ANSWER
    A geolocation filter reflects the operational reality that performance issues are often regional — a Zscaler data center in Frankfurt could be experiencing congestion, or the ISP path through Europe could be degraded, while North American users are unaffected. Scoping the alert to Europe means you get a signal specific enough to act on ("Europe users are degraded") rather than a global alert that could mean anything. It also prevents an issue affecting 10 European users from being masked by good data from 990 users in other regions.
  • What is a ZDX Score, and why is a threshold of less than 10 combined with Page Fetch Time ≥ 800ms appropriate as dual alert criteria?
    Q3 — MODEL ANSWER
    The ZDX Score (0-100) is a composite metric aggregating probe results, cloud path quality, and device health signals. A score below 10 represents severely degraded experience. Using dual criteria (Page Fetch Time ≥ 800ms AND ZDX Score < 10) ensures the alert fires only when BOTH the application layer (slow HTTP response) AND the overall experience score are simultaneously bad. This avoids alerting when: the page is slow but the device is healthy overall (transient app-side issue), or the ZDX Score is low but HTTP is fast (device-side problem, not app-related). Dual criteria = precision.

Configuration Requirements

Work through these four requirements in the ZDX Admin Portal. Check each one off as you complete it.

Alert Configuration · ZDX Admin Portal → Alerts
  1. Log in to ZDX Admin Portal
  2. Navigate to Alerts → Add Alert
  3. Name: TestApp_Alert. Alert type: Application. Severity: High
  4. Select Application: Test Application
  5. Continue to the next step
  1. In the alert configuration, find the Device Filters section
  2. Add a geolocation filter: select Europe (continent or region as available)
  3. This scopes the alert to devices physically located in Europe only
  4. Continue to the next step
  1. In the Criteria section, add the first condition: Page Fetch Time ≥ 800 (ms)
  2. Add the second condition: ZDX Score < 10
  3. Ensure the logic is set to AND (both must be true simultaneously)
  4. Continue to the next step
  1. In the Trigger section, set "Alert when condition is met" to 3 consecutive times
  2. Set delivery method to Email
  3. Add the recipient email address(es)
  4. Click Save to activate the alert

Testing Requirements

  • Save the alert and confirm it appears in the Alerts list as active
  • Review the configured criteria — confirm application, geolocation filter, thresholds, and email action are all correct
  • (Live triggering requires a degraded-experience event from a Europe device — verify configuration correctness rather than waiting for a real trigger)

MENTAL NOTE
A ZDX alert has four components: (1) the entity (which app), (2) the scope (which devices — by geolocation here), (3) the criteria (the thresholds — Page Fetch Time ≥ 800ms AND Score < 10), and (4) the action (when to fire — after 3 consecutive hits — and how — email). AND logic means both criteria must be met simultaneously; using OR would create too many false positives.
