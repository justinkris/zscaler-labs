Lab 3: Forward Traffic with the Zscaler Client Connector

LAB 3

Scenario: Configure the Zscaler Client Connector (ZCC) to securely forward all user traffic, regardless of protocol, to Zscaler for inspection and protection. This enables firewall functionality, secure internet access, and multi-platform compatibility for users on and off trusted networks.

Time: ~20 minutes
💭 THINK FIRST · ZCC TRAFFIC FORWARDING

ZCC uses a Forwarding Profile to decide how traffic is sent to Zscaler depending on the network context (on-trusted, off-trusted, VPN). Z-Tunnel 2.0 with Packet Filter-Based driver intercepts all protocols — not just HTTP/HTTPS — enabling full firewall functionality. The Windows App Profile links the forwarding config to specific devices and users.

  • Z-Tunnel 2.0 is required for Firewall functionality. What does Z-Tunnel 1.0 support that makes it insufficient for firewall policy enforcement?
  • You set VPN Trusted Network and Off-Trusted Network to "Same as On-Trusted Network." In what real-world scenario would you want different forwarding behavior for VPN vs. Off-Trusted traffic?
  • The Windows App Profile sets "Install Zscaler SSL Certificate: On." Why is the SSL certificate installation necessary, and what would break if it were missing?
Q1 — MODEL ANSWER
Z-Tunnel 1.0 only forwards HTTP and HTTPS proxy traffic — it works at the application layer for web browsing but ignores other protocols like SSH, DNS, ICMP, or custom TCP/UDP. Z-Tunnel 2.0 operates at the network layer, intercepting all IP traffic regardless of protocol, which is what gives the Zscaler Cloud Firewall visibility and enforcement capability over non-web traffic.
Q2 — MODEL ANSWER
A company might want VPN traffic to use "None" (bypass Zscaler) if the corporate VPN already provides equivalent security inspection. Off-Trusted traffic (home networks, coffee shops) might need a stricter "Tunnel" mode with no local proxy bypass. Setting these the same simplifies the lab, but in production the differences often align with how much the organization trusts each network type.
Q3 — MODEL ANSWER
Zscaler performs SSL inspection by acting as a man-in-the-middle for HTTPS traffic, re-signing certificates with the Zscaler CA. For the browser to trust these re-signed certs, the Zscaler root CA must be installed in the system trust store. Without it, every HTTPS site would show an untrusted certificate error, rendering secure web browsing unusable.

Task 3.1: Configure Traffic Forwarding Options

1 Login to the Experience Center with the student account

  1. Navigate to https://console.zscaler.com and authenticate with:
    • Username: student@<Student FQDN>
    • Password: the password you set in Lab 1
  2. Click Sign in.

2 Create Forwarding Profile

  1. Navigate to Infrastructure > Connectors > Client > Forwarding Profile for Platforms.
  2. Click Add Forwarding Profile.
  3. Name the profile: HandsOnLab
  4. Scroll down. In Windows Driver Selection, select Packet Filter Based.
  5. In Forwarding Profile Action for ZIA — On-Trusted Network: Tunnel | Tunnel version: Z-Tunnel 2.0
  6. For VPN Trusted Network and Off-Trusted Network: click Same as "On-Trusted Network".
  7. In Forwarding Profile Action for ZPA — On-Trusted Network: Tunnel | VPN & Off-Trusted: Same as "On-Trusted Network"
  8. Click Save.
Infrastructure → Connectors → Client → Forwarding Profile for Platforms. The Client section controls all ZCC deployment and behavior settings.
Click Add Forwarding Profile to create the HandsOnLab profile that defines how ZCC tunnels traffic to Zscaler.

Note: If Zscaler Client Connector version 4.8 or higher is installed on Windows, the application will default to the packet filter-based driver, ignoring the Route-Based configuration option. Tunnel 2.0 is required to intercept all traffic for Firewall functionality.

Forwarding profile ZIA configuration: Profile Name = HandsOnLab, Packet Filter-Based driver, ZIA On-Trusted = Tunnel with Z-Tunnel 2.0.
ZPA forwarding settings: On-Trusted = Tunnel, VPN and Off-Trusted both set to Same as On-Trusted Network. Click Save when complete.

3 Add Windows App Profile

  1. Use the left navigation panel to go to Platform Settings > Windows.
  2. Click Add Windows Policy under the App Profile section.
  3. Configure:
    • Name: HandsOnLab
    • Rule Order: 1
    • Status: Enable
    • Forwarding Profile: HandsOnLab (the profile created above)
    • Install Zscaler SSL Certificate: On
    • User Groups: Select All
  4. Click Add.
Platform Settings → Windows → App Profiles tab. Click Add Windows Policy to create the profile that applies the HandsOnLab forwarding profile to all Windows devices.
Windows Policy configuration: Name = HandsOnLab, Rule Order = 1, Forwarding Profile = HandsOnLab, SSL Certificate = On, User Groups = All Selected.

Task 3.2: Sign In to ZCC and Verify Protection

1 Open and authenticate ZCC

  1. On the Corp: Client PC, in the Windows Task Bar (bottom-right corner), click the Show hidden icons option.
  2. Select the Zscaler Client Connector icon and click Open Zscaler.
  3. Login with the student credentials: student@<Student FQDN> / password set in Lab 1 (<Session Password>).
  4. Click Sign in.
ZCC tray icon — click Show hidden icons (^) then right-click or hover the ZCC icon to access Open Zscaler.
Sign in with the student account — this is the main admin account, which has full ZIA, ZPA, and ZDX service entitlements.

2 Verify Private Access Settings

  1. On the Private Access page, confirm:
    • Service Status: ON
    • Client and Broker IP addresses are populated
ZCC Private Access — Service Status ON, Broker IP populated. If ZPA or ZDX is missing, verify the Student Admin user has ZPA/ZDX service entitlements in Administration → Identity → Users.

Note: If you do not see ZPA or ZDX in ZCC, your user may not have the correct service entitlements. Navigate to Administration > Users, click the Edit icon for Student Admin, and confirm ZPA and ZDX are assigned in the Entitlements tab.

3 Verify Internet Security Settings

  1. On the Internet Security page, confirm:
    • Service Status: ON
    • Username: displays correctly
    • Total Bytes Sent / Received: values increment as you load web pages
ZCC Internet Security ON — Tunnel Version v2.0-DTLS confirms Z-Tunnel 2.0 is active. Total Bytes Sent/Received should increment as you browse the web.

4 Verify Digital Experience Settings

  1. On the Digital Experience page, confirm:
    • Service Status: ON
    • Authentication Status: Authenticated
    • Username: displays correctly
ZCC Digital Experience ON — Authentication Status: Authenticated confirms ZDX is actively monitoring this device's digital experience.

5 Verify Applied App Policy and Traffic Forwarding

  1. On the More page, confirm App Policy: HandsOnLab
  2. Navigate to https://ip.zscaler.com in a browser on the Corp: Client PC.
  3. Confirm that you are accessing the Internet via a Zscaler service edge geographically close to the user.
More page — App Policy: HandsOnLab confirms the Windows profile you created is applied. If not reflected, click Update Policy to force a refresh.
ip.zscaler.com confirms traffic is flowing through the Zscaler Zero Trust Exchange. The Service Edge location reflects the geographically closest Zscaler node.

Note: If the updated policy HandsOnLab is not reflected in ZCC, click Update Policy in the ZCC under the More tab to force a policy refresh. Wait a few seconds and re-verify.

🧠
MENTAL NOTE
Two objects to create for ZCC traffic forwarding: (1) Forwarding Profile (defines tunnel behavior — Z-Tunnel 2.0 + Packet Filter for all-protocol forwarding); (2) Windows App Profile (links the forwarding profile to devices/groups + installs Zscaler SSL cert). Verify with: ZCC status pages (Private Access, Internet Security, Digital Experience all ON) + ip.zscaler.com showing a Zscaler service edge.

ZPA software in the data centre that makes outbound TLS tunnels to ZPA cloud. No inbound firewall rules required.

One-time key placed at /opt/zscaler/var/provision_key. Authorises a new connector to join a specific connector group.

ZPA object defining a private resource by domain/IP + port. Together with an Access Policy, enables ZPA access.
