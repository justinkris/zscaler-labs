Lab 7: Configure DNS Security

LAB 7

Scenario: DNS is a foundational attack and bypass vector. You'll configure a DNS wildcard domain in the App Profile for ZPA private access, create a DNS Control block rule for Gambling domains, and verify the block in both nslookup and DNS Insights logs.

Time: ~20 minutes
💭 THINK FIRST · DNS SECURITY

DNS is the first step in almost every network connection. By controlling DNS resolution, ZIA can block access to malicious or policy-violating domains before a TCP connection is even attempted — lighter and faster than blocking at the HTTP/HTTPS layer.

  • A DNS Control block rule prevents resolution of a gambling domain. But if a user has a hardcoded DNS server (8.8.8.8) set on their NIC rather than using the default DNS, will your DNS Control rule still apply — and why?
  • After creating a DNS Control block rule, you run nslookup and the domain still resolves. What are the two most likely reasons, and what's the correct fix for each?
  • Why does the App Profile's DNS wildcard domain inclusion (*.labtenant.com) matter for ZPA — and what would happen to ZPA private app resolution if you omitted it?
Q1 — MODEL ANSWER
With Z-Tunnel 2.0 enabled, ZCC intercepts all network traffic including DNS queries regardless of the configured DNS server. The user's NIC might point to 8.8.8.8, but the DNS query packet is intercepted by the ZCC kernel driver and forwarded to Zscaler's service edge for resolution — DNS Control applies. Without Z-Tunnel 2.0 (e.g., only Z-Tunnel 1.0 which operates at TCP/HTTPS level), UDP-based DNS bypasses ZCC entirely and the block would fail.
Q2 — MODEL ANSWER
Two common causes: (1) Policy not activated — the rule was saved but not pushed to service edges; click Activate and re-test. (2) Stale DNS cache — the domain was resolved before the block rule took effect and the response is cached locally; run ipconfig /flushdns then re-test. A third cause is that ZCC hasn't received the updated policy; click Update Policy in the ZCC More tab.
Q3 — MODEL ANSWER
The DNS wildcard inclusion (*.labtenant.com) tells the App Profile to handle DNS for the private domain through ZPA rather than forwarding it to a public DNS server. Without it, a DNS query for intranet.labtenant.com would be sent to the internet-facing DNS resolver, which either returns NXDOMAIN or a wrong IP — the private app becomes unreachable even if the ZPA app segment is correctly configured. Private domain DNS must be directed through ZPA's resolution path.

Task 7.1 — Configure DNS Domain Inclusion in the App Profile

1 Edit the HandsOnLab App Profile

  1. Navigate to Policies → Endpoint Management → App Profiles.
  2. Click on the HandsOnLab app profile (Windows) to edit it.
  3. Select the DNS tab.
App Profiles under Windows platform settings — the HandsOnLab profile controls DNS behaviour for enrolled endpoints.
HandsOnLab app profile — edit it to add the private domain DNS wildcard that directs *.labtenant.com resolution through ZPA.

2 Add the private domain wildcard

  1. In the DNS tab, navigate to Domain Inclusions (or DNS Domains).
  2. Add *.labtenant.com as a DNS domain wildcard.
  3. This ensures all *.labtenant.com DNS queries are handled through ZPA's private DNS path.
  4. Click Save.
DNS tab in the App Profile — the wildcard inclusion routes private domain queries through ZPA instead of public DNS.
*.labtenant.com wildcard — any ZPA private app domain under this parent will resolve correctly through the ZPA cloud path.

3 Apply the updated policy

  1. In ZCC on the Corp: Client PC, navigate to More → click Update Policy to force a policy refresh.
Update Policy forces ZCC to pull the latest profile from the Zscaler cloud immediately.

Task 7.2 — Create a DNS Control Block Rule for Gambling

1 Verify gambling site resolves before the rule

  1. On the Corp: Client PC, open Command Prompt.
  2. Run: nslookup bet365.com
  3. The domain should resolve to an IP address — confirming DNS is working and the site is currently accessible via DNS.
Before DNS Control rule: gambling domain resolves normally to an IP — no block in place yet.

2 Add a DNS Filtering Rule

  1. Navigate to Policies → Firewall → DNS Control.
  2. Click Add DNS Filtering Rule.
  3. Set Name to Block Gambling DNS.
  4. In DNS Categories, select Gambling.
  5. Set Action to Block.
  6. Click Save, then Activate.
DNS Control under Firewall policies — DNS-layer filtering acts on the query before any TCP connection is attempted.
Add DNS Filtering Rule — category-based DNS blocking prevents resolution before a connection can be established.
Gambling category selected — all domains ZIA classifies as Gambling will be blocked at the DNS layer.
Action: Block — ZIA returns NXDOMAIN or refuses resolution for matching domains.
Complete DNS filtering rule — name, Gambling category, Block action. Save then Activate.
Activate to push the DNS Control policy change to service edges.

3 Flush DNS cache and verify block

  1. On the Corp: Client PC, run: ipconfig /flushdns
  2. Run: nslookup bet365.com
  3. The query should now time out or return a block response — confirming DNS Control is active.
  4. Run: nslookup google.com — this should still resolve normally.
ipconfig /flushdns clears the local DNS cache — required to see the block effect for previously-resolved domains.
Gambling domain blocked — nslookup times out or receives a block response from ZIA's DNS resolver.
Google.com still resolves — the block is category-specific, not a broad DNS blackout.

Task 7.3 — View DNS Insights Logs

1 Filter DNS Insights for blocked queries

  1. Navigate to Analytics → DNS Insights.
  2. Apply a filter for Action: Block or filter for the gambling domain by name.
  3. Confirm blocked gambling DNS queries appear in the log.
DNS Insights filter — Block action filter narrows the view to all DNS queries ZIA refused to resolve.
DNS Insights log — blocked gambling queries visible with domain, category, action, and timestamp for audit purposes.
🧠
MENTAL NOTE
Two DNS configurations to remember: (1) App Profile DNS Domain Inclusion — wildcard (*.labtenant.com) routes private domain queries through ZPA's resolver so private apps are reachable; (2) DNS Control policy — category-based block rule prevents resolution of policy-violating domains before any TCP connection is attempted. After creating a DNS block rule, always run ipconfig /flushdns and Update Policy in ZCC to clear stale cached responses.
