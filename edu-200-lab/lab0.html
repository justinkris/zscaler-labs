💭 THINK FIRST · LAB SETUP

The lab runs inside a Skytap virtual environment with three pre-configured VMs. Each student gets a unique pod with its own FQDN. Before you can configure anything, you need to know exactly who can log in and how traffic is meant to flow.

  • Why does each student need a unique FQDN (e.g., zs9901.labtenant.com) rather than a shared domain? What problems would a shared domain create in a multi-tenant lab?
  • The lab has three VMs: Corp: Client PC, Corp: Win Svr 2016, and Corp: App Connector. Which VM acts as the endpoint being protected by ZCC, and which enables ZPA private access — and why does that distinction matter?
  • The SyncZPA attribute is set on lab users. What would happen in later ZPA labs if you forgot to set it, and what does that tell you about how identity sync works between ZIdentity and ZPA?
Q1 — MODEL ANSWER
Each student pod needs an isolated FQDN so that user accounts, policies, and Zscaler cloud configuration don't collide between students. A shared domain would mean Joe in one student's pod could inherit policies or entitlements set by another student, making validation impossible and breaking the lab reset mechanism.
Q2 — MODEL ANSWER
The Corp: Client PC is the endpoint being protected — it's the Windows 10 machine where ZCC is installed to forward traffic through ZIA and connect to private apps via ZPA. The Corp: App Connector is the ZPA component that sits in the data center and creates the outbound tunnel to the ZPA cloud, allowing the Client PC to reach private resources without inbound firewall rules.
Q3 — MODEL ANSWER
Without SyncZPA set to "Zscaler", the lab automation scripts won't sync the user into ZPA's identity store. This means the user won't appear as a valid ZPA identity, and ZPA access policies based on user identity would fail. It reveals that ZPA maintains its own identity directory that must be populated from ZIdentity — it isn't automatic unless explicitly triggered by this attribute.

Lab Description and Scenario

Your organization (Labtenant) has deployed the Zscaler Zero Trust Exchange to protect against Internet security threats, control Internet access, and provide secure access to private resources. You will deploy the Zscaler Client Connector (ZCC) to end-user devices to control Internet access, provide access to private applications, and probe, benchmark, and measure digital experiences.

Your tasks include:

  • Connect users to the Zero Trust Exchange by deploying ZCC to Windows client devices
  • Configure SSL inspection
  • Deploy Zscaler Digital Experience (ZDX) reporting and create an initial set of probes
  • Configure user access to both Internet and private applications
  • Verify Cyberthreat Protection and Data Protection services

Lab Setup

You will have access to Zscaler Zero Trust Exchange (ZIdentity/ZIA/ZPA/ZDX) and a Skytap 'Pod'. You will configure ZIdentity, ZIA, ZPA, and ZDX services using a Windows 10 Client PC in Skytap to connect to the Zero Trust Exchange.

Lab Topology

Lab topology: Corp: Client PC connects through the Zscaler Zero Trust Exchange (ZTE) to the Internet, and via App Connector to the Corporate Data Center (10.0.0.0/24).

Lab Access Information

Each student receives a unique Student ID (e.g., zs9901) and Student FQDN (e.g., zs9901.labtenant.com). The admin login format is student@<Student FQDN>. The password is unique per session and identical across all configured accounts in the pod.

QUICK REFERENCE ▲
Student FQDN
Your unique tenant domain, e.g. zs9901.labtenant.com. Used in all login IDs.
Experience Center
console.zscaler.com — unified admin portal for ZIA, ZPA, ZDX, ZIdentity.
ZCC
Zscaler Client Connector — endpoint agent that tunnels all traffic to the ZTE.
Z-Tunnel 2.0
Network-layer tunnel that intercepts all IP protocols. Required for Cloud Firewall.
Service Entitlement
Grants a user access to use a Zscaler service (ZIA, ZPA, ZDX) via ZCC.
Admin Entitlement
Grants a user access to configure a service in the admin console, with a specific role.
SyncZPA
User attribute (value: Zscaler) that syncs the user into ZPA's identity store for policy use.
Forwarding Profile
Defines tunnel type/version and forwarding action per network context (on/off/VPN trusted).
Cloud App Control
Application-layer policy that identifies apps by DPI (e.g., WhatsApp) regardless of port. Separate from Firewall Filtering.
SSL Inspection
Trusted MitM: ZIA decrypts → inspects → re-encrypts using Zscaler CA. Requires Zscaler Root CA trusted on endpoint.
Cert Pinning Bypass
"Do Not Inspect" SSL rule for apps that verify server cert fingerprints (e.g., Threema). Must be above the Inspect All rule.
DLP Engine
Defines what sensitive content looks like (dictionary patterns + thresholds). Paired with a DLP Rule to define the action.
App Connector
ZPA software in the data centre that makes outbound TLS tunnels to ZPA cloud. No inbound firewall rules required.
Provisioning Key
One-time key placed at /opt/zscaler/var/provision_key. Authorises a new connector to join a specific connector group.
App Segment
ZPA object defining a private resource by domain/IP + port. Together with an Access Policy, enables ZPA access.
SIPA
Secure Internet and Private Access — routes ZPA App Connector internet traffic through ZIA for consistent egress IP + policy.
ZDX Score
0–100 composite metric of end-to-end digital experience. 0 = worst, 100 = best. <50 = Poor, 50–79 = Okay, 80+ = Good.
Y-Engine
ZDX ML engine that analyses probe data to identify root-cause factors (Wi-Fi, DNS, TCP, server response) and ranks their contribution.
ZDX Probe
Synthetic background test run by ZCC that measures DNS Time, TCP Connect, Server Response, and Page Fetch Time continuously.
