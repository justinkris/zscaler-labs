Lab 12: Traffic Forwarding through SIPA

LAB 12

Scenario: Internet traffic from ZPA App Connectors currently exits with the data centre IP. You'll configure SIPA to route this traffic through ZIA, giving it a consistent Zscaler service edge egress IP, then verify the IP change using ip.zscaler.com and ping.eu.

Time: ~30 minutes
💭 THINK FIRST · SIPA TRAFFIC FORWARDING

SIPA (Secure Internet and Private Access) routes internet-bound traffic from ZPA App Connectors through ZIA before it reaches the internet. Normally, app connector traffic egresses directly to the internet with the connector's IP. SIPA changes the egress to a Zscaler service edge IP — enabling consistent policy enforcement and a predictable egress address.

  • Without SIPA, internet traffic from a ZPA App Connector exits directly to the internet with the connector's public IP. What problem does this create for an organisation that wants consistent, auditable internet egress from all users including those in data centres?
    Q1 — MODEL ANSWER
    Without SIPA, each App Connector (and the servers behind it) egresses with the data centre's public IP — a different IP from the ZCC users who egress through ZIA service edges. This creates two egress profiles that are difficult to manage consistently: allow-lists, IP reputation policies, and audit logs show different IPs for the same organisation. SIPA routes all data centre internet traffic through ZIA, giving it the same consistent Zscaler egress IP and the same policy enforcement as user traffic.
  • SIPA involves configuring both ZPA (app segment + gateway) and ZIA (forwarding rule). Why must both be configured — what does each side contribute, and what happens if only one is configured?
    Q2 — MODEL ANSWER
    ZPA's side (app segment + gateway) defines the destination for internet-bound traffic from the connector and routes it to a ZIA forwarding point. ZIA's side (forwarding rule) tells ZIA how to handle traffic arriving from the ZPA connector — which policy to apply, which service edge to use for egress. If only ZPA is configured, traffic reaches the ZIA forwarding point but ZIA doesn't know what to do with it. If only ZIA is configured, the connector never knows to send its traffic there in the first place.
  • After SIPA is configured, checking ip.zscaler.com shows a different egress IP than before. What does this new IP represent, and how would you tell if the traffic was going through ZIA versus exiting directly?
    Q3 — MODEL ANSWER
    The new egress IP is a Zscaler service edge IP — the same category of IP that ZCC users see when they visit ip.zscaler.com. The key confirmation is that ip.zscaler.com returns a response showing "You are accessing the Internet via Zscaler" with a ZTE service edge location, rather than showing a raw data centre IP. To distinguish ZIA from direct exit: check ip.zscaler.com for the Zscaler branding, and check Analytics in ZIA for traffic logs sourced from the connector.

Task 12.1 — Configure SIPA in ZPA

1 Check initial egress IP

  1. Ensure ZCC has ZIA disabled (Internet Security OFF) so you can see the connector's native egress IP.
  2. Navigate to http://ip.zscaler.com on the Corp: Client PC and note the current IP address.
ZCC with ZIA disabled — checking the native (non-Zscaler) egress IP before SIPA configuration as a baseline.
Native egress IP — the data centre's public IP before SIPA is configured. This will change after SIPA is active.

2 Create a SIPA App Segment in ZPA

  1. Re-enable ZCC (Internet Security ON), then in the ZPA Admin Console navigate to Applications → App Segments.
  2. Click Add Application Segment.
  3. Set Name to SIPA (or ip.zscaler.com).
  4. Set Domain to ip.zscaler.com (the test destination via SIPA).
  5. Add port ranges: 80 and 443.
  6. Assign the On-Prem App Connector Group.
App Connectors page — the On-Prem connector is Connected and will handle the SIPA app segment traffic.
SIPA App Segment — defines ip.zscaler.com as a ZPA-routed destination, forcing that traffic through the SIPA path.
Port ranges 80 and 443 — HTTP and HTTPS access to the SIPA destination domain.
SIPA segment connector group — On-Prem handles the SIPA traffic through the same connector used for Intranet access.

3 Configure the ZPA Gateway for SIPA

  1. In the ZPA Admin Portal, go to Infrastructure → App Connectors → Gateways.
  2. Click the existing gateway to open it (or click Add Gateway if none exists yet). In the gateway form, locate the SIPA section and set SIPA Forwarding to Enabled.
  3. In the SIPA App Segments field, select the SIPA app segment you created in step 2. Save the gateway.
Troubleshoot

If no gateway appears in the list, the App Connector may not have registered yet — check Infrastructure → App Connectors to confirm the On-Prem connector shows Connected status. If the SIPA section is absent from the gateway form, verify your ZPA subscription includes the SIPA add-on under Administration → Subscriptions.

ZPA Gateway SIPA configuration — the gateway tells ZPA where to forward internet-bound traffic from the connector for ZIA inspection.
SIPA server group — defines the server-side configuration for the SIPA app segment in ZPA.
SIPA App Segment review — domain, ports, connector group, and server group configured. Save to create the segment.

4 Create an Access Policy for SIPA

  1. In Policy → Access Policy, add a rule allowing access to the SIPA segment.
  2. Set criteria and user scope appropriately, then Save.
SIPA access policy — allows the appropriate users to use the SIPA forwarding path for internet-bound traffic.
Allow access via SIPA — the policy controls who can use the SIPA forwarding path.
SIPA access policy criteria — segment + user identity criteria define who gets SIPA forwarding.
Policy step in SIPA segment wizard — verify the access policy is linked before saving.

Task 12.2 — Configure ZIA Forwarding Rule for ZPA

1 Enable ZCC and navigate to ZIA forwarding

  1. In ZCC, ensure Internet Security is ON.
  2. In ZIA, navigate to Policy → Network Policies → ZPA (or Forwarding Control).
ZCC Internet Security ON — ZIA is now inspecting traffic. The SIPA configuration will route connector traffic through here.
ZIA Network Policies → ZPA — configure how ZIA handles traffic arriving from ZPA App Connectors via the SIPA path.

2 Add a forwarding rule for ZPA SIPA traffic

  1. Click Add Forwarding Rule.
  2. Configure the rule to match traffic from the ZPA SIPA gateway and forward it through ZIA.
  3. Set the appropriate destination and forwarding action.
  4. Click Save, then Activate.
ZIA Forwarding Rule for SIPA — routes traffic arriving from the ZPA connector through ZIA's inspection and egress path.
Forwarding rule destination criteria — scope the rule to the SIPA-related traffic destinations.
On-Prem App Connector shown in ZIA — confirms ZIA sees the connector and can route its traffic through the SIPA path.
Activate Forwarding Control changes — push the SIPA forwarding rule to ZIA service edges.

Task 12.3 — Verify SIPA Egress

1 Compare egress IP before and after SIPA

  1. Navigate to http://ip.zscaler.com with ZCC active.
  2. The egress IP should now be a Zscaler service edge IP — different from the native connector IP noted in Task 12.1.
  3. The page should confirm: "You are accessing the Internet via Zscaler."
SIPA App Segment targeting ip.zscaler.com — traffic to this domain routes through the SIPA forwarding path.
SIPA forwarding policy details — bypass vs. forward rules determine which connector traffic goes through ZIA.
SIPA forward action in the forwarding policy — traffic matching the SIPA criteria is routed through ZIA.
After SIPA: ip.zscaler.com shows a Zscaler service edge IP — connector internet traffic now egresses through ZIA, not the data centre IP.
SIPA segment extended to ping.eu — an external IP echo service used to verify the SIPA egress IP from a second source.
App Segments list: Discovery, Intranet, and SIPA all configured — each serves a distinct purpose in the ZPA architecture.
ping.eu confirms the new SIPA egress IP — connector internet traffic now leaves via the Zscaler service edge, consistent with ZCC user traffic.
🧠
MENTAL NOTE
SIPA configuration requires changes in both ZPA and ZIA: (1) ZPA: create an App Segment for the internet destination + configure the Gateway to use SIPA + create an access policy; (2) ZIA: add a Forwarding Control rule to route connector traffic through ZIA's service edge. Verification: ip.zscaler.com should change from the data centre's native IP to a Zscaler service edge IP, and show "accessing the Internet via Zscaler."
