Lab 5: Configure SSL Inspection Policies

LAB 5

Scenario: Without SSL inspection, encrypted traffic is a blind spot. You'll enable inspect-all for users, verify Zscaler's CA certificate is trusted, then add a bypass rule for the Threema cert-pinned messaging app, and confirm inspection logs in Web Insights.

Time: ~30 minutes
💭 THINK FIRST · SSL INSPECTION

Over 90% of internet traffic is encrypted. Without SSL inspection, ZIA can see that traffic exists but cannot inspect its content — malware, data exfiltration, and policy violations all hide inside TLS. SSL inspection performs a trusted man-in-the-middle using Zscaler's CA certificate.

  • SSL inspection requires re-signing every HTTPS certificate with Zscaler's CA. What must be true on the endpoint for browsers to trust that re-signed certificate — and what breaks if that condition isn't met?
  • Certain apps use certificate pinning, meaning they verify the server's certificate against a hard-coded expected value rather than the OS trust store. What happens to a cert-pinned app when SSL inspection is enabled, and how do you fix it without disabling inspection globally?
  • If you enabled "Inspect All Users" but then later added a bypass rule for a specific app — which rule wins, and why does order matter here the same way it does for firewall rules?
Q1 — MODEL ANSWER
The Zscaler Root CA certificate must be installed in the endpoint's trusted certificate store (and in Chrome's certificate store, since Chrome has its own trust store on Windows). If the CA isn't trusted, every HTTPS site triggers a certificate error and the browser refuses the connection. This is why the HandsOnLab app profile installs the CA during ZCC enrollment — without it, inspection would break all HTTPS.
Q2 — MODEL ANSWER
A cert-pinned app compares the server certificate's fingerprint against a value baked into the app's binary. When ZIA re-signs the certificate with Zscaler's CA, the fingerprint changes and the app rejects the connection outright — typically as a generic network error. The fix is a "Do Not Inspect" bypass rule scoped to that specific app: ZIA lets the original TLS certificate pass through unmodified, satisfying the app's pinning check.
Q3 — MODEL ANSWER
SSL inspection rules are also evaluated top-down, first match wins. A bypass rule for Threema must be placed above the "Inspect All" rule — otherwise Inspect All matches first and Threema's traffic is inspected before the bypass rule is reached. The same implicit logic as firewall rules: specific exceptions must always sit above broad rules.

Task 5.1 — Enable SSL Inspect All Users Rule

1 Verify pre-inspection certificate state

  1. On the Corp: Client PC, navigate to https://www.wikipedia.org.
  2. Click the padlock icon in the browser address bar → Certificate.
  3. Note the issuer — it should be Let's Encrypt (Wikipedia's real CA, not yet inspected by Zscaler).
Before SSL inspection: Wikipedia's real certificate from Let's Encrypt is visible end-to-end.

2 Add an SSL Inspection rule

  1. In ZIA, navigate to Policies → SSL Inspection → SSL/TLS Inspection Policy.
  2. Click Add SSL Inspection Rule.
  3. Set Name to Inspect All Users.
  4. Set Action to Inspect.
  5. Leave Users as all users and click Save.
  6. Click Activate to push the policy.
SSL/TLS Inspection Policy — the rule list follows top-down first-match logic, same as Firewall Filtering.
Action: Inspect — ZIA will decrypt, inspect content, then re-encrypt with Zscaler's CA before forwarding.
Inspect All Users rule configuration — broad rule that catches all HTTPS traffic not matched by a bypass rule above it.
Activate to push the SSL inspection policy change globally.

3 Verify Zscaler certificate is now used

  1. On the Corp: Client PC, navigate to https://www.wikipedia.org again.
  2. Check the certificate — the issuer should now be Zscaler Intermediate Root CA.
Before: Let's Encrypt CA — the original certificate reaching the browser.
After: Zscaler Intermediate Root CA — ZIA has re-signed the certificate, confirming inspection is active.

Task 5.2 — Verify Zscaler Root CA Certificate (Optional)

1 Check Windows certificate store

  1. On the Corp: Client PC, press Win + R and run certmgr.msc (current user store).
  2. Expand Trusted Root Certification Authorities → Certificates.
  3. Locate and verify the Zscaler Root CA certificate (installed via the HandsOnLab app profile).
Windows certmgr.msc — Zscaler Root CA must appear here for Windows to trust ZIA-re-signed certificates.
Zscaler Root CA in Trusted Root Certification Authorities — installed automatically by the HandsOnLab ZCC app profile.

2 Verify Chrome certificate store

  1. In Chrome, navigate to Settings → Privacy and Security → Security → Manage Certificates.
  2. Under Trusted Root Certification Authorities, confirm Zscaler Root CA is present.
Chrome maintains its own certificate store on Windows — the Zscaler CA must be present here as well as in the OS store.
Zscaler Root CA visible in Chrome's Trusted Root CA list — both OS and browser stores are covered.

Task 5.3 — Add a Certificate Pinning Bypass for Threema

1 Install Threema desktop app

  1. On the Corp: Client PC, navigate to the Threema download page (URL in the lab instructions).
  2. Download the Threema for Windows installer.
  3. Install via PowerShell or the downloaded installer.
  4. Open Threema — it will fail to connect because SSL inspection breaks its certificate pinning.
Threema download page — download the Windows installer to test cert-pinned app behaviour under SSL inspection.
Installing Threema via PowerShell — the app uses certificate pinning, so it will fail once SSL inspection is active.
Threema connection error — SSL inspection re-signs the certificate, causing the fingerprint check to fail.

2 Add a Do Not Inspect bypass rule for Threema

  1. In ZIA SSL/TLS Inspection Policy, click Add SSL Inspection Rule.
  2. Set Name to Bypass Threema.
  3. Set Action to Do Not Inspect.
  4. In Cloud Applications, search for and add Threema.
  5. Click Save.
  6. Important: Ensure the Bypass Threema rule is positioned above the Inspect All Users rule.
  7. Click Activate.
Do Not Inspect action — ZIA passes Threema's TLS session through unmodified, preserving the original server certificate.
Bypass Threema above Inspect All Users — the bypass rule must be matched first, before the broad Inspect rule.

3 Verify Threema connects after bypass

  1. Reopen Threema on the Corp: Client PC.
  2. Threema should now connect successfully — the original server certificate passes through intact.
Threema connects after the bypass rule — the cert-pinned connection succeeds because ZIA no longer re-signs the certificate.

Task 5.4 — Analyze SSL Inspection Logs

1 View inspected traffic in Web Insights

  1. In ZIA, navigate to Analytics → Web Insights.
  2. Filter by SSL Inspected: Yes to see only traffic that was inspected.
  3. Confirm that recent HTTPS sessions (e.g., Wikipedia) appear with Zscaler Intermediate Root CA as the issuer.
Web Insights SSL Inspected filter — confirms which sessions ZIA decrypted and re-encrypted for content inspection.
🧠
MENTAL NOTE
SSL inspection is a trusted MitM: ZIA decrypts → inspects → re-encrypts using its own CA. Two prerequisites: (1) Zscaler Root CA must be trusted by both the OS and browser certificate stores; (2) cert-pinned apps need a "Do Not Inspect" bypass rule positioned above the global Inspect rule. Confirm inspection is active by checking that site certificates now show Zscaler Intermediate Root CA as the issuer in the browser.
