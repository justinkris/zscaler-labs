Lab 4: Configure Firewall Filtering Policy

LAB 4

Scenario: Your organisation needs network-layer control via ZIA Cloud Firewall. You'll verify the default deny-all rule, create an explicit allow for ICMP and SSH administrative traffic, then add a Cloud App Control rule to block WhatsApp by application identity regardless of port.

Time: ~25 minutes
💭 THINK FIRST · FIREWALL FILTERING

ZIA's Cloud Firewall operates at the network layer and inspects all traffic forwarded via Z-Tunnel 2.0. Rules are evaluated top-down and the first match wins — making rule order a security decision, not just an organisational preference.

  • Why does a default Block and Drop rule sit at the bottom of every Firewall Filtering policy, and what would happen to unmatched traffic if it didn't exist?
  • If you create an Allow rule for ICMP and SSH, where must it sit relative to the default Block and Drop — and what breaks if the order is wrong?
  • WhatsApp uses port 443 like normal HTTPS. Why does blocking it require a Cloud App Control rule rather than a Firewall Filtering rule?
Q1 — MODEL ANSWER
The Block and Drop rule is the implicit-deny that keeps the firewall a closed system. Without it, unmatched traffic would be forwarded by default, reversing the security posture. With it, any new protocol or port that lacks an explicit allow rule is blocked automatically — exceptions must be made deliberately, not accidentally inherited.
Q2 — MODEL ANSWER
The Allow rule must be placed above Block and Drop. Rules are evaluated top-down and the first match terminates evaluation. If Block and Drop is first, ICMP/SSH traffic matches it and is dropped before the Allow rule is checked. By placing Allow above, matching traffic exits the chain at the first rule; everything else falls through to the implicit deny.
Q3 — MODEL ANSWER
A port-based firewall rule blocking 443 would also block all other HTTPS — unacceptable collateral damage. Cloud App Control works at the application layer, using deep packet inspection to identify WhatsApp's specific traffic signature even when it shares port 443 with HTTPS. Application-aware control is the only way to make targeted decisions on apps that blend in with legitimate traffic.

Task 4.1 — Verify the Default Block and Drop Rule

1 Navigate to Firewall Filtering Policy

  1. In the ZIA Admin Console, navigate to Policies → Firewall Control → Firewall Filtering Policy.
  2. Locate the Block and Drop rule at the bottom of the list.
  3. Note the action: Block with Drop response means packets are silently discarded — no TCP reset is sent to the client.
Default Block and Drop at the bottom — any traffic not matched by a rule above it is silently dropped.

Task 4.2 — Create an Allow Rule for ICMP and SSH

1 Add a new Firewall Filtering Rule

  1. Click Add Firewall Filtering Rule.
  2. Set Name to Allow ICMP and SSH.
  3. Set Action to Allow.
Add Firewall Filtering Rule — the rule wizard opens. Position in the list determines evaluation order.

2 Configure services and scope

  1. In Services, add ICMP_ANY and SSH.
  2. In Users, select the appropriate users or groups.
  3. Leave Destinations as Any.
  4. Click Save.
Scope the rule to specific users or groups to limit the allow surface.
ICMP_ANY and SSH services, Action: Allow — these are the two protocols needed for administrative connectivity.

3 Verify rule order and activate

  1. Confirm Allow ICMP and SSH appears above the default Block and Drop rule.
  2. Click Activate to push the policy to all service edges.
Activate pushes the rule to all Zscaler service edges globally within seconds.
Allow ICMP and SSH above Block and Drop — ICMP and SSH traffic hits Allow first; everything else falls through to the implicit deny.

Task 4.3 — Block WhatsApp via Cloud App Control

1 Add a Cloud Application Control Rule

  1. Navigate to Policies → URL & Cloud App Control → Cloud Application Control.
  2. Click Add Cloud Application Control Rule.
  3. Set Name to Block WhatsApp.
  4. In Users, select the appropriate users.
Scope the Cloud App Control rule to the same users who should have WhatsApp blocked.

2 Select application and action

  1. In Cloud Applications, search for and select WhatsApp.
  2. Set Action to Block / Drop.
  3. Click Save, then Activate.
WhatsApp selected by name — ZIA identifies it via DPI regardless of the port it uses.
Block / Drop silently discards packets — WhatsApp fails to connect without a visible block page.
Activate to publish the Cloud App Control rule change.

3 Verify WhatsApp is blocked

  1. On the Corp: Client PC, attempt to access WhatsApp in a browser.
  2. Confirm the connection is blocked or fails to load.
WhatsApp blocked — Cloud App Control intercepts at the application layer, not the port level.
Final policy: Allow ICMP and SSH above Block and Drop — explicit permit over implicit deny.
🧠
MENTAL NOTE
ZIA Cloud Firewall has two complementary layers: (1) Firewall Filtering — network/transport layer, port and protocol based, top-down first-match, implicit Block and Drop at the bottom; (2) Cloud App Control — application layer, identifies apps by DPI regardless of port. Blocking WhatsApp requires Cloud App Control specifically because it shares port 443 with HTTPS — a port-based rule would block all web traffic.
