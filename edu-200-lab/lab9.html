💭 THINK FIRST · ZPA INFRASTRUCTURE

ZPA App Connectors are lightweight software components that establish outbound TLS tunnels to the ZPA cloud. There are no inbound firewall rules — the connector calls out, so the data centre firewall doesn't need to be opened. A provisioning key is what authorises a new connector to join a specific connector group.

  • Traditional VPNs require inbound firewall holes (port 443 or IPSec ports) to the data centre. How does the ZPA App Connector architecture eliminate this requirement, and what is the security implication of the difference?
  • A provisioning key is used once to activate the App Connector, after which it is consumed. Why is it designed this way rather than using a persistent shared secret or password?
  • If an App Connector loses its network connection to the ZPA cloud, what happens to users who are actively connected to private apps through it — and does ZPA have a mechanism to handle this?
Q1 — MODEL ANSWER
The App Connector makes outbound TLS connections to the ZPA broker cloud — no inbound ports need to be opened on the data centre firewall. Traditional VPNs require inbound holes (IPSec or SSL VPN ports) that become attack surfaces: scannable, targetable, and subject to vulnerabilities in the VPN gateway software. ZPA's outbound-only model means the internal network is never directly reachable from the internet — an attacker can't scan for it.
Q2 — MODEL ANSWER
A one-time provisioning key is designed for zero-trust enrollment: the key authorises the initial registration but is consumed during activation, so there's no persistent secret that could be reused to register rogue connectors. A shared password would be a liability — if compromised, any machine could register as a connector in your environment. One-time keys limit the blast radius: a leaked key can only be used once before it expires or is used.
Q3 — MODEL ANSWER
Active sessions through a disconnected App Connector will drop — users lose their private app connections. ZPA handles this through redundancy at the connector group level: multiple App Connectors in the same group share the load, and ZPA automatically fails over to a healthy connector for new connections. For active sessions, ZPA can re-establish them through a surviving connector in the group, depending on the application's session tolerance.

Lab 9: Provision ZPA Infrastructure

LAB 9

Scenario: Before ZPA can provide access to internal applications, an App Connector must be activated in the data centre. You'll create a Connector Group and Provisioning Key in the ZPA Admin Console, then upload the key to the Corp: App Connector VM and start the ZPA service.

Time: ~25 minutes

Task 9.1 — Create App Connector Group and Provisioning Key

1 Create an App Connector Group

  1. In the ZPA Admin Console (navigate to ZPA from console.zscaler.com), go to Infrastructure → App Connectors → Connector Groups.
  2. Click Add App Connector Group.
  3. Set Name to On-Prem (or as instructed).
  4. Configure location and server settings as needed, then click Save.
App Connector Groups — logical groupings that share App Connectors and are referenced by app segment configurations.
On-Prem connector group — the group that the Corp: App Connector VM will join after provisioning.
Connector group settings — location and connection settings for the on-premises connector group.

2 Generate a Provisioning Key

  1. Navigate to Infrastructure → App Connectors → Connector Keys.
  2. Click Add Provisioning Key.
  3. Set Name and select the On-Prem connector group.
  4. Click Save.
  5. Copy the provisioning key value — you will need it in Task 9.2.
Connector Keys page — keys authorise new App Connectors to register with a specific connector group.
Add Provisioning Key linked to the On-Prem group — once used to activate a connector, the key is consumed.
Copy the provisioning key value — you'll upload it to the Corp: App Connector VM in the next task.

Task 9.2 — Activate the App Connector via WinSCP

1 Connect to the App Connector VM using WinSCP

  1. On the Corp: Client PC, open WinSCP from the desktop.
  2. Connect to the Corp: App Connector VM using the IP address and credentials from the lab topology.
  3. Accept any SSH host key prompt.
WinSCP on the Client PC — used to transfer the provisioning key file to the App Connector VM.
WinSCP login to the Corp: App Connector — the connector VM runs Linux (CentOS/RHEL). Use the IP from the lab topology.

2 Upload the provisioning key

  1. In WinSCP, navigate to /tmp/ on the remote VM.
  2. Create a file named provision_key and paste the provisioning key value into it, or upload the saved key file.
Provisioning key uploaded to /tmp/ on the connector VM — the activation script will read it from this location.

3 Run activation commands in terminal

  1. In WinSCP, open an SSH terminal session (press Ctrl+P or click Open in PuTTY).
  2. Run: cp /tmp/provision_key /opt/zscaler/var/provision_key
  3. Verify the file is present: ls /opt/zscaler/var/
  4. Start the ZPA connector service: /opt/zscaler/bin/zsproxy start
  5. Check service status: systemctl status zpa-connector
Copying the provisioning key to the ZPA connector's var directory — the activation process reads it from here.
Verify the provisioning key is in place before starting the service.
Starting the ZPA connector service — it reads the provisioning key, registers with the ZPA cloud, and joins the On-Prem group.
Service status: active (running) — the App Connector has registered with ZPA and established its outbound TLS tunnel.

4 Verify connector appears in ZPA console

  1. Back in the ZPA Admin Console, navigate to Infrastructure → App Connectors.
  2. The new connector should appear with status Connected.
App Connector Connected — the outbound TLS tunnel to the ZPA cloud is established. The data centre is now ZPA-accessible.
🧠
MENTAL NOTE
ZPA infrastructure setup: (1) Connector Group in ZPA console — logical grouping for connectors sharing the same location; (2) Provisioning Key — one-time key that authorises a new connector to join the group; (3) Activation — copy key to /opt/zscaler/var/provision_key on the Linux VM, then start zpa-connector service. No inbound firewall rules needed — the connector makes outbound-only TLS connections to ZPA cloud.
