💭 THINK FIRST · DATA LOSS PREVENTION

DLP in ZIA operates inline on HTTP/HTTPS traffic — it inspects file uploads, form posts, and web content for sensitive data patterns. For DLP to inspect encrypted traffic, SSL inspection must already be enabled.

  • Why is SSL inspection a hard prerequisite for DLP to work on HTTPS traffic — and what does DLP see if SSL inspection is disabled?
  • A DLP Engine defines what counts as sensitive data. A DLP Rule defines what to do when it's found. What's the consequence of creating a DLP rule without a properly configured engine — would it ever trigger?
  • A user uploads a file containing credit card numbers to a personal cloud storage service. The DLP rule blocks the upload and shows a notification. What must be true about the user's network path for ZIA to have been able to intercept and inspect that upload?
Q1 — MODEL ANSWER
DLP analyzes the plaintext content of HTTP/HTTPS traffic. Without SSL inspection, HTTPS payloads are encrypted — ZIA sees only ciphertext and cannot identify patterns like credit card numbers or SSNs inside the encrypted stream. DLP simply has nothing to inspect. SSL inspection decrypts the traffic first, allowing DLP engines to scan the plaintext before re-encryption and forwarding.
Q2 — MODEL ANSWER
A DLP rule without a properly configured engine would never match any content — an engine with no dictionaries or conditions matches nothing. The engine is the definition of what sensitive content looks like (credit card patterns, SSN regex, etc.). The rule is the action layer. Without a functional engine attached, the rule has no criteria to evaluate and will never trigger, even if files containing matching data are uploaded.
Q3 — MODEL ANSWER
The user's traffic must be flowing through ZIA — either via ZCC with Z-Tunnel 2.0 active, or through a Zscaler proxy/gateway. ZIA must be acting as the SSL inspection proxy for HTTPS uploads. If the user bypassed ZCC (VPN split-tunnel, trusted network exemption, or no ZCC), the upload goes directly to the internet and ZIA never sees it. DLP enforcement is only as good as the enforcement boundary.

Lab 8: Enforce Policy with Unified DLP

LAB 8

Scenario: Sensitive data (credit card numbers) must not leave the organisation via web uploads. You'll build a DLP Engine from dictionary patterns, create a DLP Rule to block and notify, then test with a file upload containing synthetic sensitive data.

Time: ~15 minutes

Prerequisite: SSL Inspection must be enabled (Lab 5 Task 5.1). DLP cannot inspect encrypted HTTPS uploads without it.

Task 8.1 — Build DLP Engine, Rule, and Test

1 Confirm SSL Inspect All is active

  1. Navigate to Policies → SSL Inspection and verify the Inspect All Users rule is enabled and activated.
SSL Inspect All active — prerequisite for DLP to inspect HTTPS upload content.

2 Create a DLP Engine

  1. Navigate to Policy → Data Loss Prevention → DLP Engines.
  2. Click the Engines tab, then click Add Custom DLP Engine.
  3. Set Name to Sensitive Data.
  4. Add a condition using the Credit Cards DLP dictionary with an appropriate match threshold.
  5. Click Save.
DLP Engines tab — custom engines define the content patterns that constitute a policy violation.
Credit Cards dictionary — Zscaler's built-in pattern library for financial data detection.
Sensitive Data engine configured with Credit Cards dictionary — the engine defines what to look for; the rule defines what to do.

3 Create a DLP Rule

  1. Navigate to Data Loss Prevention → Rules.
  2. Click Add DLP Rule.
  3. Set Name to Block Sensitive Data.
  4. Set DLP Engine to Sensitive Data (the engine you just created).
  5. Set Action to Block.
  6. Enable Notify User and set a custom notification message.
  7. Click Save, then Activate.
DLP Rule linked to the Sensitive Data engine — Action: Block with a user notification for blocked upload attempts.
Block + Notify — the upload is blocked and the user sees a notification explaining the policy violation.
Custom notification message — informs the user why their upload was blocked and what to do next.
Block Sensitive Data rule saved — the DLP engine + rule chain is complete. Activate to push the policy.

4 Test with a sensitive file upload

  1. On the Corp: Client PC, navigate to https://dlptoolbox.com.
  2. Click File Audit and upload the Sensitive Data.txt file (provided in the lab materials).
  3. ZIA should intercept the upload and display a DLP block notification.
DLP Toolbox File Audit — the upload endpoint used to trigger DLP inspection of the Sensitive Data.txt file.
DLP block notification — ZIA intercepted the upload, detected credit card patterns, and blocked the transmission with user notification.
🧠
MENTAL NOTE
ZIA DLP requires two objects: (1) DLP Engine — defines what sensitive content looks like (dictionary patterns, thresholds); (2) DLP Rule — links the engine to an action (Block, Allow, Notify). SSL inspection must be enabled for DLP to inspect HTTPS uploads. Test path: DLP Toolbox File Audit → upload a file with matching patterns → expect a block notification if the engine + rule + SSL inspection chain is correctly configured.
