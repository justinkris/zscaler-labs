Lab 11: Troubleshoot ZPA Access for Joe

LAB 11

Scenario: Joe cannot access private applications via ZPA. You'll log into ZCC as Joe, identify the root cause (missing ZPA service entitlement), then assign the entitlement in ZIdentity and verify Joe can reach the intranet site.

Time: ~25 minutes
💭 THINK FIRST · ZPA TROUBLESHOOTING

ZPA access requires two independent grants: a Service Entitlement (does the user have permission to use ZPA at all?) and an Access Policy (does the user have permission to access this specific app segment?). Troubleshooting begins by separating these two layers.

  • A user logs into ZCC but the Private Access tab is not visible. What is the most likely cause — an Access Policy issue or a Service Entitlement issue — and how would you confirm which one it is?
  • If the ZPA service entitlement is assigned but the user still can't reach a specific private app, where should you look next, and what information would you need?
  • Why must a ZPA Service Entitlement be assigned in ZIdentity rather than directly in the ZPA Admin Console — and what does this design reveal about the relationship between ZIdentity and ZPA?
Q1 — MODEL ANSWER
If Private Access is not visible in ZCC at all, the most likely cause is a missing ZPA Service Entitlement — the user simply doesn't have the right to use ZPA. An Access Policy issue would manifest differently: the user would have the Private Access tab, ZPA would show as active, but accessing the specific app would fail. To confirm, check the user's profile in ZIdentity under Service Entitlements and look for ZPA in the assigned services list.
Q2 — MODEL ANSWER
If the entitlement is present but the app is unreachable, check: (1) Does the Access Policy exist and is it activated? (2) Does the policy's user criteria match this user's login name or group? (3) Is the App Segment correctly configured with the right domain and port? (4) Is the App Connector for that segment currently Connected? ZPA Diagnostics (per segment, per user) can surface which of these is failing.
Q3 — MODEL ANSWER
ZIdentity is the identity provider for the Zscaler platform — it manages which services a user can access and what their identity attributes are. ZPA inherits user and entitlement data from ZIdentity rather than maintaining its own user database. This design means a single place (ZIdentity) controls who gets which Zscaler services, keeping user lifecycle management centralised. Adding a ZPA entitlement in ZIdentity triggers a sync that makes the user ZPA-eligible without any action needed in the ZPA Admin Console.

Task 11.1 — Log in as Joe and Identify the Issue

1 Switch ZCC account to Joe

  1. On the Corp: Client PC, open ZCC.
  2. Switch the account to joe@<Student FQDN> with the session password.
  3. Log into ZCC and observe the ZCC dashboard.
  4. Note: Internet Security shows ON but the Private Access section is missing or not available.
ZCC login as Joe — after login, the dashboard will reveal whether ZPA Private Access is available for this identity.

2 Check Joe's entitlements in ZIdentity

  1. In the ZIA Admin Console, navigate to ZIdentity → Users.
  2. Search for Joe and open his user profile.
  3. Navigate to the Service Entitlements or Entitlements tab.
  4. Confirm that ZPA is not listed as an assigned service entitlement — this is the root cause.
ZIdentity Users — find Joe's user profile to inspect his assigned service entitlements.
Joe's Service Entitlements: ZPA is not assigned — this is why Private Access doesn't appear in his ZCC. ZIA is present, ZPA is not.

Task 11.2 — Assign ZPA Service Entitlement to Joe

1 Navigate to Service Entitlements

  1. In ZIdentity, navigate to Policy → Service Entitlements → End User.
  2. Find the ZPA service in the list.
  3. Click on ZPA to view its assigned users.
Service Entitlements — End User view lists ZIA, ZPA, ZDX, and other services. Assign ZPA to Joe here.

2 Assign Joe to ZPA

  1. In the ZPA service, click Assign Users.
  2. Search for and select joe@<Student FQDN>.
  3. Click Save.
ZPA service Assign Users — add Joe to the ZPA service entitlement to grant him private application access.
Joe selected for ZPA entitlement assignment — after saving, ZIdentity syncs the entitlement to ZPA.
ZPA entitlement assigned to Joe — the assignment takes effect on the user's next ZCC re-authentication.

3 Verify Joe can now access private apps

  1. Back in ZCC (logged in as Joe), the Private Access status should now show ON after a policy update or re-login.
  2. Navigate to http://intranet.labtenant.com — the intranet site should load for Joe (if the access policy also covers Joe).
Joe assigned to ZPA service — Private Access will appear in his ZCC after the entitlement propagates.
Intranet accessible as Joe — ZPA service entitlement + access policy together enable private application access.
🧠
MENTAL NOTE
ZPA access requires two separate grants: (1) ZPA Service Entitlement in ZIdentity — controls whether the user can use ZPA at all (Private Access visible in ZCC); (2) ZPA Access Policy — controls which specific app segments the user can reach. If Private Access doesn't show in ZCC, check the Service Entitlement first. If it shows but the app is unreachable, check the Access Policy and App Segment configuration.
