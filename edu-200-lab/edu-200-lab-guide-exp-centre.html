💭 THINK FIRST · USER ENTITLEMENTS

Entitlements in Zscaler determine what services a user can access (service entitlements) and what they can configure in the admin console (administrative entitlements). Getting the identity hierarchy right — department → user group → user — is what makes policies and role-based access work at scale.

  • Bev is placed in the built-in "Zscaler Global Administrators" group rather than a custom group. What is the risk of always granting full super-admin access, even to trusted IT staff?
  • Joe receives the "Executive Insights App" role for ZIA — a predefined read-only role. Why might Zscaler provide pre-built roles rather than requiring admins to always create custom ones?
  • A service entitlement (ZIA) is assigned to Helpdesk L1 group, while an administrative entitlement (ZIA read-only) is also assigned. What is the difference between these two types, and could you have one without the other?
Q1 — MODEL ANSWER
Super-admin access means Bev can change any policy, delete rules, or reconfigure the entire tenant — with no guardrails. In production this violates separation of duties (e.g., someone who configures firewall rules shouldn't also have access to DLP policies). The principle of least privilege requires even IT admins to receive only the access their role demands, not blanket super-admin.
Q2 — MODEL ANSWER
Pre-built roles represent common use cases (read-only analyst, helpdesk, SOC) without requiring every organization to reinvent the permission matrix. They also encode Zscaler's own recommended permission boundaries, reducing the risk of over-permissioning by accident. Custom roles are still available when a pre-built role doesn't fit the exact job function.
Q3 — MODEL ANSWER
A service entitlement grants access to use a Zscaler service (ZIA forwards your traffic). An administrative entitlement grants access to configure that service in the admin console. Yes, you can have one without the other: Joe has both — he uses ZIA (service) and can view-only its analytics (admin). A regular end-user would only have the service entitlement with no admin console access at all.

💭 THINK FIRST · ZCC TRAFFIC FORWARDING

ZCC uses a Forwarding Profile to decide how traffic is sent to Zscaler depending on the network context (on-trusted, off-trusted, VPN). Z-Tunnel 2.0 with Packet Filter-Based driver intercepts all protocols — not just HTTP/HTTPS — enabling full firewall functionality. The Windows App Profile links the forwarding config to specific devices and users.

  • Z-Tunnel 2.0 is required for Firewall functionality. What does Z-Tunnel 1.0 support that makes it insufficient for firewall policy enforcement?
  • You set VPN Trusted Network and Off-Trusted Network to "Same as On-Trusted Network." In what real-world scenario would you want different forwarding behavior for VPN vs. Off-Trusted traffic?
  • The Windows App Profile sets "Install Zscaler SSL Certificate: On." Why is the SSL certificate installation necessary, and what would break if it were missing?
Q1 — MODEL ANSWER
Z-Tunnel 1.0 only forwards HTTP and HTTPS proxy traffic — it works at the application layer for web browsing but ignores other protocols like SSH, DNS, ICMP, or custom TCP/UDP. Z-Tunnel 2.0 operates at the network layer, intercepting all IP traffic regardless of protocol, which is what gives the Zscaler Cloud Firewall visibility and enforcement capability over non-web traffic.
Q2 — MODEL ANSWER
A company might want VPN traffic to use "None" (bypass Zscaler) if the corporate VPN already provides equivalent security inspection. Off-Trusted traffic (home networks, coffee shops) might need a stricter "Tunnel" mode with no local proxy bypass. Setting these the same simplifies the lab, but in production the differences often align with how much the organization trusts each network type.
Q3 — MODEL ANSWER
Zscaler performs SSL inspection by acting as a man-in-the-middle for HTTPS traffic, re-signing certificates with the Zscaler CA. For the browser to trust these re-signed certs, the Zscaler root CA must be installed in the system trust store. Without it, every HTTPS site would show an untrusted certificate error, rendering secure web browsing unusable.

Q2 — MODEL ANSWER
A one-time provisioning key is designed for zero-trust enrollment: the key authorises the initial registration but is consumed during activation, so there's no persistent secret that could be reused to register rogue connectors. A shared password would be a liability — if compromised, any machine could register as a connector in your environment. One-time keys limit the blast radius: a leaked key can only be used once before it expires or is used.
Q3 — MODEL ANSWER
Active sessions through a disconnected App Connector will drop — users lose their private app connections. ZPA handles this through redundancy at the connector group level: multiple App Connectors in the same group share the load, and ZPA automatically fails over to a healthy connector for new connections. For active sessions, ZPA can re-establish them through a surviving connector in the group, depending on the application's session tolerance.

💭 THINK FIRST · TROUBLESHOOT POOR WI-FI

Wi-Fi problems are among the most common causes of degraded digital experience for remote and office workers. ZDX correlates device-level Wi-Fi metrics (signal strength, SNR, channel) with application performance scores, making it possible to confirm Wi-Fi as the root cause before dispatching an IT ticket or replacing hardware.

  • A user reports that Salesforce has been slow since yesterday morning. ZDX shows their Wi-Fi score dropped sharply at that time. What would you check next to confirm Wi-Fi is the cause and not just a coincidence — and what ZDX feature would help?
  • Two users on the same floor both use "CorpNet" Wi-Fi. One has a poor ZDX score, the other is fine. What are the most likely explanations, and how does the Compare feature help distinguish them?
  • ZDX Device Events shows a user switched from CorpNet-5GHz to CorpNet-2.4GHz. How does this event explain a sudden drop in ZDX score, and what would you tell the user to do?
Q1 — MODEL ANSWER
Confirm Wi-Fi causation by examining the Analyze Score breakdown — if Wi-Fi accounts for the majority of the ZDX score degradation, that's strong evidence. Then check Device Events for the same timestamp: a SSID change, roaming event, or sudden drop in Wi-Fi RSSI at that time correlates the cause with the symptom. If other users on the same access point also degraded at the same time, the AP itself may be the issue. ZDX Hop View can also confirm that the first hop (Wi-Fi) shows elevated latency.
Q2 — MODEL ANSWER
Same SSID doesn't mean same access point — one user may be associated with a closer, less congested AP while the other is on an overloaded or distant one. Or one user may be on 5GHz (faster, shorter range) while the other auto-connected to 2.4GHz (slower, longer range). ZDX Compare shows side-by-side Wi-Fi metrics (RSSI, SNR, channel) for both users, making it easy to identify the difference. If RSSI is similar but SNR is poor for one user, channel congestion or interference is likely.
Q3 — MODEL ANSWER
5GHz Wi-Fi offers significantly more bandwidth and less interference than 2.4GHz but has shorter range. When the user's device switched to 2.4GHz (possibly because they moved farther from the AP or the 5GHz radio was temporarily unavailable), their effective bandwidth dropped and latency increased — directly causing the ZDX score decline. The fix: tell the user to move closer to the access point, or manually prefer 5GHz SSIDs on their device. If many users are auto-downgrading, review AP placement or configure SSID steering.

💭 THINK FIRST · TROUBLESHOOT SLOW DNS

DNS resolution is the first step in every HTTPS request. A slow DNS response adds directly to Page Fetch Time, often without the user understanding why the "page is slow" — they just see it as a generally slow experience. ZDX's web probe metrics break Page Fetch Time into its components, making DNS latency visible as a distinct, measurable factor.

  • A user's ZDX score is Poor across all apps. How does "all apps affected" help you narrow down where the problem is — in the device, Wi-Fi, DNS, the Zscaler ZTE, or the applications themselves?
  • ZDX shows DNS Time = 3000ms for a user. What are the two most likely causes of high DNS latency, and how would you distinguish between them?
  • ZDX Copilot uses AI to surface insights. What is the risk of relying solely on Copilot's output rather than verifying the underlying metrics yourself — and in what scenario would Copilot's answer be wrong?
Q1 — MODEL ANSWER
If all apps are affected, the issue is in the shared path — something every application request passes through. App-specific issues (server down, CDN problem) affect only one app. A shared path problem points to the device itself (CPU, memory, disk I/O), the Wi-Fi or LAN connection, DNS resolution, or the Zscaler ZTE the user is routed through. Compare Web Probe metrics across multiple apps: if DNS Time is high for all of them, DNS is the bottleneck. If TCP Connect and Server Response are high but DNS is normal, the ZTE or network path is the bottleneck.
Q2 — MODEL ANSWER
Two likely causes of high DNS latency: (1) The user's DNS server is slow, overloaded, or far away — check the device's configured DNS server and whether ZCC is using Zscaler's DNS resolver or a corporate DNS server; (2) ZIA DNS Control is applying complex processing or the Zscaler service edge nearest the user has a slow DNS path to authoritative servers. Distinguish them by comparing with other users on the same network/ZTE: if only this user has high DNS time, it's device-specific DNS configuration. If all users on the same ZTE show high DNS, it's the ZTE or its upstream DNS.
Q3 — MODEL ANSWER
ZDX Copilot's answer is only as good as the data available to it and the patterns it was trained on. It could be wrong if the issue is subtle (e.g., intermittent DNS that averages out over the probe window) or if it draws from a limited time range that misses the root cause. Always verify Copilot's suggested root cause by checking the underlying probe metrics yourself — Copilot should accelerate your investigation, not replace it. Treat it as a first hypothesis, not a final verdict.
