💭 THINK FIRST · USER ENTITLEMENTS

Entitlements in Zscaler determine what services a user can access (service entitlements) and what they can configure in the admin console (administrative entitlements). Getting the identity hierarchy right — department → user group → user — is what makes policies and role-based access work at scale.

  • Bev is placed in the built-in "Zscaler Global Administrators" group rather than a custom group. What is the risk of always granting full super-admin access, even to trusted IT staff?
  • Joe receives the "Executive Insights App" role for ZIA — a predefined read-only role. Why might Zscaler provide pre-built roles rather than requiring admins to always create custom ones?
  • A service entitlement (ZIA) is assigned to Helpdesk L1 group, while an administrative entitlement (ZIA read-only) is also assigned. What is the difference between these two types, and could you have one without the other?
Q1 — MODEL ANSWER
Super-admin access means Bev can change any policy, delete rules, or reconfigure the entire tenant — with no guardrails. In production this violates separation of duties (e.g., someone who configures firewall rules shouldn't also have access to DLP policies). The principle of least privilege requires even IT admins to receive only the access their role demands, not blanket super-admin.
Q2 — MODEL ANSWER
Pre-built roles represent common use cases (read-only analyst, helpdesk, SOC) without requiring every organization to reinvent the permission matrix. They also encode Zscaler's own recommended permission boundaries, reducing the risk of over-permissioning by accident. Custom roles are still available when a pre-built role doesn't fit the exact job function.
Q3 — MODEL ANSWER
A service entitlement grants access to use a Zscaler service (ZIA forwards your traffic). An administrative entitlement grants access to configure that service in the admin console. Yes, you can have one without the other: Joe has both — he uses ZIA (service) and can view-only its analytics (admin). A regular end-user would only have the service entitlement with no admin console access at all.

Lab 2: Configure User and Administrator Access Entitlements

LAB 2

Scenario: You've secured the Zscaler console and now need to establish a clean foundation for who can sign in and what they can access in Labtenant.com. Two new users — Joe (Support/helpdesk) and Bev (IT/admin) — must be onboarded with the right departments and groups so they inherit the correct entitlements.

Time: ~40 minutes

Task 2.1: Create Departments and User Groups

1 Navigate to Departments

  1. In the Zscaler Admin Console, select the Administration tab.
  2. Go to Identity > Departments.
Administration → Identity → Departments. The Identity menu is your central hub for all user, group, and department management.

2 Create the Support department

  1. Select Add Department.
  2. Enter: Department Name: Support | Description: Support Department
  3. Click Save.
Add Department form for the Support department — name and description fields.

3 Create the IT department

  1. Select Add Department and enter: Department Name: IT | Description: IT Department
  2. Select Save.
IT department creation — confirm both Support and IT appear in the Departments list after saving.

4 Confirm departments and create Helpdesk L1 user group

  1. Confirm that IT and Support appear in the Departments section.
  2. Go to Identity > User Groups.
  3. Select Add User Group.
  4. Enter User Group Name: Helpdesk L1
  5. Click Save.
Navigate to Identity → User Groups to create the Helpdesk L1 group that Joe will inherit entitlements through.
Add Group form — the group name Helpdesk L1 is what links Joe's account to the entitlements configured in Task 2.3.

Task 2.2: Create Users

1 Create User Joe

  1. From the left navigation panel, go to Users.
  2. Click Add User and set:
    • Login ID: joe@<Student FQDN>
    • Name: Joe | First Name: Joe
    • Same As Login ID: Enabled
    • Primary Email: joe@<Student FQDN>
    • Department: Support
  3. Click Assign Groups and select Helpdesk L1.
  4. Click Security Settings tab:
    • Password option: Set By Administrator
    • Password / Confirm Password: Admin-123!
    • Prompt for Password Change After Initial Login: Unchecked
    • Multi-Factor Bypass — Skip Second Factor Authentication: Enabled
    • Skip until: Set as far out as allowed
  5. Navigate to Additional Attributes tab. Under INFORMATION set: SyncZPA: Zscaler
  6. Click Save.
Navigate to Users via the left nav panel under Identity → User Management.
Security Settings: Set password as Administrator, disable the initial password change prompt, and enable MFA bypass for lab purposes.
SyncZPA = "Zscaler" ensures this user is synced into the ZPA identity store so ZPA labs and lab-reset scripts work correctly.

Note: Setting SyncZPA to Zscaler tells the lab environment to sync this user with ZPA so later ZPA labs and lab-reset scripts function correctly. For lab purposes, the second factor of authentication is skipped. Zscaler recommends enabling MFA for all users in production.

2 Create User Bev

  1. Click Add User and set:
    • Login ID: bev@<Student FQDN>
    • Name / First Name: Bev
    • Same As Login ID: Enabled
    • Primary Email: bev@<Student FQDN>
    • Department: IT
  2. Click Assign Groups and select Zscaler Global Administrators.
  3. Click Security Settings tab — configure same password settings as Joe (Admin-123!, MFA bypass enabled, skip as far out as allowed).
  4. Navigate to Additional Attributes — set SyncZPA: Zscaler
  5. Click Save.

Task 2.3: Add Administrative and Service Entitlements

1 Assign ZIA Admin Entitlement to Helpdesk L1

  1. Navigate to Administration > Admin Management > Role Based Access Control > Administrative Entitlements.
  2. Click Zscaler Internet Access.
Administration → Admin Management → Role Based Access Control → Administrative Entitlements — your access control hub for all Zscaler services.
The Administrative Entitlements list — each entry controls who can access that service's admin console. Click ZIA to assign Helpdesk L1.

2 Assign Helpdesk L1 to ZIA with Executive Insights App role

  1. In the ZIA Administrative pane, click Assign Groups.
  2. Select the checkbox for Helpdesk L1 Group.
  3. From the Role drop-down, select Executive Insights App.
  4. Click Next and Assign.
ZIA Administrative pane — click Assign Groups to add Helpdesk L1 with the read-only Executive Insights App role.
Select Groups & Roles — Helpdesk L1 checked, Executive Insights App role selected from the dropdown. Click Next to confirm.

Note: The Executive Insights App role provides read-only access to the Zscaler portal. In production, create a custom role under Administration > Admin Management > Role Based Access Control > Roles for finer control.

3 Assign ZDX Admin Entitlement to Helpdesk L1

  1. Select the Back option next to "Assign Groups" to return to the entitlements list.
  2. Click Zscaler Digital Experience.
  3. Click Assign Groups. Select Helpdesk L1. From the Role drop-down, select ZDX Read-only Admin.
  4. Click Next and Assign.
Return to the entitlements list and click Zscaler Digital Experience to assign Helpdesk L1 read-only ZDX access.
ZDX Select Groups & Roles — Helpdesk L1 gets ZDX Read-only Admin. This pre-built role is purpose-built for Level 1 support monitoring.

4 Assign ZIA Service Entitlement to Helpdesk L1

  1. Navigate to Administration > Entitlements > End User Entitlements > Service Entitlements.
  2. Click Zscaler Internet Access.
  3. Click Assign Groups. Select checkbox for Helpdesk L1.
  4. Click Next and Assign.
Service Entitlements (distinct from Admin Entitlements) — assigning ZIA here means Helpdesk L1 users can actually use the ZIA service via ZCC.
ZIA Service Entitlements — Helpdesk L1 selected. This is what enables Joe's traffic to be forwarded through ZIA when he logs into ZCC.

Task 2.4: Validate Entitlement Access for Users

1 Verify Joe's entitlements in the user profile

  1. Navigate to Administration > Identity > Users.
  2. Click the Edit button (pencil icon) for user Joe.
  3. Navigate to the Entitlements tab and verify both Administrative and Service entitlements are assigned.
Users list — click the pencil icon for Joe to verify entitlements have been applied correctly.
Joe's Entitlements tab confirms: ZDX Read-only Admin (admin), ZIA Executive Insights App (admin), and ZIA (service) — all inherited through Helpdesk L1 group membership.

2 Validate Joe's restricted admin view

  1. On the Corp: Client PC, open a new incognito window in Chrome.
  2. Navigate to the Experience Center: https://console.zscaler.com/
  3. Authenticate as Joe: Username joe@<Student FQDN> | Password: Admin-123!
  4. Confirm that the Administration and Policy tabs are restricted compared to the Student admin user — Joe cannot access Admin Management or Sign-On Policies.
  5. Confirm Joe can only view existing Digital Experience data and cannot create or modify objects.
  6. Click the Account settings icon (top right) to Sign out.
Log in as Joe in an incognito window to verify his restricted admin view without affecting your student session.
Joe's restricted view — only Role Based Access Control and Digital Experience sections are visible. Admin Management, Sign-On Policies, and most configuration areas are hidden.
Sign out Joe using the Account settings icon in the top-right corner before logging in as Bev.

Note: The Helpdesk Level 1 Group has read-only privileges for ZIA and ZDX. Joe inherits this read-only access, which removes access to core administrative areas and limits him to view-only analytics and experience data.

3 Validate Bev's full admin access

  1. In the same incognito window, navigate to https://console.zscaler.com/
  2. Authenticate as Bev: Username bev@<Student FQDN> | Password: Admin-123!
  3. Navigate to Administration > Admin Management > Sign-On Policies.
  4. Confirm the Add Rule button is enabled — Bev has full admin access as a member of Zscaler Global Administrators.
  5. Sign out via the Account settings icon.
Bev's Admin Sign-On Policy view — Add Rule is enabled, confirming full administrative access. Joe's view of this same page would not show the Add Rule button.
Sign out Bev after validation. Next you'll validate Joe's ZIA service entitlement via ZCC on the Client PC.

4 Validate Joe's ZIA service entitlement via Zscaler Client Connector

  1. On the Corp: Client PC, in the Windows taskbar (bottom-right corner), click the Show hidden icons arrow ^.
  2. Select the Zscaler Client Connector icon, then click Open Zscaler.
  3. Authenticate with Joe's account: joe@<Student FQDN> / Admin-123!
  4. Click Sign in.
  5. In the ZCC console, verify that Internet Security is ON for user Joe.
  6. After verifying, log out of Zscaler Client Connector.
Right-click or hover the ZCC tray icon to access the Open Zscaler menu — this is how you authenticate a different user into ZCC.
ZCC login screen — enter Joe's credentials to verify that his ZIA service entitlement allows ZCC to authenticate and forward traffic.
ZCC Internet Security ON for Joe — Service Status ON confirms the ZIA service entitlement is active. Log Joe out after verification; you'll re-authenticate with the student account in Lab 3.
🧠
MENTAL NOTE
Identity hierarchy: Department → User Group → User. Entitlement types: (1) Administrative — controls what you can configure in the admin console; (2) Service — controls what Zscaler services your traffic flows through. Joe needs both: ZIA service entitlement (so ZCC forwards his traffic) and ZIA/ZDX admin entitlements (so he can view analytics). Always set SyncZPA on users who need ZPA.

Student FQDN
Your unique tenant domain, e.g. zs9901.labtenant.com. Used in all login IDs.
Experience Center
console.zscaler.com — unified admin portal for ZIA, ZPA, ZDX, ZIdentity.
ZCC
Zscaler Client Connector — endpoint agent that tunnels all traffic to the ZTE.
Z-Tunnel 2.0
Network-layer tunnel that intercepts all IP protocols. Required for Cloud Firewall.
Service Entitlement
Grants a user access to use a Zscaler service (ZIA, ZPA, ZDX) via ZCC.
Admin Entitlement
Grants a user access to configure a service in the admin console, with a specific role.
SyncZPA
User attribute (value: Zscaler) that syncs the user into ZPA's identity store for policy use.
Forwarding Profile
Defines tunnel type/version and forwarding action per network context (on/off/VPN trusted).
Cloud App Control
Application-layer policy that identifies apps by DPI (e.g., WhatsApp) regardless of port. Separate from Firewall Filtering.
SSL Inspection
Trusted MitM: ZIA decrypts → inspects → re-encrypts using Zscaler CA. Requires Zscaler Root CA trusted on endpoint.
Cert Pinning Bypass
"Do Not Inspect" SSL rule for apps that verify server cert fingerprints (e.g., Threema). Must be above the Inspect All rule.
DLP Engine
Defines what sensitive content looks like (dictionary patterns + thresholds). Paired with a DLP Rule to define the action.
App Connector
ZPA software in the data centre that makes outbound TLS tunnels to ZPA cloud. No inbound firewall rules required.
Provisioning Key
One-time key placed at /opt/zscaler/var/provision_key. Authorises a new connector to join a specific connector group.
App Segment
ZPA object defining a private resource by domain/IP + port. Together with an Access Policy, enables ZPA access.
SIPA
Secure Internet and Private Access — routes ZPA App Connector internet traffic through ZIA for consistent egress IP + policy.
ZDX Score
0–100 composite metric of end-to-end digital experience. 0 = worst, 100 = best. <50 = Poor, 50–79 = Okay, 80+ = Good.
Y-Engine
ZDX ML engine that analyses probe data to identify root-cause factors (Wi-Fi, DNS, TCP, server response) and ranks their contribution.
ZDX Probe
Synthetic background test run by ZCC that measures DNS Time, TCP Connect, Server Response, and Page Fetch Time continuously.
