Lab 16: Troubleshoot Slow DNS

LAB 16

Scenario: Linda Lucas is reporting that all her cloud applications are slow. Using ZDX Copilot and the SDC environment, you'll identify Linda's poor ZDX score across all apps, analyse web probe Page Fetch Time to isolate high DNS Time, and use the Y-Engine to confirm DNS as the root cause.

Time: ~15 minutes
💭 THINK FIRST · TROUBLESHOOT SLOW DNS

DNS resolution is the first step in every HTTPS request. A slow DNS response adds directly to Page Fetch Time, often without the user understanding why the "page is slow" — they just see it as a generally slow experience. ZDX's web probe metrics break Page Fetch Time into its components, making DNS latency visible as a distinct, measurable factor.

  • A user's ZDX score is Poor across all apps. How does "all apps affected" help you narrow down where the problem is — in the device, Wi-Fi, DNS, the Zscaler ZTE, or the applications themselves?
  • ZDX shows DNS Time = 3000ms for a user. What are the two most likely causes of high DNS latency, and how would you distinguish between them?
  • ZDX Copilot uses AI to surface insights. What is the risk of relying solely on Copilot's output rather than verifying the underlying metrics yourself — and in what scenario would Copilot's answer be wrong?
Q1 — MODEL ANSWER
If all apps are affected, the issue is in the shared path — something every application request passes through. App-specific issues (server down, CDN problem) affect only one app. A shared path problem points to the device itself (CPU, memory, disk I/O), the Wi-Fi or LAN connection, DNS resolution, or the Zscaler ZTE the user is routed through. Compare Web Probe metrics across multiple apps: if DNS Time is high for all of them, DNS is the bottleneck. If TCP Connect and Server Response are high but DNS is normal, the ZTE or network path is the bottleneck.
Q2 — MODEL ANSWER
Two likely causes of high DNS latency: (1) The user's DNS server is slow, overloaded, or far away — check the device's configured DNS server and whether ZCC is using Zscaler's DNS resolver or a corporate DNS server; (2) ZIA DNS Control is applying complex processing or the Zscaler service edge nearest the user has a slow DNS path to authoritative servers. Distinguish them by comparing with other users on the same network/ZTE: if only this user has high DNS time, it's device-specific DNS configuration. If all users on the same ZTE show high DNS, it's the ZTE or its upstream DNS.
Q3 — MODEL ANSWER
ZDX Copilot's answer is only as good as the data available to it and the patterns it was trained on. It could be wrong if the issue is subtle (e.g., intermittent DNS that averages out over the probe window) or if it draws from a limited time range that misses the root cause. Always verify Copilot's suggested root cause by checking the underlying probe metrics yourself — Copilot should accelerate your investigation, not replace it. Treat it as a first hypothesis, not a final verdict.

Task 16.1 — Use ZDX Copilot to Surface the Issue

1 Open ZDX Copilot and query for issues

  1. In ZDX, navigate to Copilot (or the AI assistant icon).
  2. Enter a natural-language query such as: Which users are experiencing poor performance?
  3. Review Copilot's response — it should surface Linda Lucas as a user with consistently poor scores across all applications.
ZDX Copilot — AI-powered query interface that synthesises ZDX data to surface insights. Useful as a starting point for investigations.
Copilot response surfaces Linda Lucas — all-apps poor score is a strong signal of a shared-path issue (device, DNS, or network) rather than an app-specific problem.

Task 16.2 — Find Linda Lucas and View ZDX Score

1 Search for Linda Lucas in ZDX Users

  1. Navigate to ZDX → Users and search for Linda Lucas.
  2. Open her device detail page.
  3. Observe that her ZDX score is consistently Poor across all applications.
Linda Lucas in ZDX Users — consistently poor ZDX score across all apps confirms the issue is in the shared path.
Linda Lucas device header — Poor ZDX score persisting over time. All-apps impact rules out application-specific causes.
Consistently poor ZDX score — not an intermittent spike but a persistent degradation, suggesting a configuration-level problem.
All apps poor — when every application shows a poor ZDX score, look at the shared elements: device, DNS, Wi-Fi, or network path.
Copilot application scores — corroborates the all-apps poor finding across multiple app measurements.

Task 16.3 — Analyse Page Fetch Time (DNS Time)

1 View web probe metrics

  1. On Linda's device detail page, navigate to the Web Probe or application metrics section.
  2. View the Page Fetch Time breakdown.
  3. Identify that DNS Time is disproportionately high — approximately 3000ms.
Page Fetch Time breakdown — DNS Time at ~3000ms is the dominant contributor to Linda's poor page load times. TCP Connect and Server Response are normal.
DNS Time 3000ms — a 3-second DNS resolution adds 3 seconds to every page load. This explains why all apps appear slow: every app request shares the DNS bottleneck.
Server response and availability normal — the application server is healthy. The problem is entirely in DNS resolution, not the app or network path.

Task 16.4 — Y-Engine Confirms DNS as Root Cause

1 Click Analyze Score and confirm DNS finding

  1. Click Analyze Score on Linda's ZDX score panel.
  2. Review the factor breakdown — DNS Time should appear as the highest-weighted contributing factor.
  3. The Y-Engine's analysis confirms DNS resolution latency as the primary root cause of Linda's poor ZDX score.
Y-Engine Analyze Score — DNS Time is the top contributing factor to Linda's poor ZDX score. The machine learning confirms what the raw probe metrics showed.

Root Cause: Linda Lucas's consistently poor ZDX score across all applications is caused by high DNS resolution time (~3000ms). The fix: investigate Linda's DNS server configuration — she may be using a slow or misconfigured DNS resolver, or her ZCC DNS path has an issue. Correcting the DNS server assignment or flushing the DNS configuration should resolve the performance degradation.

🧠
MENTAL NOTE
ZDX DNS troubleshooting workflow: (1) ZDX Copilot → surfaces all-apps poor user → (2) ZDX Users → confirm consistently poor score → (3) Web Probe metrics → Page Fetch Time breakdown → DNS Time is the outlier (~3000ms, normal is <100ms) → (4) Server response and availability are normal → DNS is isolated as the bottleneck → (5) Analyze Score → Y-Engine confirms DNS as primary root cause. All-apps poor = shared path problem. DNS is one of the most impactful shared-path bottlenecks because every HTTPS request starts with DNS.
