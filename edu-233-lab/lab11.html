Lab 11: Securing Access to Private Applications from Unmanaged Devices via ZPA Isolation

LAB 11

Scenario: ZPA Browser Access extends private application access to unmanaged devices — no ZCC required. When an Isolation Policy is applied to a Browser Access app, the session is rendered in an isolated container with the configured security controls. Users access internal apps through the User Portal without installing any software, while the organisation enforces clipboard, file transfer, and print controls at the container level.

Time: ~30 minutes
💡 THINK FIRST · ZPA PRIVATE APP ISOLATION

ZPA + Browser Access + Isolation is the complete zero-trust access story: no endpoint agent, no network access, no lateral movement possible — the user sees only a pixel stream of the private app. Even if the unmanaged device is compromised, the application and its data stay inside the isolation container.

  • A contractor accesses Gitlab via ZPA Browser Access + Isolation (TZTE ZPA Render Only profile). They try to download a repository archive from Gitlab. What happens, and why?
  • The TZTE ZPA Render Only profile disables all file transfers, copy/paste, and printing. This makes the app essentially read-only. When is this the right security posture for a private application?
  • A new private application is added to ZPA. The admin creates an Isolation Policy but forgets to configure the application for Browser Access. What happens when an unmanaged device user tries to access the app from the User Portal?
Q1 — MODEL ANSWER
The download is blocked. The TZTE ZPA Render Only profile has "File Transfer From: Isolation to local computer" disabled. The Gitlab download request executes inside the isolation container — the file is downloaded to Protected Storage — but cannot be transferred to the unmanaged device's local disk. The user sees the file in Protected Storage but has no download path to their device.
Q2 — MODEL ANSWER
Read-only isolation is appropriate for: (1) sensitive internal tools (code repos, HR systems, financial dashboards) where viewing is required but exfiltration must be prevented, (2) unmanaged/contractor devices where the organisation has no endpoint control and cannot verify device hygiene, (3) high-value intellectual property where even copy-paste creates exfiltration risk.
Q3 — MODEL ANSWER
The app would not appear in the User Portal if Browser Access is not configured. Browser Access requires the application to be published with an external-facing domain (a Browser Access hostname), which generates the CNAME entry. Without Browser Access, the app is only accessible to ZCC-enrolled devices via the tunnel. The Isolation Policy would have no effect since the app is not reachable via browser without it.

Task 11.1: Review the ZPA Isolation Configuration for Private Apps

1 Verify Browser Access for Gitlab

  1. In the ZPA Admin Portal, navigate to Resource Management › Application Management › Browser Access.
  2. Click the expand arrow (›) next to gitlab.thezerotrustexchange.com.
  3. Confirm the Canonical Name (CNAME) — this proves the app is published for browser-based access.
  4. Verify the Certificate is a wildcard cert for *.thezerotrustexchange.com.

2 Review the Isolate - Render Only Isolation Policy

  1. Navigate to Policy › Isolation Policy.
  2. Click the pencil icon next to Isolate - Render Only.
  3. Verify: Isolation Profile: TZTE ZPA Render Only, Criteria: Application Segments = Gitlab.
  4. Click Cancel.

3 Verify the User Portal and Gitlab Portal Link

  1. Navigate to Resource Management › User Portal › User Portals.
  2. Expand The Zero Trust Exchange User Portal and confirm its configuration.
  3. Navigate to Portal Links and verify the Gitlab link: URL = https://gitlab.thezerotrustexchange.com.

Task 11.2: Test Isolation for a Private Application

Warning: Ensure you are logged out of Zscaler Client Connector before this task to simulate an unmanaged device.

1 Verify CNAME Resolution and Access Gitlab via User Portal

  1. On the Client PC VM, open Command Prompt and run: nslookup gitlab.thezerotrustexchange.com.
  2. Confirm the CNAME chain resolves to a *.h.p.zpa-app.net address — confirming ZPA routing.
  3. Navigate to https://userportal.thezerotrustexchange.com.
  4. Log in with the Marketing Username: <prefix>-MKT@thezerotrustexchange.com and Session Password.
  5. Click the Gitlab tile.
  6. Gitlab opens in an isolation session. Log in with:
    • Username: z-se
    • Password: SE@Zscal3r
  7. Navigate through the Gitlab interface. Confirm that copy/paste, file downloads, and printing are disabled.

2 Verify the TZTE ZPA Render Only Profile Security Settings

  1. In the ZPA Admin Portal, navigate to Configuration & Control › Browser Isolation › Isolation Profiles.
  2. Click the pencil icon next to TZTE ZPA Render Only.
  3. Navigate to the Security tab.
  4. Confirm: Copy & Paste (both directions): Disabled, File Transfers (both directions): Disabled, Allow Printing: Disabled, Read-Only Isolation: Enabled.
  5. Click Cancel.

MENTAL NOTE
ZPA Browser Access + Isolation = zero-footprint access to private apps. The CNAME routes browser requests through ZPA App Connectors. The Isolation Policy wraps the session in a remote browser. The TZTE ZPA Render Only profile enforces maximum data control: no clipboard, no files, no printing. The endpoint sees only pixels.
