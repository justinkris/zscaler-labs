Lab 3: Managing One-click Smart Browser Isolation

LAB 3

Scenario: Smart Browser Isolation uses Zscaler's AI/ML models to identify potentially malicious web pages in real time and render them in an isolated container — without needing a URL category policy. This provides a safety net for uncategorised or newly malicious domains that fall outside explicit blocking rules.

Time: ~20 minutes
💡 THINK FIRST · AI-DRIVEN ISOLATION

Smart Browser Isolation fills the gap between known-bad (block) and known-good (allow). For the large grey zone of uncategorised or newly suspicious sites, AI/ML isolation provides a safe pixel-stream without blocking productivity.

  • Smart Browser Isolation uses AI/ML to identify suspicious content. How is this different from a URL category-based Isolate policy, and what threat class does it specifically address?
    Q1 — MODEL ANSWER
    A URL category rule triggers on known categories (e.g., News, Finance). Smart Browser Isolation triggers on AI/ML signals — page content, hosting patterns, reputation scores — regardless of category. It specifically targets zero-day phishing pages, newly registered domains, and pages exhibiting exploit-kit behaviour that haven't yet been categorised.
  • The Smart Isolate feature uses the "TZTE Render Only" profile, which has Read Only enabled. Why is read-only mode the right default for suspected malicious content?
    Q2 — MODEL ANSWER
    Read-only mode prevents the user from entering credentials or data into a potentially malicious form while still allowing them to see the content. If the site turns out to be benign, the impact is minor inconvenience. If it is a phishing page, read-only mode prevents credential harvesting even if the user tries to submit data.
  • A user reports that a legitimate news site is being loaded in isolation as "suspicious." What is the likely cause, and how would you investigate?
    Q3 — MODEL ANSWER
    The AI model may have flagged the site based on hosting provider, domain age, or content patterns that match malicious templates. Check Analytics › Web Insights logs for the site and look at the "Policy Action" column — it should show "Smart Isolate." Then review the URL category and Zscaler's threat intelligence for the domain. If it is a false positive, you can add a URL bypass exception in the SSL Inspection or URL Filtering policy.

Task 3.1: Review Smart Browser Isolation Configuration

1 Review the Smart Isolate Setting

  1. In the ZIA Admin Portal, go to Policy › Secure Browsing › Smart Isolate.
  2. Verify that Enable AI/ML based Smart Browser Isolation is toggled on.
  3. Confirm that the Browser Isolation Profile is set to TZTE Render Only.

Note: The TZTE Render Only profile has Read Only enabled in its Security settings, which prevents text input into isolated pages.

Task 3.2: Test Smart Browser Isolation

1 Access a Suspicious Site in Isolation

  1. On the Client PC VM, open a browser and navigate to https://www.fakebankstatement.co.uk.
  2. Observe that the site loads with the isolation banner at the top: "Heads up, you've been redirected to Browser Isolation!"
  3. Notice the orange notification: "You are accessing this webpage in read-only mode."
  4. Click the Zscaler icon in the bottom-right corner of the browser window to reveal the isolation URL bar.
  5. Click the menu icon (≡) to view additional information, including the Isolation Profile in use: TZTE Render Only.

2 Verify Read-Only Restriction

  1. On the isolated site, click ORDER NOW and then click Bank Statements Order Form.
  2. Scroll to the Email field and attempt to type text.
  3. Confirm that you cannot enter any text — the page is in read-only mode.

Note: When a page loads in isolation, the browser renders a pixel-stream of the actual website from a remote container. Any malicious scripts run in the container, not on your endpoint.
