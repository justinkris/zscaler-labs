Lab 10: Securing Access to SaaS Apps from BYO/Unmanaged Devices via ZPA

LAB 10

Scenario: Organisations increasingly need to provide contractors and unmanaged devices with access to sanctioned SaaS applications. ZPA User Portal 2.0 with Browser Isolation provides frictionless, agentless access: the user logs into the User Portal from any browser, clicks an app tile, and the SaaS app opens in an isolated browser session with all security controls enforced.

Time: ~40 minutes
💡 THINK FIRST · AGENTLESS SAAS ISOLATION VIA ZPA

ZPA User Portal 2.0 + BI is the "zero-footprint" BYOD solution. No endpoint agent, no VPN, no corporate image required. The device only needs a browser. All access is isolated — exfiltration controls apply even on unmanaged hardware.

  • The ZPA Isolation Profile for BYOD has "Forward Internet Traffic via ZIA" enabled. Why is this setting necessary when the SaaS app (M365) is accessed via ZPA Browser Access?
    Q1 — MODEL ANSWER
    ZPA Browser Access routes the SaaS app connection through ZPA App Connectors for private/corporate apps. But inside the M365 isolated session, the user's isolated browser may load internet resources (CDN scripts, fonts, API endpoints). "Forward Internet Traffic via ZIA" ensures those outbound internet calls from the isolation container are inspected by ZIA, maintaining full policy coverage even inside the isolated SaaS session.
  • The BYOD Access Policy allows Client Types: Web Browser AND Client Connector. Why include both, and what happens if a managed device user (with ZCC installed) clicks the same M365 tile in the User Portal?
    Q2 — MODEL ANSWER
    A managed device user (ZCC installed) accesses the same M365 tile and gets isolation applied — the Isolate BYOD policy matches both Web Browser and Client Connector client types. This is intentional: even managed device users accessing M365 via the User Portal get the same BYOD Restricted isolation controls. If you want to differentiate, create separate policies scoped by client type.
  • The lab uses IdP Conditional Access to block direct M365 access from unmanaged devices. Why is this the right enforcement point rather than a ZPA policy?
    Q3 — MODEL ANSWER
    ZPA policies control ZPA-mediated access (through the User Portal / Browser Access). They cannot control direct browser access to M365 (which goes directly to Microsoft). IdP Conditional Access is enforced by the IdP (e.g., Azure AD) — it checks the source IP before allowing authentication. If the request comes from a Zscaler IP (proving it went through ZPA/ZIA), access is granted. If it comes from a random ISP IP, access is denied.

Task 10.1: Review the ZPA Isolation Profile for BYOD

1 Log In to ZPA and Review the TZTE BYOD Restricted Profile

  1. In a browser, go to the ZPA Admin Portal: https://admin.private.zscaler.com.
  2. Log in with:
    • Admin User ID: <prefix>-ADMIN@thezerotrustexchange.com
    • Session Password: <Session Password>
  3. Navigate to Configuration & Control › Browser Isolation › Isolation Profiles.
  4. Click the pencil (edit) icon next to TZTE BYOD Restricted.
  5. On the Company Settings tab, scroll to TRAFFIC FORWARDED TO ZIA.
  6. Confirm Forward Internet Traffic via ZIA is Enabled, and note the Organisation ID and Cloud Name.
  7. Click Cancel.

Task 10.3: Review Access and Isolation Policies

1 Review the BYOD Apps Access Policy

  1. Navigate to Policy › Access Policy.
  2. Click the pencil icon next to BYOD Apps Access.
  3. Review the Criteria: Application Segments (ServiceNow, Salesforce, M365), Client Types (Web Browser, Client Connector).
  4. Click Cancel.

2 Review the Isolate BYOD Policy

  1. Navigate to Policy › Isolation Policy.
  2. Click the pencil icon next to Isolate BYOD.
  3. Verify: Rule Action: Allow Isolation, Isolation Profile: TZTE BYOD Restricted.
  4. Review Criteria: Application Segments (ServiceNow, M365), Client Types (Web Browser).
  5. Click Cancel.

Task 10.5: Review DNS (CNAME) Configuration

1 Find the CNAME for the M365 Browser Access App

  1. Navigate to Resource Management › Application Management › Browser Access.
  2. Click the pencil icon next to office.thezerotrustexchange.com.
  3. Click Cancel, then click the expand arrow (›) next to the application.
  4. Note the Canonical Name (CNAME) shown. This is what needs to be configured in the domain registrar for office.thezerotrustexchange.com.

Note: In your own environment, you would create a CNAME record for your SaaS alias hostname (e.g., office.yourdomain.com) pointing to the ZPA CNAME shown here. This routes browser traffic through ZPA for isolation enforcement.

Task 10.6: Test the BYOD SaaS Access Flow

1 Simulate an Unmanaged Device — Direct Access Blocked

  1. On the Client PC VM, open Zscaler Client Connector and click Log Out, then Continue.
  2. Open a new browser tab and go to https://www.office.com/login.

2 Access M365 via the User Portal in Isolation

  1. Navigate to https://userportal.thezerotrustexchange.com.
  2. Log in with the Contractor Username and Session Password.
  3. Click the M365 tile in the portal.
  4. ZPA redirects to the IdP for authentication. Log in again with the Contractor Username and Session Password.
  5. Microsoft 365 loads in an isolation session — note the TZTE BYOD Restricted watermark.
  6. Confirm the isolation controls: no copy/paste, file transfers restricted.
