Lab 2: Managing Browser Isolation Basics

LAB 2

Scenario: Browser Isolation configuration involves two key steps: creating Isolation Profiles and attaching them to policies. An Isolation Profile determines the region, security controls (upload/download, clipboard, print, watermark), root certificates, cookie persistence, PAC file, and user experience. In this lab you will review the pre-configured profiles and isolation banners in the lab tenant.

Time: ~20 minutes
💡 THINK FIRST · ISOLATION PROFILES

Isolation Profiles are reusable templates that define HOW a user is isolated. Policies define WHEN isolation applies. Separating the two lets you reuse a single profile across many policies without duplicating security control settings.

  • An Isolation Profile includes Region Selection, Security Controls, Root Certificates, Cookie Persistence, PAC File Configuration, and User Experience Settings. Why is region selection a performance consideration rather than a security one?
  • Isolation Profiles created in the ZIA Admin Portal can only be referenced in ZIA policies, and profiles created in the ZPA Admin Portal can only be referenced in ZPA policies. What problem would arise if this boundary did not exist?
  • Cookie Persistence controls whether cookies survive across isolation sessions. What is the security risk of enabling cookie persistence, and when might you accept that risk?
Q1 — MODEL ANSWER
Region selection determines which Zscaler data centre spins up the isolated container. A container in Tokyo serving a user in London adds significant round-trip latency to every pixel-streamed interaction. Security controls (clipboard, uploads, etc.) are enforced regardless of region — region only affects proximity and therefore user experience.
Q2 — MODEL ANSWER
ZIA and ZPA policy engines are separate. Allowing cross-portal profile references would mean a ZPA Isolation Policy could inadvertently invoke a ZIA-managed profile with different security controls than intended, creating inconsistent enforcement. The boundary enforces administrative separation between internet and private-app isolation.
Q3 — MODEL ANSWER
Cookie persistence allows the isolated session to remember login state across multiple isolation sessions — useful for SaaS apps where re-authentication is disruptive. The risk is that a compromised or shared isolated container could reuse another user's session cookies. Organisations typically enable it only for trusted, authenticated SaaS applications.

Task 2.1: Review Isolation Profiles and Isolation Banners

1 Explore Isolation Profiles

  1. In the ZIA Admin Portal, navigate to Administration › Browser Isolation.
  2. Scroll through the list of pre-configured Isolation Profiles.
  3. Click the eye icon next to TZTE Render Only to review its settings.
  4. On the Company Settings tab, note the PAC file URL and the Root Certificates applied to the isolation browser.
  5. Click Next to review Security settings, then Regions.
  6. Click Next to reach the Isolation Experience tab. Note the Isolation Banner referenced by this profile.
  7. Click Close.

Note: Isolation Profiles created in the ZIA Admin Portal can only be used in ZIA policies. Profiles created in the ZPA Admin Portal can only be used in ZPA Isolation Policies.

2 Review Isolation Banners

  1. Click the Isolation Banner tab in Administration › Browser Isolation.
  2. Review the configured banners: Default, Default TZTE, and TheZeroTrustExchange Up Down Block.
  3. Note that banners are referenced by each Isolation Profile under the Isolation Experience section.
