Lab 6: Inspecting Cloud Browser Isolation Logs

LAB 6

Scenario: When a URL Filtering or Cloud App Control policy isolates a session, ZIA logs two location categories: Road Warrior (the user's request to ZIA) and Cloud Browser (the isolated container's subsequent requests). Understanding this dual-log model is essential for accurate troubleshooting and billing analysis.

Time: ~25 minutes
💡 THINK FIRST · ISOLATION LOG ANALYSIS

Every isolated session produces at least two log entries: one from the user's device (Road Warrior) showing the isolation trigger, and one or more from the Cloud Browser container (Cloud Browser location) showing the actual page traffic. Filtering only on Road Warrior hides the full picture.

  • An isolated bbc.com session produces entries with Location: Road Warrior AND Location: Cloud Browser. Why are there two location types, and what does each represent?
    Q1 — MODEL ANSWER
    Road Warrior entries represent the user's initial HTTP request from their device to ZIA. When the URL Filtering policy matches "Isolate," ZIA responds by launching an isolated container and redirecting the browser to it. Cloud Browser entries represent the traffic generated by that isolated container as it fetches bbc.com and subsequent page resources. They appear under the Cloud Browser location because the container itself is a Zscaler-managed instance.
  • In the Road Warrior log entry for bbc.com, the Policy Action shows "Allowed for Isolation." Why is the action "Allowed" and not "Isolated" — what does this wording tell you about how Zscaler processes the request?
    Q2 — MODEL ANSWER
    "Allowed for Isolation" means ZIA allowed the traffic to proceed — but into an isolated container rather than directly to the browser. The word "Allowed" distinguishes it from "Blocked" actions. The isolation itself is a form of conditional access, not a block. This terminology is important for interpreting reports: "Allowed for Isolation" is not a policy miss — it is the intended action.
  • A user reports they visited bbc.com but the logs show only Road Warrior entries and no Cloud Browser entries. What does this indicate?
    Q3 — MODEL ANSWER
    The absence of Cloud Browser entries typically means the isolated session never fully established. Possible causes: the isolation container failed to launch (check ZIA health), the user closed the tab immediately before the container connected, or SSL inspection was not active and the request was passed through uninspected.

Task 6.1: Review Isolation Log Details

1 Filter Logs for Road Warrior Isolation Events

  1. In the ZIA Admin Portal, go to Analytics › Web Insights.
  2. Click the Logs tab.
  3. Under Timeframe, select Current Day.
  4. Under Select Filters, click Add Filter and type loc to find Location.
  5. Select Road Warrior and click Done.
  6. Click Add Filter again, type URL Search, change Domain to Host, select Contains, and type bbc.com.
  7. Click Apply Filters and review the log entries. Note the Policy Action: Allowed for Isolation column.

Note: Log entries may take a few minutes to appear after your Lab 4 test. If no entries show for bbc.com, re-run the Lab 4 test and wait 2-3 minutes.

2 Add Cloud Browser to the Location Filter

  1. Click the Location filter and add Cloud Browser to the selection (keeping Road Warrior selected).
  2. Click Apply Filters.
  3. Review the expanded log. Cloud Browser entries show the traffic generated by the isolated container itself — all the page resources bbc.com loaded.

Note: The combined Road Warrior + Cloud Browser view shows the complete isolation flow: the trigger request (Road Warrior) and all subsequent container activity (Cloud Browser). Cloud Browser entries show Action: Allowed because they are normal outbound requests from the container.
