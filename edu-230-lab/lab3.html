Lab 3: DNS Security

LAB 3

Scenario: DNS is a critical attack vector — it is often the first step in malware communications, phishing, and C2 callbacks. This lab configures DNS control policies to block malicious and unwanted domains at the DNS layer, before any TCP connection is established.

Time: ~20 min
Phase: ZIA
Tasks: 3
💭 THINK FIRST · LAB 3

DNS blocking intercepts threats before any network connection is established — it is the cheapest layer of defence in terms of performance overhead and one of the highest-signal sources for threat intelligence.

  • Why is DNS-layer blocking more effective than URL filtering alone for blocking malicious domains?
  • What does the Domain Inclusion wildcard * in the App Profile accomplish?
  • What risks remain even after implementing DNS-layer blocking?
Q1 — MODEL ANSWER
DNS blocking prevents resolution before any TCP/IP connection is established — the client never sends a packet to the malicious server. URL filtering operates after a connection is already formed. DNS blocking also catches non-web protocols that use the same malicious domain.
Q2 — MODEL ANSWER
It tells Zscaler Client Connector to forward ALL DNS queries (for all domains) through ZIA's DNS resolver. Without this, only specific listed domains are subject to DNS policies — the wildcard ensures universal DNS inspection coverage.
Q3 — MODEL ANSWER
Attackers can use DNS-over-HTTPS (DoH) or hardcoded IP addresses to bypass DNS controls. Fast-flux domains also change IPs rapidly. These are addressed by blocking DoH endpoints via Cloud Firewall and using IP reputation policies.

Task 3.1: Configure App Profile

Enable domain inclusion in the App Profile so that all DNS queries from the endpoint are forwarded through ZIA.

  1. In ZIdentity > Zscaler Client Connector, go to App Profiles > Windows.
  2. Edit the HandsOnLab policy and go to the DNS tab.
  3. Under Domain Inclusion, type * and press Enter to create the wildcard entry.
  4. Click Save.
  5. On the endpoint, open Zscaler Client Connector and click More > Update Policy.

Task 3.2: Configure DNS Control Policy

1 Pre-test

  1. Open a command prompt and run: nslookup www.gambling.com 8.8.8.8
  2. Confirm the domain resolves successfully (no policy in place yet).

2 Create the DNS Filtering Rule

  1. In the ZIA Admin Portal, go to Policy > Firewall > DNS Control.
  2. Click + Add DNS Filtering Rule and configure:
    • Rule Order: 1
    • Name: Block Gambling
    • DNS Application tab — Request Categories + Response Categories: Gambling
    • Action: Block
  3. Click Save then Activate.

3 Post-test

  1. Run ipconfig /flushdns to clear the DNS cache.
  2. Run nslookup www.gambling.com 8.8.8.8 — the query should now time out.
  3. Verify a legitimate domain still resolves: nslookup www.sdxcentral.com 8.8.8.8

Task 3.3: DNS Insight Logs

  1. In the ZIA Admin Portal, go to Analytics > Insights > DNS Insights > Logs.
  2. Set Timeframe to Current Day.
  3. Add a filter: Rule Name: Block Gambling.
  4. Click Apply Filters and review the blocked gambling domain requests (e.g., www.888.com).

MENTAL NOTE
DNS filtering operates before any connection — it is your fastest and lowest-overhead layer of block. The wildcard * in Domain Inclusion is what makes ZIA see every DNS query; without it, DNS policy is a partial view.
