Lab 8: Deception-Based Active Defense

LAB 8

Scenario: Zscaler Deception deploys fake decoy assets throughout the environment. Any attacker interaction with these hidden decoys is a high-confidence breach indicator. In this lab you simulate attacker reconnaissance against public decoys, then investigate the resulting alerts in the Deception Admin Console.

Time: ~30 min
Phase: ZIA
Tasks: 2
💭 THINK FIRST · LAB 8

Deception fills the gap that perimeter controls leave open: lateral movement after a breach. Decoys have zero legitimate traffic — any interaction is a high-confidence, zero-false-positive breach indicator.

  • Why is "any interaction with a decoy" a higher-confidence breach indicator than a traditional IDS alert?
    Q1 — MODEL ANSWER
    Legitimate users and applications have no reason to access decoy assets. Any interaction definitively indicates unauthorized exploration — reconnaissance, lateral movement, or credential harvesting. IDS alerts fire on legitimate traffic patterns that resemble attacks, generating false positives.
  • What data does Zscaler Deception capture during a recon event that is valuable to a SOC analyst?
    Q2 — MODEL ANSWER
    Deception captures: attacker IP, decoy name and type, recon method (web login attempt, network scan, etc.), request URI, entered credentials, timestamp, and optionally the source user identity if ZCC is running. This provides attribution data without requiring the attacker to succeed.
  • How does Deception complement ZIA and ZPA in a layered defence strategy?
    Q3 — MODEL ANSWER
    ZIA and ZPA focus on north-south traffic (user-to-internet and user-to-private-apps). Deception addresses east-west lateral movement after a breach. Together: ZIA/ZPA prevent initial compromise; Deception detects post-breach movement before the attacker reaches real assets.

Task 8.1: Generate Recon Activity

SDC Lab — Read-Only: Switch to the Solutions Demo Center (SDC) tenant. Your MKT department credentials are in the Resources panel of your lab environment.
  1. On Corp: Client PC, log out of Zscaler Client Connector.
  2. Log back in with the SDC Marketing department user (MKT_department_username + Session_Password).
  3. Open a browser and navigate to one of the public-facing decoys:
    • http://wordpress.thezerotrustexchange.com/
    • http://helpdesk.thezerotrustexchange.com/
  4. At the login form, enter traceable fake credentials:
    • Username: sdcuser_<youridentifier>__
    • Password: SDCaccess
  5. Click Sign On. The page will appear to clear and re-prompt — this is expected. The decoy has logged your credentials and source IP.
    Note: In a real scenario this triggers an immediate SOC alert. Any interaction with a decoy is a zero-false-positive breach indicator.

Task 8.2: Investigate Deception Alerts

SDC Lab — Read-Only: Use your Admin credentials for the SDC tenant (Admin_User_ID + Session_Password).

1 Access the Deception Admin Console

  1. Go to https://sdc.zslogin.net → click Solutions Demo Center Login.
  2. Log in with your Admin_User_ID and Session_Password.
  3. Click the Deception tile and accept the End User Subscription Agreement.

2 Search for Your Recon Event

  1. Click Investigate in the main menu → select ThreatParse.
  2. Set the time filter to Last 10 minutes.
  3. Search: decoy.name is "moveit.thezerotrustexchange.com" (substitute the decoy you visited).
  4. Also try: web.method · web.host · web.request
  5. Press Enter to run, click an attacker entry to see the Event Summary.
  6. Click View Extended Details for forensic information.

3 Review Event Logs

  1. Click the Event Logs tab and review the detailed log entries.
  2. Sort or search by recon post_data user to find the entry for the username you entered.
    Tip: If this column is not visible, click the chevron icon at the top-left of the table to add it from the field selection menu.
