💭 THINK FIRST · LAB 2

SSL inspection gives Zscaler visibility into 90%+ of modern web traffic — but breaks apps that hard-code expected certificate hashes. Understanding this trade-off is essential before deploying SSL inspection to production.

  • Why does Z-Tunnel 2.0 with the Packet Filter Based driver matter for Firewall functionality?
  • What is certificate pinning and why does it cause application failures when SSL inspection is enabled?
  • How would you proactively identify pinned-certificate apps before rolling SSL inspection to production?
Q1 — MODEL ANSWER
Z-Tunnel 2.0 with Packet Filter Based driver allows ZIA to enforce Cloud Firewall policies on all traffic from the endpoint, not just HTTP/HTTPS. Without this combination, non-web protocols and ports bypass ZIA inspection entirely.
Q2 — MODEL ANSWER
Certificate pinning is where an application validates the server certificate against a hard-coded expected value. When Zscaler presents its intermediate CA certificate instead, the pinned check fails — triggering the "failed to connect" errors seen in Signal and Adobe.
Q3 — MODEL ANSWER
Pilot on a small user group, review Web Insights logs for "Dropped due to failed client SSL handshake", consult vendor documentation for known pinned apps, and maintain an SSL bypass list that grows as issues are discovered.

Task 2.1: Configure Forwarding Options

Create a Forwarding Profile and App Profile in Zscaler Client Connector so all traffic is sent through ZIA via Z-Tunnel 2.0.

1 Create a Forwarding Profile

  1. Log in to ZIdentity and open Zscaler Client Connector.
  2. Navigate to Administration > Forwarding Profile > Add.
  3. Name the profile HandsOnLab and configure:
    • Windows Driver: Packet Filter Based
    • ZIA On-Trusted Network: Tunnel
    • Tunnel version: Z-Tunnel 2.0
    • VPN / Off-Trusted: Same as On-Trusted
  4. Click Save.

2 Create an App Profile

  1. Go to App Profiles > Windows > Add Windows Policy.
  2. Configure: Name HandsOnLab · Rule Order 1 · Forwarding Profile HandsOnLab · Install Zscaler SSL Cert On · User Groups All.
  3. Click Save.
  4. On the endpoint, open Zscaler Client Connector — verify Service Status ON for both Private Access and Internet Security.
  5. Click More > Update Policy.
Important: The Packet Filter Based driver is required for Cloud Firewall functionality on Windows. Without Z-Tunnel 2.0, non-web protocols bypass ZIA inspection entirely.

Task 2.2: Enable SSL Inspection for All Destinations

  1. Verify no SSL inspection is active: visit x.com and check the certificate — the issuer should show Let's Encrypt.
  2. In the ZIA Admin Portal, go to Policy > Web > ACCESS CONTROL > SSL Inspection.
  3. Click + Add SSL Inspection Rule:
    • Rule Order: 1
    • Name: Enable SSL
    • Action: Inspect
  4. Click Save then Activate.
  5. Verify: revisit x.com — the certificate issuer should now show Zscaler Intermediate Root CA.

Task 2.3: Enable SSL Exemption

Some categories (such as healthcare) should not be decrypted for privacy compliance. Create a bypass rule above the inspection rule.

  1. In the ZIA Admin Portal, go to Policy > SSL Inspection and click + Add SSL Inspection Rule.
  2. Configure:
    • Rule Order: 1 (above Enable SSL)
    • Name: SSL Bypass for Health Care
    • URL Categories: Health
    • Action: Do Not Inspect
  3. Click Save then Activate.
  4. Verify: visit health.com — the certificate should now show R3 / Let's Encrypt (not Zscaler).

Task 2.4: Verify Zscaler CA Installation

Confirm the Zscaler Root Certificate is installed in the Windows certificate trust store.

  1. Press Win + R, type certmgr, and press Enter.
  2. Expand Trusted Root Certification Authorities > Certificates.
  3. Confirm the Zscaler Root CA certificate is present.
  4. Alternatively, in Chrome navigate to chrome://certificate-manager/ > Local certificates > View imported from Windows — confirm Zscaler Root CA is listed.

Task 2.5: Certificate Pinning Error

Some applications use certificate pinning and will fail when Zscaler intercepts their TLS connections. Learn to identify and resolve these issues.

1 Signal Messenger

  1. Download Signal from signal.org and attempt to install — you should see: "Something went wrong! Failed to connect to server".
  2. In the ZIA Admin Portal, add a new SSL Inspection Rule:
    • Rule Order: 1
    • Name: Enable Signal
    • Cloud Applications: Signal (under Instant Messaging)
    • Action: Do Not Inspect
  3. Save and Activate.
  4. End the Signal task in Task Manager, then re-install Signal — it should now succeed.
  5. In Analytics > Web Insights > Logs, filter by SSL Inspected + Bypassed Transaction — find the "Dropped due to failed client SSL handshake" entries for signal.org.

2 Adobe Acrobat

  1. Download Adobe Reader — you should see: "Application initialization error".
  2. Create a custom URL category "Adobe URL List" with Adobe hosted domains from the Adobe Product Support page.
  3. Add an SSL Inspection Rule: Name Enable Adobe · URL Category Adobe URL List · Action Do Not Inspect.
  4. End Adobe Download Manager in Task Manager, then re-install Adobe Reader — it should now succeed.
