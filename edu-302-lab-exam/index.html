TASK 3 Configure ZIA Policy ▾

Configure policies that provide protection against security threats and control access to web-based applications based on department membership.

Configuration Requirements
  1. SSL inspect all traffic
  2. Exempt some web traffic from SSL inspection:
    • Health and Government URL Categories (to address privacy concerns)
    • A custom list of URLs that may be maintained by adding/removing URLs to respond to SSL inspection-related issues
    • Ensure all other policies are enforced for traffic that is not SSL inspected
  3. For all departments except IT Security, block Executable file downloads from all URL categories
  4. Caution users in the IT Security department before the download of Executable files
  5. Finance users should be blocked for uploading all files except Microsoft Office files
  6. All supported file types must be sandbox inspected. While files are being inspected, users are allowed to download them — except files in the following URL categories which must be quarantined and inspected before allowing download if determined safe:
    • Illegal or Questionable
    • Security
    • Miscellaneous
  7. All traffic must be firewall inspected. Add specific rules to allow:
    • An approved list of network services including SSH, Kerberos, and pc Anywhere for users in the IT Security department
    • The customer's IT Security team needs access to www.eicar.org site file downloads; block eicar.org for all other users
    • Block all users from Social Networking and Online Shopping, except employees in the Marketing department who must be allowed to view and post to LinkedIn, Twitter, and Facebook
    • Office 365: Forward all O365 traffic and enable Microsoft Recommended One-Click configuration
    • Enable advanced policy settings for URL & Cloud App Controls:
      1. Suspicious New Domains Lookup
      2. AI/ML based Content Categorization
      3. Embedded Sites Categorization
      4. Enforce SafeSearch
  8. All users need access to all URL Categories, except a default rule to block Legal Liability URL Categories:
    • Adult Material, Drugs, Gambling, Illegal or Questionable, Militancy/Hate and Extremism
    • A custom URL category for any additional URLs deemed off-limits by Safemarch security policies (Hint: add any arbitrary URLs as part of this new custom URL Category)
  9. Allow login to only the Microsoft O365 Safemarch corporate tenant and prevent personal login to Microsoft O365 products like OneDrive and Outlook:
    • Tenant Profile Name: Safemarch Microsoft O365
    • Tenant Directory: deadbeef-0001-ffee-0001-314159265358
    • Office 365 Tenant: safemarch.com
  10. Block access to all streaming media except the Safemarch YouTube channel. Legacy rule for reference:
    • Rule Name: Allow Safemarch YouTube Channel
    • Policy: Allow
    • URL Category: Safemarch YouTube Custom Category
      • Sites: ytimg.com, .ytimg.com
      • Regex: www\.youtube\.com/.UCSylwuqCXM_W13ARfzASm3Q
      • Regex: youtube\.com/videoplayback
      • Regex: www\.youtube\.com/crossdomain\.xml
      • Regex: youtube\.com/generate
    To retrieve the YouTube Channel ID, see the Configuration Hints below.
Testing Requirements
  • Check Web Insights logs and certificate info on various URLs to confirm SSL inspection/exemptions
  • Verify access to a legal liability URL (e.g. www.freeslots-machines.com) is blocked
  • Verify access to online shopping sites is blocked for all users
  • Verify access to social networking sites is blocked for users NOT in the Marketing department
  • Check Firewall logs and make sure SSH traffic is hitting the intended firewall rule
  • Go to youtube.com and try to watch a video to confirm it is blocked
  • Go to a streaming media site (e.g. vimeo.com, netflix.com) and verify these sites are blocked
Configuration Hints — YouTube Channel ID

The YouTube Channel ID is a unique 24-character string starting with "UC". Retrieve it using one of these methods:

Method 1 — Page Source:

  1. Open https://www.youtube.com/@ZscalerInc in Chrome
  2. Right-click → View Page Source (or Ctrl+U / Opt+Cmd+U)
  3. Search for "og:url" content= or https://www.youtube.com/channel/
  4. The string starting with "UC" (24 characters) is the Channel ID

Method 2 — Third-party tools:

  • commentpicker.com/youtube-channel-id.php
  • seostudio.tools/youtube-channel-id
  • tunepocket.com/youtube-channel-id-finder/

TASK 5 Configure ZPA Policy ▾

Implement ZPA policies for the initial Safemarch deployment, including App Connector provisioning and application access control.

Configuration Requirements
  1. Provision App Connector: Corp: ZPA Connector
  2. Allow all employees access to the following private applications:
    • Intranet: intranet.patraining.safemarch.com — Use TCP port 80, 443
    • File Share: shares.patraining.safemarch.com — Use TCP port 139, 445
  3. Allow access to a server via SSH only on domain-joined devices — Use TCP port 22
  4. Only users in the Maintenance and Administrator departments should access hvac.patraining.safemarch.com — Use TCP port 80
  5. All users excluding Administrator department are to be blocked from using Safemarch's remote desktop application: rdp.patraining.safemarch.com — Use TCP and UDP port 3389:
    • Blocked users should get a notification
    • Ensure that idle, open connections are closed after 15 minutes
    • RDP connections require re-authentication every 10 minutes
  6. Limit access to devices on 10.0.100.0/24 across any port to the Administrator department
  7. Auto-discover any other applications running on the Safemarch corporate network
Testing Requirements
  • To transfer the provisioning key to the App Connector, use WinSCP on the Corp: Win Svr VM
  • For this lab, it is sufficient to enable one App Connector in the data center
  • Check the status of the App Connector
  • To test access to the file share application, access \\shares.patraining.safemarch.com from the Client PC
  • To test RDP access, open from the Windows taskbar and enter rdp.patraining.safemarch.com
  • Check Diagnostics logs and Live Logs in the ZPA Admin Portal to verify policies
If App Connector Provisioning Fails — Re-provision Steps
  1. Stop the service: sudo systemctl stop zpa-connector
  2. Delete files: sudo find "/opt/zscaler/var" -mindepth 1 -delete
  3. Reboot the VM
  4. Stop service again (auto-starts on boot): sudo systemctl stop zpa-connector
  5. Copy provisioning key: sudo cp provision_key /opt/zscaler/var/
  6. Start service: sudo systemctl start zpa-connector
  7. Check status: sudo systemctl status zpa-connector

TASK 7 Prevent the Leakage of PII and PCI Data ▾

Implement DLP policies to block confidential PCI data from being uploaded to external websites, and export logs confirming the blocks.

Configuration Requirements
  1. Ensure that all file upload traffic is fully inspected so that confidential PCI data is blocked and the activity logged. Attempts to upload files to the dlptoolbox website should fail.
  2. After the file upload is blocked by DLP policy, extract the relevant event logs from Web Insight:
    • Export format: CSV
    • The logs should contain only entries related to the DLP block event
    • Required columns: Event time, User, Policy Action, URL, DLP MD5, DLP Identifier, Blocked Policy Type, DLP Dictionary, DLP Engine
  3. Save the exported file to the desktop on Client PC VM
Testing Requirements
  1. Login with student@zs4811.safemarch.com credentials to Zscaler Client Connector on Corp: Client PC; verify Internet Security status is enabled
  2. Open https://dlptoolbox.com in a browser and confirm access to the website
  3. Attempt to upload "Customer Credit Card.xlsx" and "Confidential SSNs and Credit Cards.xlsx" from the DLP Test Files folder on the desktop, in the File Audit pane on dlptoolbox.com
  4. Confirm that a block notification is displayed when attempting the upload
  5. Check Web Insights logs in the ZIA Admin Portal to ensure the file upload attempt is logged and blocked
Configuration Hints
  • Be sure to use predefined dictionaries (e.g. Credit Cards) while creating the DLP engine

TASK 8 Apply Zero Trust Automation: Identify Blocked Traffic ▾

Monitor web traffic logs to identify potentially malicious requests, alert administrators, and block the source using the ZIA API.

Configuration Requirements
  1. Fetch and analyze web URLs and identify the allow/deny list using the advanced endpoint of the ZIA Base URL:
    • Endpoint: {{ZIABase}}/security/advanced
    • Use this to deny/backlist sites not allowed for your organisation (up to 25,000 URLs)
    • Sample URLs to check: tinder.com, snapchat.com, gamixlabs.com, 888.com
  2. Block unauthorized URLs using the POST /security/advanced/blacklistUrls endpoint:
    • Endpoint: {{ZIABase}}/security/advanced/blacklistUrls?action=ADD_TO_LIST
    • URLs to block: betano.com, bet365.com
  3. Notify administrators of the incident using POST /dlpNotificationTemplates:
    • Required inputs: Name, Subject, Plain Text Message
  4. Validate the updates
Testing Requirements
  • Create an API client in ZIdentity Dashboard
  • Authenticate a session for the OneAPI client in Postman
  • Call the API endpoint to add the malicious URLs
  • Navigate to the ZIA Admin portal and verify the URLs are added to the blocked list
  • Notify the Admin about the update
Where to Get Help
  • help.zscaler.com — POST /security/advanced/blacklistUrls
  • help.zscaler.com — POST /dlpNotificationTemplates
