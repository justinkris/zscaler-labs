💡 THINK FIRST · ZCC ENROLLMENT

ZCC enrollment ties a device to a specific ZIdentity tenant and user. Until enrolled, no ZIA policy applies and no ZPA private apps are accessible — the device is invisible to the platform.

  • When ZCC logs into ZIdentity using student@FQDN, what two things is ZIdentity determining on the user's behalf?
  • Why does "Internet Security: ON" appear in ZCC but the user never configured a ZIA policy?
  • What would happen if a student enrolled ZCC but their user account had no ZIA service entitlement assigned?
Q1 — MODEL ANSWER
ZIdentity is determining: (1) authentication — verifying the user's identity via SAML/OIDC; and (2) authorization — which services the user is entitled to (ZIA, ZPA, ZDX) and which admin roles they hold. It then issues a session token that ZCC uses to activate the corresponding services.
Q2 — MODEL ANSWER
ZIA has default tenant-level policies that apply to all enrolled users regardless of custom rules. The "Internet Security: ON" state simply means ZCC has successfully connected to the ZIA cloud and is forwarding traffic — it does not indicate any custom policy is active.
Q3 — MODEL ANSWER
ZCC would enroll (authentication would succeed), but the Internet Security service status would remain OFF or show "No license." ZIA traffic forwarding would not be enabled because the entitlement authorization step would have returned no ZIA subscription for that user.
LAB 1

Scenario: Start your Skytap pod, connect to the Corp: Client PC VM, access the ZIdentity landing page, and verify that Zscaler Client Connector is enrolled and active for both Internet Security and Private Access.

Time: ~15 minutes

Task 1.1 — Start the Skytap Lab Environment

1 Start and connect to the lab pod

  1. Log in to the Skytap portal using the credentials from your Lab Access Instructions.
  2. Open your assigned lab environment and click Run to start all VMs.
  3. Wait until all VMs show status Running before proceeding.
  4. Click the Corp: Client PC VM thumbnail to open its console.
  5. Log in as student / Admin-123!.

Task 1.2 — Access ZIdentity Landing Page

2 Log in to ZIdentity

  1. Open a browser on the Corp: Client PC and navigate to the ZIdentity Landing Page URL from your Lab Access Instructions (e.g. https://zsXXXX.safemarch.zslogin.net).
  2. Enter student@<Student FQDN> and click Next.
  3. Enter your <Session Password> and click Sign In.
  4. Verify the ZIdentity tile dashboard loads, showing tiles for: Zscaler Internet Access, Zscaler Private Access, Zscaler Digital Experience, Zscaler Client Connector, ZIdentity Administration, and others.

Note: The ZIdentity landing page URL is unique to your tenant. It follows the pattern https://zsXXXX.safemarch.zslogin.net where XXXX is your student number.

Task 1.3 — Verify Zscaler Client Connector

3 Check ZCC enrollment status

  1. ZCC is pre-installed on the Corp: Client PC. Click the ZCC icon in the system tray to open it.
  2. If ZCC is not yet enrolled, click Login, enter student@<Student FQDN>, click Next, enter <Session Password>, and click Sign In.
  3. Wait for ZCC to complete enrollment (may take 1–2 minutes).
  4. Verify in the ZCC window:
    • Internet Security Service Status: ON
    • Private Access Service Status: ON
    • Network Type: Off-Trusted Network
    • Username: student@<Student FQDN>

Result: Both Internet Security and Private Access show ON. ZCC Tunnel Version shows v2.0-DTLS. This confirms the student user is enrolled and traffic is being forwarded through the Zscaler cloud.

🧠
MENTAL NOTE
ZCC enrollment flow: (1) User enters student@FQDN → ZIdentity authenticates. (2) ZIdentity returns service entitlements → ZCC activates Internet Security and Private Access. (3) ZCC connects to ZIA cloud (Off-Trusted Network = tunnel active). Verify: Internet Security = ON, Private Access = ON, Network Type = Off-Trusted Network.

LAB PROGRESS
QUICK REFERENCE ▲
ZIdentity
Zscaler's cloud IdP — authenticates users and authorizes service + admin entitlements for ZIA, ZPA, ZDX, ZCC, and ZWA.
Student FQDN
Your unique tenant domain, e.g. zs9901.safemarch.com. All login IDs are username@FQDN.
ZCC
Zscaler Client Connector — endpoint agent that tunnels traffic through ZTE. Required for ZIA, ZPA private access, and ZDX monitoring.
Z-Tunnel 2.0
Network-layer tunnel intercepting all IP protocols. Required for Cloud Firewall Filtering of non-web traffic.
Browser Access
ZPA feature enabling access to internal web apps without ZCC. Traffic goes through Zscaler cloud via CNAME DNS redirect.
App Connector
ZPA software in the data centre making outbound TLS tunnels to ZPA cloud. No inbound firewall rules needed.
App Segment
ZPA object defining a private resource (domain/IP + ports). Paired with Access Policy to enable ZPA access.
SSL Inspection
Trusted MitM: ZIA decrypts, inspects, re-encrypts using Zscaler CA. Certificate Issuer changes to "Zscaler Intermediate Root CA".
DLP Engine
Defines what sensitive content looks like (dictionaries + threshold). Paired with a DLP Rule that defines the action (Block/Allow).
ZWA
Zscaler Workflow Automation — DLP incident management portal with escalation, notification, and automated workflow templates.
Risk360 / MITRE ATT&CK
Risk360 maps TTPs from MITRE ATT&CK and NIST CSF to your Zscaler configurations, showing coverage and misconfiguration gaps.
ZDX Score
0–100 composite metric: 0=worst, 100=best. Probes measure DNS time, TCP connect, server response, page fetch time continuously.
OneAPI
Unified gateway for all Zscaler REST APIs (ZIA, ZPA, ZDX, ZCC, ZIdentity). OAuth 2.0 via ZIdentity API Clients.
SDC
Solutions Demo Center — shared Zscaler tenants with pre-configured data. Used for Labs 9 (Deception), 11 (ZWA), 12 (Risk360).
Log Streaming
Pushes ZIA/ZPA log records to SIEM in real time via syslog or HTTPS. Requires NSS VM for ZIA, native for ZPA.
