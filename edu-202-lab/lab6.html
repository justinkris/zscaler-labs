💡 THINK FIRST · SSL/TLS INSPECTION

SSL Inspection is a trusted MitM — ZIA presents its own certificate to the browser, decrypts the HTTPS session, inspects the content, then re-encrypts and forwards it. The Zscaler Root CA must be trusted by the endpoint for this to work without browser warnings.

  • What is "certificate pinning" and why do you need a "Do Not Inspect" bypass rule for apps that use it?
  • If a user's browser doesn't trust the Zscaler Root CA, what exactly does the user experience when SSL Inspection is active?
  • You enable SSL Inspection but a DLP policy still misses credit card numbers in an HTTPS upload. What is the most likely configuration gap?
Q1 — MODEL ANSWER
Certificate pinning is when an application hardcodes the expected certificate fingerprint or public key for a specific server. When ZIA intercepts and re-signs the traffic with the Zscaler CA, the fingerprint no longer matches, and the app throws a TLS error or refuses to connect. Apps like Dropbox, Chrome updates, and some banking apps use pinning — they need a "Do Not Inspect" rule with higher priority than the Inspect rule.
Q2 — MODEL ANSWER
The browser receives a certificate signed by the Zscaler CA, which is not in its trusted root store. It shows a "Your connection is not private" (NET::ERR_CERT_AUTHORITY_INVALID) error and blocks the page. The user cannot proceed unless they manually accept the certificate — which is a security warning, not normal behavior.
Q3 — MODEL ANSWER
The DLP rule's protocol criteria may not include HTTPS, or SSL Inspection is not enabled for the traffic category that includes the upload destination. DLP can only inspect plaintext — if traffic bypasses SSL Inspection (due to a "Do Not Inspect" rule or category bypass), the DLP engine sees only encrypted bytes and cannot match content patterns.
LAB 6

Scenario: Enable SSL Inspection so ZIA can decrypt and inspect HTTPS traffic for threat and DLP policy enforcement. Create the Inspect rule and verify it is working by checking the certificate issuer in the browser.

Time: ~20 minutes

Task 6.1 — Create an SSL Inspection Rule

1 Add an Enable SSL Inspection rule

  1. In ZIA Admin Portal, navigate to Policy > SSL Inspection.
  2. Click + Add SSL Inspection Rule.
  3. Configure the rule:
    • Name: Enable-SSL-Inspection
    • Action: Inspect
    • Criteria: Leave blank to apply to all HTTPS traffic (catch-all)
  4. Click Save and Activate.

Note: The Zscaler Root CA certificate must be installed in the Corp: Client PC browser's trusted root store. In this lab environment, it has been pre-installed via GPO.

Task 6.2 — Verify SSL Inspection

2 Confirm certificate issuer has changed

  1. On the Corp: Client PC, open a browser and navigate to any HTTPS website (e.g. https://www.example.com).
  2. Click the padlock icon in the browser address bar.
  3. View the certificate details and check the Issuer field.
  4. Verify the Issuer is now Zscaler Intermediate Root CA (not the site's original CA).
  5. Navigate to https://www.example.com before and after the rule activation — compare the certificate issuers.

Result: The certificate issuer changes from the website's CA (e.g. DigiCert) to "Zscaler Intermediate Root CA," confirming ZIA is decrypting and re-encrypting the HTTPS session.

🧠
MENTAL NOTE
SSL Inspection: ZIA decrypts HTTPS by presenting a Zscaler-signed certificate to the browser. Verification: open any HTTPS site, click the padlock, inspect the certificate — Issuer should change from the site's real CA to "Zscaler Intermediate Root CA." Two rule types: "Inspect" (decrypt) and "Do Not Inspect" (bypass). Do Not Inspect rules for cert-pinned apps must be ordered above the Inspect rule.
