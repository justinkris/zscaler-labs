LAB 14

Scenario: Create a ZDX Alert Rule for the Mail_server application targeting Asia geolocations, and configure a Deep Tracing Diagnostic Session for the student user's device to enable on-demand troubleshooting.

Time: ~30 minutes
💡 THINK FIRST · ZDX ALERTING & DIAGNOSTICS

ZDX Alerts notify helpdesk proactively when performance degrades — before users call. Diagnostic sessions collect deep packet-level trace data from the user's device on demand, giving helpdesk a precise root-cause dataset.

  • An alert rule has criteria: Page Fetch Time ≥ 800 ms AND ZDX Score < 20. How does the AND logic affect alert noise compared to using OR?
  • A Diagnostic Session uses Deep Tracing. What specific data does Deep Tracing collect that a regular ZDX probe does not?
  • The "Alert Only if Repeated: 2" throttle setting means the alert fires only after the condition is met 2 times in a row. Why is this throttle important in a production environment?
Q1 — MODEL ANSWER
AND logic requires both conditions to be true simultaneously, which is a much narrower trigger than OR. A high Page Fetch Time alone might be a one-off slow server response; a low ZDX Score alone might reflect a brief DNS hiccup. When both conditions are simultaneously true, it strongly indicates a sustained performance problem. AND logic significantly reduces false-positive alerts compared to OR, which would fire on every individual metric fluctuation.
Q2 — MODEL ANSWER
Deep Tracing captures per-packet network flow data (similar to a Wireshark trace) from the ZCC endpoint — including detailed DNS resolution timing, TCP handshake, TLS negotiation, and application-layer transaction timings. Regular ZDX probes only report aggregated metrics (total DNS time, TCP connect time, page fetch time). Deep Tracing gives the precise millisecond-level breakdown needed to pinpoint whether a problem is in the last mile, the Zscaler node, or the server.
Q3 — MODEL ANSWER
Without throttling, a sustained performance degradation (e.g. a server that is slow for 30 minutes) would send a new alert email every 5 minutes as each probe cycle exceeds the threshold. Helpdesk would be buried in redundant notifications. "Alert Only if Repeated: 2" means the condition must persist for at least 2 consecutive probe cycles before alerting, filtering out one-off transient spikes and reducing alert fatigue.

Task 14.1 — Create an Alert Rule

1 Add the Test_Alert rule in ZDX

  1. In the ZDX Admin Portal, click the Alerts tab in the left menu, then click the Rules tab.
  2. Click + Add Alert Rule.
  3. Step 1 — Configure Rule:
    • Name: Test_Alert
    • Status: Enabled
    • Type: Application
    • Severity: Medium
  4. Click Next.
  5. Step 2 — Filters:
    • Applications: Mail_server
    • Web Probe: Mail_Server_Web
    • Click + Add Filter and add Geolocations: Include Asia (select all countries in the Asia region)
  6. Click Next.
  7. Step 3 — Criteria:
    • Add: Page Fetch Time >= 800 ms
    • Add: ZDX Score < 20
    • Set logic: ALL (AND)
  8. Click Next.
  9. Step 4 — Actions:
    • Alert Only if Repeated: 2 times in a row
    • Number of Active Devices: 1
    • Alert Delivery Method: Email
    • Alert Recipients: enter any email address
  10. Click Next, review the configuration, and click Save.
  11. Go to Activation and click Activate.

Task 14.2 — Configure a Diagnostic Session

2 Enable hostname collection and start Deep Tracing

  1. On the Corp: Client PC, navigate to the ZIdentity landing page → Zscaler Client Connector tile.
  2. In the ZCC portal, navigate to Administration > Client Connector Support.
  3. Click the User Privacy tab.
  4. Enable Collect Machine Hostname Information and click Save.
  5. Return to the ZDX Admin Portal (close and reopen from ZIdentity if needed).
  6. Click Diagnostics in the left menu and click + Start New Diagnostic Session.
  7. General settings:
    • Name: Test Diagnostic Session
    • User: Student Admin User (your student@FQDN user)
    • Device: Select the Corp: Client PC device
    • What to run: Deep Tracing (checked)
    • Run Session For: 5 Minutes
  8. Click Next.
  9. Configure probes:
    • Applications: Mail_server
    • Web Probe: Mail_Server_Web
    • Cloud Path Probe: Mail_Server_cloud
    • Cloud Path Probe Thresholds: Packet Loss = 20%, Latency = 10 ms
  10. Click Save. The diagnostic session appears in the progress table with status Created.

Result: The diagnostic session is created and will start collecting Deep Tracing data from the student user's device. Check the Diagnostics page history after 5 minutes to see the collected session results.

🧠
MENTAL NOTE
Alert rule config: ZDX → Administration → Alerts → Rules → + Add Alert Rule. 5 steps: Configure Rule (name, severity, type) → Filters (application, probe, geolocation) → Criteria (Page Fetch Time ≥ 800ms AND ZDX Score < 20) → Actions (throttle, delivery method email) → Review. Diagnostic session prerequisite: ZCC Client Connector Support → User Privacy → enable "Collect Machine Hostname Information." Then ZDX Diagnostics → Start New Diagnostic Session.
