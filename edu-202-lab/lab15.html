LAB 15

Scenario: Configure a ZIdentity API client to enable programmatic access to Zscaler APIs. Use POSTMAN to authenticate, test ZCC and ZPA API endpoints, and automate the creation of a ZPA App Connector Group.

Time: ~35 minutes
💡 THINK FIRST · ONEAPI & AUTOMATION

Zscaler OneAPI provides programmatic access to all Zscaler services (ZIA, ZPA, ZDX, ZCC, ZIdentity) through a unified OAuth 2.0 gateway. API clients replace per-service API keys with short-lived access tokens scoped to specific resources.

  • Why does OneAPI use OAuth 2.0 Client Credentials instead of a static API key for authentication?
  • The App Connector Group POST requires a signingCertId from the Enrollment Certificate endpoint. Why can't you hardcode this ID?
  • An API client is configured with ZPA Super Admin scope. A developer accidentally deletes an App Connector Group via the API. What ZPA feature would let you recover from this?
Q1 — MODEL ANSWER
Static API keys are long-lived and if leaked, give permanent access. OAuth 2.0 Client Credentials issue short-lived access tokens (30–90 minutes) — a leaked token expires quickly, limiting the exposure window. Additionally, Client Credentials flow enables fine-grained scope control: the same client can be granted only the specific API resources it needs (e.g. ZCC read-only, ZPA full admin), following least-privilege principles.
Q2 — MODEL ANSWER
The Enrollment Certificate ID is unique to each ZPA tenant and changes when certificates are rotated. Hardcoding it would cause the API call to fail after a certificate rotation. The correct pattern is to dynamically query the current cert ID from the Enrollment Certificate Controller at runtime, ensuring the App Connector Group is always created with a valid, current certificate.
Q3 — MODEL ANSWER
ZPA maintains an audit log (Insights > Audit Logs) that records all API and admin portal changes with the actor, timestamp, and changed resource. However, ZPA does not have a built-in "undo" for deleted resources. Recovery would require: (1) finding the Group's configuration in the audit log; (2) recreating it via the admin portal or API; (3) re-assigning any provisioning keys and connectors to the new group. This highlights why API clients should use least-privilege scopes.

Task 15.1 — Add an API Client in ZIdentity

1 Create the TestClient API client

  1. On the Corp: Client PC, navigate to the ZIdentity landing page and click ZIdentity Administration.
  2. In ZIdentity Admin, navigate to Integration > API Clients.
  3. Click + Add API Client.
  4. In the Client tab, configure:
    • Name: TestClient
    • Description: Testing API Client
    • Status: Enable
    • Access Token Lifetime: 90 minutes
    • Validation Type: Secret
  5. Click + Add to generate a client secret. Copy the secret immediately — it will not be displayed again.
  6. Click the Resources tab and assign scopes:
    • Under ZCC INTERNAL SAFEMARCH: select config:1:Super Admin
    • Under ZPA INTERNAL SAFEMARCH: select the first ZPA scope (API Full Access)
  7. Click Save. Note the auto-generated Client ID.

Warning: Copy the Client Secret immediately after generation. ZIdentity will not show it again. Save both the Client ID and Client Secret to a local text file.

Task 15.2 — Test API Endpoints using POSTMAN

2 Import the OneAPI collection and authenticate

  1. On the Corp: Client PC, open the POSTMAN Client from the Desktop.
  2. Sign up for a Postman account with your email if prompted.
  3. Download the Zscaler OneAPI Postman collection from https://automate.zscaler.com/docs/tools/postman/adding-oneapi-postman-collection.
  4. In POSTMAN, go to File > Import and import the downloaded OneAPI collection JSON file.
  5. Verify the Updated OneAPI Collection appears in the Collections panel with sub-folders: ZIA, ZPA, Zscaler Client Connector, ZDX, ZIdentity.
  6. Click the collection root Updated OneAPI Collection and open the Authorization tab.
  7. Configure:
    • Access Token URL: paste your ZIdentity landing page URL (e.g. https://zsXXXX.safemarch.zslogin.net) — the field shows {{"{{"}}zIdentityBaseUrl{{"}}"}}/oauth2/v1/token
    • Client ID: paste the Client ID from Task 15.1
    • Client Secret: paste the Client Secret from Task 15.1
  8. Scroll down and click Get New Access Token. Authentication completes.
  9. In the Manage Access Tokens window, click Use Token.
  10. Verify in ZIdentity Admin → Integration → Tokens that the new token appears.

3 Test ZCC API endpoints

  1. In the Collections panel, expand Zscaler Client Connector.
  2. Click GET Get all enrolled devices.
  3. Click Send and verify the response shows status 200 OK.
  4. Select GET Download device details as a CSV file.
  5. Click Send and verify the response shows 200 OK with CSV headers in the body.

Task 15.3 — Add App Connector Group via OneAPI

4 Get ZPA Customer ID and Enrollment Certificate ID

  1. In ZPA Admin Portal, navigate to Account and note the Customer ID (a long number).
  2. In POSTMAN, navigate to Zscaler Private Access (ZPA) > Enrollment Certificate Controller > Gets all configured enrollment certificate details for the specific customer.
  3. Click the Params tab and enter the Customer ID in the customerId Path Variable.
  4. Click Send. In the response Body, find the first "id" value — note it down (e.g. "8662").

5 POST to create the App Connector Group

  1. In POSTMAN, navigate to ZPA > Connector Group Controller > Adds a new App Connector Group for the specified customer.
  2. Enter the Customer ID in the Path Variables.
  3. Click the Body tab, select raw / JSON, and enter:
    • {{ '{' }}"name": "ZPALAB APPC Group", "description": "App Connector training Lab at San Jose", "upgradeTimeInSecs": "66600", "latitude": "37.3382032", "longitude": "-121.886286", "location": "San Jose, CA, USA", "signingCertId": "YOUR_CERT_ID_HERE"{{ '}' }}
    • Replace YOUR_CERT_ID_HERE with the cert ID obtained in Step 4
  4. Click Send. Verify the response code is 201 Created.
  5. In ZPA Admin Portal, navigate to Configuration & Control > Private Infrastructure > APP Connector Management > APP Connector Group.
  6. Verify ZPALAB APPC Group appears in the list.

Result: The App Connector Group was created programmatically via the OneAPI. The 201 Created response confirms successful resource creation. This same pattern can be scripted to automate ZPA infrastructure provisioning.

🧠
MENTAL NOTE
OneAPI auth flow: ZIdentity API Client (Client ID + Client Secret) → POST to ZIdentity /oauth2/v1/token → receive Bearer access token → use token in POSTMAN Authorization header for all ZIA/ZPA/ZDX/ZCC API calls. POSTMAN OneAPI collection: Updated OneAPI Collection (import from Zscaler Automation Hub). App Connector Group POST: endpoint {{ZPABase}}/mgmtconfig/v1/admin/customers/:customerId/appConnectorGroup — requires signingCertId from GET enrollment cert.
