LAB 7

Scenario: Configure the Zscaler Sandbox to quarantine unknown files on first download and block files with known-malicious verdicts on subsequent attempts. Test with a sandbox-triggering file from a test site.

Time: ~20 minutes
💡 THINK FIRST · CLOUD SANDBOX

The Zscaler Sandbox detonates unknown files in an isolated environment before delivering them to the user. "First-Time Good" would let unknown files through once — "First-Time Quarantine" is stricter: the user waits until analysis completes.

  • What is the trade-off between "First-Time Good" and "First-Time Quarantine" sandbox policies?
  • A known malware sample (MD5 hash already in Zscaler's threat database) is downloaded. Does it need to go through the sandbox? Why or why not?
  • Sandbox inspection only works for files in the supported format list. What happens when a user downloads a file type not supported by the sandbox?
Q1 — MODEL ANSWER
"First-Time Good" allows an unknown file through immediately and sends it to sandbox in parallel — faster user experience but the file reaches the endpoint before analysis. "First-Time Quarantine" blocks delivery until sandbox completes — eliminates zero-day delivery risk but adds latency (often 30–120 seconds). Use Quarantine in high-security environments; use Good for productivity-sensitive users.
Q2 — MODEL ANSWER
No. Known malware is caught by ZIA's inline hash-based threat intelligence before sandbox analysis is even triggered. The sandbox is specifically for files with unknown verdicts. This is why sandbox and threat detection are complementary — hash lookups are instant; sandbox covers what hash lookups miss.
Q3 — MODEL ANSWER
The file passes through uninspected by the sandbox (sandbox cannot analyze it). It is still subject to other ZIA policies — URL filtering, DLP, cloud app control — but the sandbox verdict will not be applied. Supported formats typically include: PE executables, Office documents, PDF, ZIP, and script files.

Task 7.1 — Create a Sandbox Catch-All Rule

1 Configure sandbox policy

  1. In ZIA Admin Portal, navigate to Policy > Advanced Threat Protection > Malware Protection.
  2. Click + Add Rule.
  3. Configure:
    • Name: Sandbox-Catch-All
    • File Types: Select all available file types
    • ZPA App Segments: Include (to cover ZPA-tunneled traffic too)
    • First-Time Action: Quarantine
    • Subsequent Action: Block
  4. Click Save and Activate.

Task 7.2 — Test the Sandbox

2 Download a test file and observe sandbox behavior

  1. On the Corp: Client PC, ensure SSL Inspection is active (from Lab 6).
  2. Open a browser and navigate to a safe sandbox test URL (provided by your instructor).
  3. Attempt to download the test file.
  4. Observe: the download is held (Quarantine) while the sandbox analyzes the file. A ZIA block/hold page appears.
  5. In ZIA Admin Portal, navigate to Analytics > Web Insights > Sandbox Reports to see the sandbox verdict.

Result: The file download is held in quarantine. The ZIA sandbox block page is displayed to the user. The Sandbox Reports page shows the file hash, verdict, and analysis timestamp.

🧠
MENTAL NOTE
Sandbox rule: Policy > Advanced Threat Protection > Malware Protection. Catch-all rule: all file types, Action: Quarantine (first-time), Block (subsequent). The sandbox intercepts files before delivery, detonates them in an isolated VM, and returns a verdict. First-time quarantine = user waits; subsequent block = file with known malicious verdict is blocked immediately.
