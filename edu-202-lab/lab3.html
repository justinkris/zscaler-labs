LAB 3

Scenario: Safemarch has an internal HVAC monitoring application that third-party contractors need to access from their own devices — without installing ZCC. Configure ZPA Browser Access to make hvac.patraining.safemarch.com accessible from any browser on the internet.

Time: ~35 minutes
💡 THINK FIRST · ZPA BROWSER ACCESS

Browser Access lets users reach internal web applications without ZCC. Zscaler acts as a reverse proxy — the CNAME points the app's public hostname to a Zscaler edge node, which forwards authenticated requests to the App Connector.

  • Browser Access requires a certificate for the external hostname. Why can't you use a self-signed certificate here?
  • The HVAC app is accessed at hvac.patraining.safemarch.com. The internal server is at an RFC-1918 IP. How does Zscaler know where to forward the request?
  • A vendor using Browser Access submits a web form that POSTs 5 MB of data. What would break if ZPA's max body size limit is exceeded, and why does this matter for BA?
Q1 — MODEL ANSWER
Browser Access proxies traffic through Zscaler's cloud edge. The browser connects to Zscaler's node and validates its TLS certificate. A self-signed cert would cause a browser TLS error because it isn't trusted by a public CA. The certificate must be signed by a CA the user's browser trusts — either a public CA or an enterprise CA whose root is distributed via GPO.
Q2 — MODEL ANSWER
The App Segment in ZPA maps the external hostname (hvac.patraining.safemarch.com) to the internal domain/IP and port. The App Connector, sitting in the corporate network, resolves and reaches the internal server on behalf of the request.
Q3 — MODEL ANSWER
ZPA Browser Access is a reverse proxy with configurable body size limits. If the POST body exceeds the limit, ZPA returns an error and the upload fails. This matters because BA is designed for browser-based web apps — large file transfers should use ZPA private app segments with ZCC instead.

Task 3.1 — Create and Sign a Certificate

1 Generate a CSR on the Windows Server

  1. On the Corp: Windows Server VM, open Internet Information Services (IIS) Manager.
  2. Click the server node in the left panel, then open Server Certificates.
  3. In the right pane, click Create Certificate Request.
  4. Fill in the distinguished name fields:
    • Common Name: hvac.patraining.safemarch.com
    • Organization: Safemarch
    • Organizational Unit: IT
  5. Keep the default cryptographic settings (2048-bit RSA) and save the CSR to a file.

2 Sign the CSR with the corporate CA

  1. On the Windows Server, open the Certificate Authority console (certsrv).
  2. Right-click the CA name and select All Tasks > Submit new request, then select the CSR file.
  3. In Pending Requests, right-click the request and select All Tasks > Issue.
  4. In Issued Certificates, open the certificate and export it as a .pfx (include the private key, set a password).

Task 3.2 — Upload Certificate to ZPA

3 Add the BA certificate in ZPA Admin Portal

  1. On the Corp: Client PC, access ZPA Admin Portal via the ZIdentity landing page → Zscaler Private Access tile.
  2. Navigate to Configuration & Control > Certificate Management > Browser Access Certificates.
  3. Click + Add Browser Access Certificate.
  4. Upload the .pfx file and enter the certificate password.
  5. Click Save and confirm the certificate appears in the list with the correct CN (hvac.patraining.safemarch.com).

Task 3.3 — Create App Segment with Browser Access

4 Create the HVAC app segment

  1. In ZPA Admin Portal, navigate to Resource Management > Application Segments.
  2. Click + Add Application Segment and fill in:
    • Name: HVAC-Browser-Access
    • Domain/IP: hvac.patraining.safemarch.com
    • TCP Port Range: 443
    • Segment Group: select an existing group or create one
    • Browser Access: Enable and select the certificate uploaded in Task 3.2
  3. Click Save.

Note: Enabling Browser Access on an app segment tells ZPA this app can be reached without ZCC via CNAME-based redirect.

Task 3.4 — Configure Access Policy and CNAME

5 Create an access policy

  1. In ZPA Admin Portal, navigate to Policy > Access Policy.
  2. Click + Add Rule.
  3. Set Rule Name: Allow-HVAC-Browser-Access, Action: Allow.
  4. In Criteria, add the Application: HVAC-Browser-Access.
  5. Click Save and then Save & Activate.

6 Configure the CNAME on the Remote DNS Server

  1. On the Remote: DNS Server VM, open DNS Manager.
  2. Navigate to the safemarch.com forward lookup zone.
  3. Add a new CNAME record:
    • Alias Name: hvac.patraining
    • FQDN Target: the ZPA Browser Access CNAME shown in the App Segment (ends in .zpath.net)
  4. Click OK to save.

Task 3.5 — Test Browser Access

7 Access HVAC app from the Remote Linux Client (no ZCC)

  1. On the Remote: Linux Client PC, open a browser.
  2. Navigate to https://hvac.patraining.safemarch.com.
  3. You will be prompted to authenticate via ZIdentity — enter student@<Student FQDN> and <Session Password>.
  4. Verify the HVAC application loads in the browser.
  5. Check the certificate in the browser — it should be issued to hvac.patraining.safemarch.com by your corporate CA.

Result: The HVAC internal application is accessible from a remote machine with no ZCC installed. Traffic flows: Remote Browser → CNAME → ZPA cloud BA node → App Connector → internal HVAC server.

🧠
MENTAL NOTE
Browser Access pipeline: External DNS CNAME → Zscaler BA edge node → ZPA cloud → App Connector tunnel → internal server. Required steps: (1) Create/sign certificate for the public hostname. (2) Upload cert to ZPA. (3) Create App Segment with BA enabled. (4) Create Access Policy. (5) Add CNAME on public DNS. Test from a device with NO ZCC.
