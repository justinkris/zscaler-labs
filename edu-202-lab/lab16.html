LAB 16 · CAPSTONE

Scenario: Apply what you have learned to configure a complete Safemarch deployment. Each task has specific configuration requirements and testing steps. Your solution will be evaluated against a checklist of requirements.

Time: ~60 minutes
💡 THINK FIRST · CAPSTONE INTEGRATION

The capstone integrates identity, access control, threat protection, data governance, and automation — the five pillars of zero-trust architecture. Each task builds on prior labs; a misconfiguration in one layer often surfaces as a failure in another.

  • Task 16.1 creates a ZPA Helpdesk group with Read-Only ZPA access. Why must you also assign ZIdentity admin entitlements in addition to the ZPA admin role?
    Q1 — MODEL ANSWER
    The ZPA admin role (Read-Only Administrator) defines what the user can do in the ZPA Admin Portal once they are there. But the ZIdentity admin entitlement is what allows the user to log into the ZPA Admin Portal in the first place — it gates the authentication step. Without the ZIdentity admin entitlement, the user cannot reach the portal even if a ZPA role exists for them.
  • Task 16.3 requires a "Do Not Inspect" SSL rule for banking sites while maintaining inspection elsewhere. What is the correct rule ordering and why?
    Q2 — MODEL ANSWER
    The "Do Not Inspect" rule must be placed above the "Inspect" rule in the SSL policy order. SSL rules are evaluated top-down and the first match wins. If the Inspect rule is evaluated first, banking sites will be inspected before the Do Not Inspect rule is reached. Banking sites typically use certificate pinning, so inspecting them would break the connection. "Do Not Inspect" first, "Inspect All" as catch-all below.
  • Task 16.5 uses the OneAPI to create a URL filtering rule programmatically. What is the advantage of this over creating the rule in the ZIA admin portal?
    Q3 — MODEL ANSWER
    Programmatic rule creation via OneAPI enables: (1) version control — the rule definition lives in a git repository. (2) Repeatability — deploy the same rule across multiple tenants with a single script. (3) Integration with change management — changes go through code review before deployment. (4) Auditability — the API call is logged with the API client identity. Creating rules manually in the portal is not repeatable, not version-controlled, and harder to audit.

Task 16.1 — Apply Identity Services Best Practices

1 Configure ZPA Helpdesk and ZDX Rollout groups

In ZIdentity, configure the following groups to support Safemarch organizational requirements:

  1. Create a group ZPA Helpdesk — members need ZPA and ZIdentity admin portal access with a ZPA Read-Only Administrator role (examine but not change ZPA and ZIdentity configurations).
  2. Create a group ZDX Rollout — members need ZIA service entitlement and ZDX service entitlement so they can enroll ZCC and have Digital Experience monitoring active.
  3. Add test users:
    • appuser@<FQDN> to the ZPA Helpdesk group
    • reportuser@<FQDN> to the ZDX Rollout group

Hints: Check that the role ZPA Read Only Administrator exists for Administrative Entitlement Zscaler Private Access. ZDX Rollout group should be member of service entitlements Zscaler Internet Access and Zscaler Digital Experience.

Test: Log in as appuser@FQDN and verify access to ZPA and ZIdentity admin portals with read-only permissions (cannot change settings). Log in as reportuser@FQDN and verify ZCC enrolls with both Internet Security and Digital Experience ON.

Task 16.2 — Apply Access Control Services Requirements

2 Enable ZPA access to the Safemarch intranet

Configure ZPA to allow all Safemarch employees to access the internal intranet application:

  1. Create a ZPA App Segment for intranet.patraining.safemarch.com with TCP ports 80 and 443.
  2. Create an Access Policy rule allowing the Employees group (or all users) to access this app segment.
  3. Activate the ZPA policy.

Test: From Corp: Client PC (with ZCC enrolled), open a browser and navigate to http://intranet.patraining.safemarch.com and https://intranet.patraining.safemarch.com. Verify the intranet page loads. Check ZPA Admin Portal Diagnostics and Live Logs to confirm the policy matched.

Task 16.3 — Apply Cyberthreat Protection Services Requirements

3 Configure SSL inspection with banking site exclusion

Safemarch privacy policy prohibits inspecting personal banking activity. Configure SSL Inspection to bypass banking sites:

  1. In ZIA Admin Portal, go to Policy > SSL Inspection.
  2. Create a custom URL category (e.g. Banking Exclusions) containing common banking sites or choose the built-in Finance category.
  3. Add a Do Not Inspect SSL rule for the Banking category — place it above the Inspect All rule.
  4. Activate the policy.

Test: Navigate to a banking website (e.g. https://www.icicibank.com) and confirm the certificate issuer is the original bank CA (not Zscaler — meaning inspection is bypassed). Navigate to a non-banking HTTPS site and confirm the issuer is still Zscaler Intermediate Root CA.

Task 16.4 — Apply Data Protection Services Requirements

4 Block HIPAA-covered health data uploads

Create a DLP policy to block users from sending private health information (HIPAA):

  1. In ZIA Admin Portal, create a new DLP Rule:
    • Rule Name: Block HIPAA Data Sharing
    • DLP Engine: Use a HIPAA-related engine or the Health Information dictionary
    • Action: Block
  2. Activate the policy.

Test: Navigate to https://dlptoolbox.com/filemaster.aspx and attempt to submit private health information (e.g. Medical Info sample). Verify the submission is blocked and logged. Test with non-sensitive data to confirm the rule does not block unrelated submissions.

Task 16.5 — Apply Zero Trust Automation Requirements

5 Create a URL filtering rule via OneAPI to block gambling sites

Use Zscaler OneAPI to programmatically create a URL filtering rule that blocks gambling websites:

  1. In ZIdentity Admin, create a new API Client with ZIA resource scope (Super Admin).
  2. In POSTMAN, configure the Authorization tab with the new Client ID and Secret, then Get a New Access Token.
  3. In the Collections panel, navigate to ZIA > URL Filtering Policy and find the POST endpoint to create a URL filtering rule.
  4. In the Body tab, build a JSON payload:
    • Rule name: Block-Gambling
    • Action: Block
    • URL Categories: Gambling
    • Protocols: ["ANY_RULE"]
    • Status: Enabled
  5. Click Send and verify 200/201 response.
  6. In ZIA Admin Portal, verify the URL filtering rule was created.
  7. Activate the ZIA policy.

Test: Navigate to a gambling website (e.g. https://www.888.com) and confirm the navigation is blocked by the URL filtering policy.

Hint: When building the JSON body for the URL filtering rule, include "protocols": ["ANY_RULE"] to ensure the rule applies to all protocols, not just HTTP/HTTPS.

Result: Congratulations — you have completed all 5 Capstone tasks, demonstrating mastery of ZIdentity, ZPA, ZIA SSL Inspection, Unified DLP, and Zero Trust Automation via OneAPI.

ZIdentity
Zscaler's cloud IdP — authenticates users and authorizes service + admin entitlements for ZIA, ZPA, ZDX, ZCC, and ZWA.

Your unique tenant domain, e.g. zs9901.safemarch.com. All login IDs are username@FQDN.
ZCC
Zscaler Client Connector — endpoint agent that tunnels traffic through ZTE. Required for ZIA, ZPA private access, and ZDX monitoring.

Network-layer tunnel intercepting all IP protocols. Required for Cloud Firewall Filtering of non-web traffic.
Browser Access
ZPA feature enabling access to internal web apps without ZCC. Traffic goes through Zscaler cloud via CNAME DNS redirect.
App Connector
ZPA software in the data centre making outbound TLS tunnels to ZPA cloud. No inbound firewall rules needed.
App Segment
ZPA object defining a private resource (domain/IP + ports). Paired with Access Policy to enable ZPA access.
SSL Inspection
Trusted MitM: ZIA decrypts, inspects, re-encrypts using Zscaler CA. Certificate Issuer changes to "Zscaler Intermediate Root CA".
DLP Engine
Defines what sensitive content looks like (dictionaries + threshold). Paired with a DLP Rule that defines the action (Block/Allow).
ZWA
Zscaler Workflow Automation — DLP incident management portal with escalation, notification, and automated workflow templates.
Risk360 / MITRE ATT&CK
Risk360 maps TTPs from MITRE ATT&CK and NIST CSF to your Zscaler configurations, showing coverage and misconfiguration gaps.
ZDX Score
0–100 composite metric: 0=worst, 100=best. Probes measure DNS time, TCP connect, server response, page fetch time continuously.
OneAPI
Unified gateway for all Zscaler REST APIs (ZIA, ZPA, ZDX, ZCC, ZIdentity). OAuth 2.0 via ZIdentity API Clients.
SDC
Solutions Demo Center — shared Zscaler tenants with pre-configured data. Used for Labs 9 (Deception), 11 (ZWA), 12 (Risk360).
Log Streaming
Pushes ZIA/ZPA log records to SIEM in real time via syslog or HTTPS. Requires NSS VM for ZIA, native for ZPA.
