💭 THINK FIRST · DLP ARCHITECTURE & ENFORCEMENT

DLP has three architectural layers that build on each other: dictionaries contain the detection patterns; engines combine dictionaries with Boolean logic; rules apply engines to traffic. Troubleshooting a DLP gap requires knowing which layer broke.

  • A DLP rule uses the PCI-Medium DLP engine. The engine uses the Credit Cards dictionary with a threshold of > 5 matches. A file with 3 credit card numbers passes uninspected. Is this a misconfigured rule, engine, or dictionary — and how would you diagnose it?
  • Zscaler DLP supports EDM (Exact Data Match), IDM (Indexed Document Match), and regex pattern dictionaries. For protecting a list of 10,000 specific employee SSNs, which dictionary type would you use and why?
  • A DLP rule is set to Block, but the Zscaler Incident Receiver is set to None. What happens to block events — are they visible anywhere, and what is the admin missing?
Q1 — MODEL ANSWER
This is likely working as designed. The engine threshold (> 5) means the file must contain more than 5 credit card numbers before the engine triggers. A file with 3 credit card numbers won’t match the PCI-Medium engine — it would match PCI-Low (with a lower threshold) if that rule/engine existed. To diagnose: check the engine expression to confirm the threshold; check if there’s a separate PCI-Low rule for 1–5 matches; then check whether a file with >5 numbers is correctly blocked.
Q2 — MODEL ANSWER
EDM (Exact Data Match) is designed for this use case. You upload a structured data file (CSV/TSV) containing the 10,000 SSNs as an index template. ZIA checks each piece of content against the exact values in the index — no false positives from regex patterns that might match non-employee numbers. EDM is much more precise than regex for protecting specific known PII datasets, at the cost of needing to keep the index template updated.
Q3 — MODEL ANSWER
The block is still enforced — the user sees a block page and the upload fails. Block events are visible in Web Insights logs (Policy Action = Block, Blocked Policy Type = Data Loss Prevention). However, without an Incident Receiver configured, the DLP admin receives no notification and no incident evidence is captured (file content, matched strings). The admin is missing the forensic and compliance data needed to respond to violations — a critical gap for audit and incident response.

Lab 6: Troubleshooting Data Protection Services

LAB 6

Scenario: A financial services company needs to protect sensitive data such as credit card numbers and bank account details. You will review DLP policy configuration, test enforcement via file uploads, examine block logs, and explore predefined dictionaries and DLP engines.

Time: ~35 minutes
Test site: https://dlp-test.com/html5_priv/

Task 6.1: Review DLP Configurations

1 Review the Block PCI Medium DLP Rule

  1. In the ZIA Admin portal, navigate to Policy > Data Loss Prevention.
  2. Search for the policy named Block PCI Medium and click the View icon (👁) to view its settings.
  3. Familiarise yourself with the Criteria options:
    • Content Matching: Select DLP Engines — set to PCI - Medium
    • URL Categories, Cloud Applications, Private Applications, File Type, File Size, Users, Groups, Departments are all available criteria
  4. Scroll to the bottom of the rule and review:
    • DLP Incident Receiver: Zscaler Incident Receiver → ZWA-IR
    • Action: Data Traffic = Block
    • Notification: Client Connector Notification = Enabled

Note: Zscaler DLP can send evidence and trigger data to DLP administrators via various means: Email, on-prem Incident Receiver, or cloud-hosted Incident Receiver (ZWA-IR).

2 Review AI/ML Data Discovery Reports

  1. Go to Analytics > Data Discovery Report.
  2. Review the various widgets and graphs. For example, check the Top 10 Users sending sensitive content to various SaaS Applications, and the Top 10 Applications receiving sensitive data (e.g. Dropbox, OneDrive).
  3. Click Analyze More to get deeper insights. In the Analyze More view, click on any content type category (e.g. Tax, Real Estate, Medical Information) to see which Applications are used to upload that content type and which users are responsible.
  4. Click on an Application to see detailed user information. Right-click on a user and select Show in Logs to jump to the Web Insights logs.
  5. Customise the logs table by clicking the ⋮ icon and adding/removing columns.
  6. To review DLP policy violation incidents, go to Analytics > Interactive Reports. Scroll down and click the Data Loss > Top Violations report to see which dictionaries and engines are triggering most frequently.

Note: To view more meaningful dashboard data, select a longer Timeframe from the dropdown (e.g. Last 30 Days).

Task 6.2: Verify DLP Rule Enforcement

3 Test PCI File Upload Block

Task 6.4: View Block Action Web Log Record

6 Examine the Blocked Upload Log Entry

  1. In the ZIA Admin portal, go to Analytics > Web Insights > Logs.
  2. Set the filters to find the record for the blocked Python file upload:
    • Timeframe: Last 15 Minutes
    • Add Filter: User = ptcb32-IT@thezerotrustexchange.com
  3. Click Apply Filters.
  4. Add additional filters to narrow the results:

  5. Click Apply Filters and review the log entry. Verify:

Task 6.5: View Predefined Dictionaries

7 Explore the Credit Cards Predefined Dictionary

  1. In the ZIA Admin portal, go to Administration > Data Loss Prevention and select DLP Dictionaries & Engines.
  2. On the DLP Dictionaries tab, search for Credit Cards and click the View icon.
  3. Review the configurable attributes:
    • Dictionary Type: Predefined
    • Confidence Score Threshold: High
    • High Confidence Phrases: American Express, Amex, Master Card, Visa, Credit Card, etc.
    • BIN Number: Include

Note: You can set different confidence thresholds (Low, Medium, High) for the same identifier. With High confidence thresholds, proximity keywords and BIN numbers reduce false positives. Up to 512 BIN numbers may be specified.

Task 6.6: View Custom Dictionaries — Regex and Phrases

8 Explore the SSN Regex Dictionary

  1. On the DLP Dictionaries tab, click the Type column header to sort dictionaries by type — this brings custom dictionary types to the top:
    • Exact Data Match (EDM)
    • Indexed Document Match (IDM)
    • Microsoft Information Protection (MIP)
    • Patterns & Phrases
  2. Click the View icon for the Social Security Numbers (US) Regex dictionary (Type: Patterns & Phrases).
  3. Review the DLP Dictionary configuration — note the Pattern field:
    • Pattern: \b[0-9]{3}([ -][0-9]{2}[0-9]{4}|[0-9]{2}-[0-9]{4}|[0-9]{6})\b
    • Action: Count Unique

Note: Administrators can create custom dictionaries based on regex, custom phrases, keywords, and proximity keywords. Advanced classifiers include Exact Data Match (EDM), Indexed Document Match (IDM for fingerprinting), and MIP labels. See the Zscaler documentation for Adding Custom Dictionaries.

Task 6.7: View Predefined and Custom DLP Engines

9 Explore the HIPAA DLP Engine

  1. On the DLP Dictionaries & Engines page, click the DLP Engines tab.
  2. Review the list of available engines including: CORP-PRIVATE, ChatGPT CatchAll, Credit Card, EDM Engine, GLBA, HIPAA, IDM Engine, and others.
  3. Click the View icon for the HIPAA engine to open it for editing.
  4. Examine the Engine Builder expression:
    • ((Medical Information > 0) AND (Social Security Numbers (US) > 5))
    • This means: trigger if the document contains any Medical Information matches AND more than 5 Social Security Number matches
  5. Observe that this DLP engine combines two dictionaries with the AND operator — this reduces false positives by requiring both conditions to be true simultaneously.

Note: When configuring a predefined or custom DLP engine, you can combine DLP dictionaries with operators to create logical expressions. Operators include All (AND), Any (OR), Exclude (AND NOT), and Sum. The Sum operator allows specifying a total number of matches across a group of dictionaries.

🧠
MENTAL NOTE
DLP hierarchy: Rule (applies to traffic, specifies action + incident notification) → Engine (combines dictionaries with Boolean operators, e.g. HIPAA = Medical Info AND SSN > 5) → Dictionary (the actual detection patterns: predefined Credit Cards, EDM for exact data, IDM for document fingerprinting, regex Patterns & Phrases). PCI-Medium uses the Credit Card engine with threshold > 5 matches. Blocked events appear in Web Insights logs even without an Incident Receiver — but the admin won’t receive notifications or evidence without one.

LAB 6 PROGRESS
QUICK REFERENCE ▲
DLP Architecture
Rule → Engine → Dictionary. Rules apply engines to traffic. Engines combine dictionaries with Boolean logic. Dictionaries contain detection patterns.
PCI-Medium Engine
Credit Card dictionary with threshold > 5 matches. Files with ≤5 credit card numbers won’t trigger this engine — expected behaviour.
EDM vs IDM vs Regex
EDM = exact data rows (structured CSV). IDM = document fingerprinting (unstructured). Regex = pattern matching. Use EDM for specific lists (SSN of employees); IDM for protecting specific documents; Regex for generic patterns.
File Type Control
Policy → File Type Control. Blocks upload or download of specific file extensions (e.g. Python .py files). Separate from DLP — no content inspection, just file extension check.
Incident Receiver
Required for forensic evidence delivery when DLP rules trigger. Without one, blocks still work but admins get no notification or file evidence. Types: Email, on-prem, or cloud-hosted (ZWA-IR).
