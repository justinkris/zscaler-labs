LAB 8 · LOGGING

Troubleshooting NSS Logging

The Nanolog Streaming Service is the bridge between Zscaler’s cloud-native logging and your SIEM. Knowing how to verify NSS server health, inspect feed configuration, and test connectivity is the last skill in a complete troubleshooting toolkit.

⚙ Lab Configuration
ZIA Admin: ptcb32-ADMIN@thezerotrustexchange.com
💭 THINK FIRST · NSS ARCHITECTURE & FEED TROUBLESHOOTING

NSS is how Zscaler log data leaves the cloud and enters your SIEM. Two delivery modes exist: an on-prem NSS virtual appliance and a Cloud NSS Feed. Both are configured in ZIA, but understanding where the data flows — and why a feed might be Disabled — determines where you troubleshoot.

  • NSS (Nanolog Streaming Service) uses an on-premises virtual appliance, while Cloud NSS Feeds stream logs directly from the Zscaler cloud to a SIEM. What is the key operational difference in terms of infrastructure ownership, and what failure modes exist in each model?
  • An NSS Feed shows Status: Disabled in the ZIA Admin Portal. What are the two most likely reasons for this, and what is the first thing you should check to determine which one applies?
  • A Cloud NSS Feed connectivity test returns OK (200), but logs are not appearing in the SIEM. What does the 200 response actually verify, and what could still prevent logs from flowing downstream?
Q1 — MODEL ANSWER
On-prem NSS requires a virtual machine (NSS Server) deployed on the customer’s network. This appliance pulls log data from the Zscaler Nanolog cluster and forwards it to the SIEM. Failure modes include: VM health, network connectivity between NSS appliance and SIEM, TCP port accessibility, and VM resource exhaustion. Cloud NSS Feeds require no on-prem infrastructure — Zscaler directly pushes log streams to a SIEM listener (IP/port). Failure modes shift to: SIEM listener availability, firewall rules allowing inbound connections from Zscaler cloud IPs, and feed configuration errors (wrong port or output format).
Q2 — MODEL ANSWER
Status: Disabled most often means either (a) the feed was explicitly disabled in the ZIA Admin Portal (intentional — no logs will stream) or (b) the NSS server is unreachable and ZIA automatically disabled the feed to prevent log accumulation. The first check is the NSS Servers tab — confirm the NSS appliance shows a healthy status. If the server is healthy and the feed is still Disabled, an administrator deliberately turned it off. Click the feed’s edit icon to re-enable it.
Q3 — MODEL ANSWER
The connectivity test (cloud icon → Test Connectivity → OK 200) only verifies that the ZIA cloud can open a TCP connection to the SIEM listener on the configured IP and port. It does NOT verify: (a) the SIEM is correctly parsing the feed output format (CSV, JSON, LEEF, CEF), (b) the SIEM has the correct log source or connector configured, (c) the feed is in an Enabled state, or (d) the feed output type matches what the SIEM expects. A 200 means the network path is open — not that logs are flowing end-to-end.

Lab 8: Troubleshooting the Nanolog Streaming Service (NSS)

LAB 8

Scenario: Your organisation uses NSS to stream ZIA log data to a SIEM. A report has come in that no new ZIA logs are appearing in the SIEM dashboard. You will navigate the ZIA Admin Portal to inspect the NSS Server configuration, review NSS Feed settings, check the Cloud NSS Feed, and test connectivity to confirm the pipeline is configured correctly.

Time: ~15 minutes
ZIA Portal: https://admin.zscalerthree.net

Task 8.1: Check NSS Servers and NSS Feeds Configuration

1 Navigate to the Nanolog Streaming Service

  1. In ZIA Admin Portal (https://admin.zscalerthree.net), sign in as ptcb32-ADMIN@thezerotrustexchange.com with the Session Password (<Session Password>) if not already signed in.
  2. In the left navigation, go to Administration → Nanolog Streaming Service. The page opens to the NSS Servers tab by default.

2 Review the NSS Server Configuration

  1. On the NSS Servers tab, locate the entry named AA-Web.
  2. Click the View icon (eye icon) to the right of AA-Web to open the server details.
  3. Review the configuration details:
    • NSS Type: NSS for Web
    • Status: Enabled
    • State: Healthy (green indicator)
    • NSS Server IP / FQDN — the IP or hostname of the on-premises NSS virtual appliance
    • TCP Port — the port NSS uses to receive log data from the Zscaler cloud
  4. Click Cancel to close the view without making changes.

Note: If the NSS Server State shows Unavailable or Disconnected, the virtual appliance on the customer network may be powered off, unreachable, or have a network/firewall issue preventing it from reaching Zscaler. This is a common cause of missing SIEM logs — check the VM health before investigating feed configuration.

3 Review the NSS Feed Configuration

  1. Click the NSS Feeds tab.
  2. Locate the feed named AA-casb and review its details shown in the table:
    • NSS Type: NSS for Web
    • NSS Server: AA-Web
    • Status: Disabled
    • SIEM Destination Type: IP Address
    • SIEM IP Address: 10.0.0.3
    • SIEM TCP Port: 4514
    • Log Type: SaaS Security Activity
    • Feed Output Type: CSV
  3. Note that the Status for AA-casb is Disabled. This means the feed is not currently streaming logs to the SIEM — this is the likely cause of missing logs in the SIEM dashboard.

Important: Do not enable or modify the AA-casb feed during this lab. The objective is to identify the disabled status as the root cause — not to change the configuration in the shared lab environment.

4 Review Cloud NSS Feeds and Test Connectivity

  1. Click the Cloud NSS Feeds tab.
  2. Review the feeds listed. Cloud NSS Feeds deliver logs directly from the Zscaler cloud to a SIEM listener — no on-prem NSS appliance is required.
  3. For each feed, you can see its NSS Type, Status, SIEM IP, and TCP Port in the table columns.
  4. To test connectivity for a Cloud NSS Feed:
    • Click the cloud icon (Test Connectivity icon) to the right of a feed entry
    • A dialog will appear showing the test in progress
    • Verify that the result shows OK (200) — this confirms Zscaler can reach the SIEM listener on the configured IP and port
  5. Click Close to dismiss the connectivity test dialog.

Result: A 200 OK response confirms the network path from the Zscaler cloud to the SIEM endpoint is open. If the test fails (e.g. Connection Refused or Timeout), the issue is the SIEM listener (not running, wrong port) or a firewall blocking the inbound connection from Zscaler cloud egress IPs.

MENTAL NOTE
NSS troubleshooting path: (1) Administration → Nanolog Streaming Service → NSS Servers tab — verify server State is Healthy (not Disconnected); (2) NSS Feeds tab — check Status is Enabled and note SIEM IP/Port/Log Type/Output Format; (3) Cloud NSS Feeds tab — Test Connectivity (cloud icon) → OK 200 confirms network path open. Common root causes: Feed disabled intentionally; NSS appliance VM offline; SIEM listener not accepting connections; firewall blocking Zscaler cloud IPs on the SIEM port.

QUICK REFERENCE ▲
NSS (Nanolog Streaming Service)
On-premises virtual appliance that receives log streams from the Zscaler Nanolog cluster and forwards them to a SIEM over TCP. Configured in ZIA under Administration → Nanolog Streaming Service.
NSS Server
The virtual appliance entry in ZIA. Healthy = VM is connected and receiving data. Disconnected/Unavailable = VM is unreachable — check VM power state and network path back to Zscaler.
NSS Feed
Defines which log type (Web, Firewall, DNS, SaaS Security) is sent to which SIEM destination (IP + TCP Port) via which NSS Server. Feed must be Enabled for logs to flow. Output formats: CSV, JSON, LEEF, CEF.
Cloud NSS Feed
Logs stream directly from Zscaler cloud to SIEM — no on-prem NSS appliance required. SIEM must have a TCP listener running on the configured port and accept inbound connections from Zscaler cloud egress IPs.
Test Connectivity (OK 200)
Verifies Zscaler can open a TCP connection to the SIEM at the configured IP and port. A 200 means the network path is open. Does not verify the SIEM is parsing the log format correctly or that the feed is Enabled.
NSS Feed: Status Disabled
Feed is not streaming. Either explicitly disabled by an admin, or automatically disabled by ZIA after repeated connectivity failures to the SIEM destination. Re-enabling is done from the NSS Feeds tab.
