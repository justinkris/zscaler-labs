💭 THINK FIRST · AUTHENTICATION TROUBLESHOOTING

Authentication failures are the most common Tier-1 ticket — and the most misdiagnosed. Before testing, think about the layers: network reachability, ZCC enrollment, and SAML assertion correctness are three completely different failure modes.

  • What is the difference between a user failing to reach login.zscalerthree.net and a user failing SAML authentication at the IdP? How would the symptom differ from the user’s perspective?
  • The --userDomain CLI flag routes ZCC directly to the org’s SAML SSO page. What problem does this solve compared to the default enrollment flow?
  • If a SAML response shows NotOnOrAfter in the past, what does that tell you about the root cause — and which team is responsible for fixing it?
Q1 — MODEL ANSWER
Network reachability failure means ZCC cannot reach the Zscaler cloud at all — the user gets no internet access and ZCC shows "Service Status: OFF". SAML failure means ZCC can reach Zscaler, but authentication with the IdP (Azure AD) fails — ZCC shows as connected but the user may be stuck in a login loop or get a "Can't access your account" error. The first requires network diagnosis; the second requires identity/SAML diagnosis.
Q2 — MODEL ANSWER
By default ZCC shows a service enrollment screen where the user must choose their organisation. With --userDomain, ZCC skips that screen and redirects the user directly to the SAML IdP page for the specified domain. This prevents users from accidentally authenticating to the wrong tenant and simplifies large deployments via MDM (SCCM/Intune) where no user interaction should be needed.
Q3 — MODEL ANSWER
A NotOnOrAfter timestamp in the past means the SAML assertion has already expired by the time Zscaler received it — this is a clock synchronisation (NTP) issue between the IdP server and the Zscaler service. The IdP admin team is responsible. Confirm by comparing the IdP server time to UTC and check if the gap exceeds the SAML validity window (typically 5 minutes).

Lab 2: Troubleshooting Authentication

LAB 2

Scenario: A new user device needs to be enrolled in Zscaler services. You will verify basic connectivity, manually install ZCC with CLI options, confirm the device enrolls in all services, then trace the ZIA and ZPA SAML authentication flows.

Time: ~45 minutes
Portal: ZIA Admin — https://admin.zscalerthree.net

Task 2.1: Check Client Connectivity to the Zero Trust Exchange

1 Verify Corp: Client PC Internet Access

  1. On the Corp: Client PC, open a Command Prompt and run ipconfig /all.
    • Note the Default Gateway IP and DNS Server IPs for the Public network adapter.
    • Ping both the Default Gateway and DNS server to confirm LAN connectivity.
  2. Check DNS resolution, routing, and ICMP response by pinging key Zscaler cloud hosts:
    • ping pac.zscalerthree.net — PAC file download host
    • ping mobile.zscalerthree.net — mobile configuration host
    • ping login.zscalerthree.net — ZIA authentication URL
  3. Check Zero Trust Exchange Private Access reachability:
    • ping samlsp.private.zscaler.com
    • Note the actual hostname that resolves and gets pinged (e.g. chi1.co2br.gslb.prod.zpath.net) — this is your Private Access broker.

Expected result: All pings should succeed. Successful resolution of samlsp.private.zscaler.com to a geo-distributed broker confirms DNS and routing to the ZPA cloud are working.

Task 2.3: Install Client Connector With CLI Options

3 Install ZCC From the Command Line

  1. On the Corp: Client PC, open a Windows Command Prompt.
  2. Change to the Downloads directory: cd downloads
  3. Run the installer with the following command line switches (replace <file-version> with the actual filename):
    • Zscaler-windows-<file-version>-installer-x64.exe --cloudName zscalerthree --hideAppUIOnLaunch 1 --userDomain thezerotrustexchange.com
  4. When prompted for administrator credentials, enter:
    • Username: administrator
    • Password: Admin-123!
  5. Follow the installation prompts and click Finish.

CLI options explained:
--cloudName zscalerthree — locks the installation to the zscalerthree.net cloud so users are not prompted to select a cloud.
--hideAppUIOnLaunch 1 — prevents the ZCC login page appearing on PC reboot; ZCC stays in the System Tray.
--userDomain thezerotrustexchange.com — bypasses the service enrollment screen and routes the user directly to the Azure AD SAML SSO page.

Note: In production, ZCC installation is automated via MDM (SCCM/Intune/Airwatch). Manual CLI install is used for testing and troubleshooting individual devices.

Task 2.4: Troubleshoot ZIA Authentication

4 Launch ZCC and Verify Enrollment

  1. In the Windows Task Bar (bottom right), click Show hidden icons, then click the Zscaler Client Connector icon, then click Open Zscaler.
  2. Because --userDomain was used, ZCC bypasses the enrollment screen and takes you directly to the Azure AD sign-in page. Enter the username of the IT department user: ptcb32-IT@thezerotrustexchange.com
  3. Click Next, enter the Session Password (<Session Password>), and click Sign in.
  4. At the Stay signed in? prompt, click No.
  5. ZCC will minimise to the System Tray. Open it again and verify the device is enrolled in all four services:
    • Private Access — Service Status: ON, Authentication Status: Authenticated
    • Internet Security — Service Status: ON, Total Bytes Sent/Received incrementing
    • Digital Experience — Service Status: ON, Authentication Status: Authenticated
    • Data Protection
  6. To confirm traffic is forwarded through Zscaler, open a browser and go to http://ip.zscaler.com. Verify you are accessing the Internet via a Zscaler data centre.

Note: If the client does not enrol in all four services, check Administration > Zscaler Service Entitlement in the ZCC Portal to ensure ZPA, ZIA, and ZDX are enabled for User Tunnel.

5 Trace the ZIA SAML Authentication Flow

  1. Open a Chrome browser window on the Corp: Client PC.
  2. Right-click anywhere on the page and click Inspect to open the Developer Tools.
  3. Select the Network tab, check Preserve log, and set the filter to show Doc content only.
  4. In the address bar, paste the following ZIA authentication test URL and press Enter:
    • https://login.zscalerthree.net/clstart?version=1&_domain=thezerotrustexchange.com&redrurl=https://mobile.zscalerthree.net
  5. In the Network tab, click the first document request (clstart?version=1) and check the Headers tab — this is Zscaler acting as a SAML Service Provider initiating the flow.
  6. Click the saml2 document in the inspector. This shows the SP-initiated redirect to the IdP (login.microsoftonline.com for Azure AD).
  7. Click the Payload tab for the saml2 document and right-click to Copy value the SAMLRequest field.
  8. Open a new browser tab and go to https://samltool.com. Click Online Tools > Code/Decode > Base64 Decode + Inflate. Paste the SAMLRequest and click Decode and Inflate XML.
  9. Verify in the decoded XML:
    • AssertionConsumerServiceURL points to the Zscaler cloud login endpoint
    • Destination points to the Microsoft Azure AD SSO endpoint
  10. Back in the browser inspector, click the sfc_sso document. Copy the response payload and decode it in samltool.com. In the decoded response, verify:
    • <NameID> matches the expected IT department user email
    • NotOnOrAfter timestamp is in the future (assertion not expired)
  11. In the ZIA Admin portal, go to Administration > Authentication Settings > Identity Providers. Click to view the thezerotrustexchange.com AAD IdP and verify the SAML Portal URL, Entity ID, and SCIM settings. Click Close.
  12. To review SCIM provisioning events, go to Administration > Authentication > Event Logs and filter for Sub-Category: SCIM, Result: Failure.

Note: The test URL initially redirects to the mobile portal. If authentication is successful it redirects back to the Zscaler Admin Portal page.

Task 2.5: Troubleshoot ZPA Authentication

6 Trace the ZPA SAML Authentication Flow

  1. In the browser, paste the following ZPA authentication test URL and press Enter:
    • https://samlsp.private.zscaler.com/auth/v2/login?ssotype=test&domain=thezerotrustexchange.com
  2. The page will display a readable JSON view of the SAML assertion and attribute values. Note:
    • The username in the name field should match your IT department user
    • SAML attributes such as groups, tenantid, and identityprovider
  3. Open Chrome DevTools (Network tab, Doc filter) and re-run the test URL. Click the saml2 document in the inspector, then click the Payload tab and copy the SAMLRequest value.
  4. Decode the request at https://samltool.com. Verify:
    • ProtocolBinding is HTTP-POST
    • The request contains a ds:Signature (the ZPA SAML request is signed)
    • AssertionConsumerServiceURL points to the ZPA service provider endpoint
  5. Click the sso document in the inspector. Copy the response payload and decode. Verify:
    • Destination is the ZPA SSO endpoint (not ZIA)
    • Status Code is urn:oasis:names:tc:SAML:2.0:status:Success
  6. Open the ZPA Admin portal at https://admin.private.zscaler.com. Sign in with your ZIA Admin User ID and Session Password (ZIA and ZPA admin usernames are identical in this lab).
  7. Navigate to Authentication > User Authentication > IdP Configuration. Review the thezerotrustexchange.com AAD IdP settings, noting the Single Sign-On URL and IdP Entity ID.
  8. Click to expand the IdP Configuration and click Import SAML Attributes to review the attributes for the IT department user. Click Save.
  9. [Optional] Explore Authentication > SCIM Management to understand what information is available for diagnosing SCIM provisioning issues.

MENTAL NOTE
Authentication troubleshooting follows a strict order: (1) Network reachability — can the client reach Zscaler endpoints? (2) ZCC enrollment — is the device enrolled in all four services? (3) SAML flow — is the assertion correct and not expired? The ZIA test URL and ZPA test URL are your fastest diagnostic tools. samltool.com decodes Base64+Deflated SAML in seconds. SCIM failures in Event Logs indicate user provisioning issues, not authentication failures.
