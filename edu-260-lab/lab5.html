💭 THINK FIRST · SECURITY SERVICES & DEFENCE IN DEPTH

Six security layers means six potential misconfiguration points. Understanding why each layer exists — and what it can and can’t protect against — is more useful than memorising where each setting lives.

  • The Sandbox has two actions: Quarantine First Time, Block Subsequent Downloads and Allow and Scan First Time, Block Subsequent Downloads. What’s the real-world tradeoff between choosing one versus the other for a catch-all rule?
  • Zscaler’s best practice is to explicitly block QUIC via Firewall Control rather than relying on the default block rule. Why is this necessary for SSL inspection to work, and what protocol does the client fall back to?
  • The Default Firewall Filtering Rule is set to Block/Drop for any traffic. If this rule didn’t exist, what would happen to non-web traffic (e.g. ICMP, SSH) flowing through Z-Tunnel 2.0?
Q1 — MODEL ANSWER
Quarantine holds unknown files before delivery — safer, but users experience delay. Allow and Scan delivers the file immediately and blocks only if it’s found malicious on subsequent downloads — better user experience but creates a brief window of risk on the first encounter. For a catch-all rule (applying to all users), Quarantine is the safer default. Higher-trust user groups (e.g. security researchers) might use Allow and Scan. Crowdstrike-integrated rules may allow-and-scan because endpoint protection provides a second line of defence.
Q2 — MODEL ANSWER
QUIC is a UDP-based protocol used by browsers for HTTP/3. If QUIC is allowed, browsers bypass the Zscaler TCP-based SSL inspection tunnel and send traffic directly over UDP — Zscaler cannot inspect it. Blocking QUIC (with Block/Reset) forces the browser’s QUIC failsafe: the browser retries over HTTP/2 via TCP, which Zscaler can inspect normally. The explicit Block QUIC rule is needed because without it, if the Default rule was Allow instead of Block, QUIC traffic would pass uninspected.
Q3 — MODEL ANSWER
With Z-Tunnel 2.0, all traffic on all ports and protocols is sent to the Zscaler Firewall — not just web traffic. Without the Default Block/Drop rule, non-web protocols (ICMP, SSH, SMTP, etc.) would pass through uninspected, completely bypassing the security stack. The catch-all block ensures that only explicitly allowed traffic (via rules above the default) can exit the organisation. This is Zero Trust applied at the network layer: deny everything unless explicitly permitted.

  1. In the ZIA Admin portal, go to Policy > Sandbox.
  2. Click the View icon (👁) for the Catch_All rule to view its settings.

Note: The Catch_All rule looks at many file types, applies to all users, and will Quarantine a previously unknown file before allowing it to be downloaded. This is the safest approach for broad user populations.

Note: The Sandbox Detail Report provides forensic intelligence: which registry keys were changed, which network connections were initiated, and which files were read. Click the Expand icon on any widget for additional details.

Task 5.4: Inspect Non-web Traffic Blocking by Default Firewall Block

5 Verify Default Firewall Block Rule

  1. In the ZIA Admin portal, go to Policy > Firewall Control.
  2. On the Firewall Filtering Policy tab, scroll to the bottom and verify:
    • The last rule (highlighted in red) is: Default Firewall Filtering Rule — Criteria: Any — Action: Block/Drop
  3. Note the rules above the default that explicitly allow specific traffic:
    • Rule 1: Block Malicious IPs and domains (Disabled in this lab)
    • Rule 2: Zscaler Proxy Traffic — Allow
    • Rule 3: Office 365 One Click Rule — Allow
    • Rule 4: UCaaS One Click Rule (Zoom, Webex) — Allow
    • Rule 5: Block QUIC — Block/Reset
    • Rule 6: Recommended Firewall Rule (DNS, HTTP, HTTPS) — Allow
    • Rule 7: Block Restricted Countries — Block/Drop
  4. To verify the default rule is blocking ICMP traffic, check the Firewall Insights logs:
    • Go to Analytics > Firewall Insights > Logs tab
    • Set Timeframe: Previous Month
    • Click Add Filter, select Rule Name, and search for and select Default Firewall Filtering Rule
    • Optionally add a filter for Network Service = ICMP
    • Click Apply Filters and review the log entries showing ICMP traffic being Block/Drop’ed

Note: Rules like Office 365 One Click, Zscaler Proxy Traffic, and Recommended Firewall Rule explicitly allow very specific traffic needed for correct operation. ICMP and SSH traffic in this lab exercise will not match any of those rules and will fall to the Default rule (Block/Drop).

Task 5.5: Blocking QUIC Traffic as a Network Service

6 Verify QUIC Network Service and Firewall Rule

  1. Verify the QUIC network service definition:
    • Go to Administration > Network Services (in the Resources section)
    • Under the Services tab, search for QUIC
    • Verify QUIC is configured to match UDP Destination Port 443
  2. Check the firewall rule that blocks QUIC:
    • Go to Policy > Firewall Control and search on the Firewall Filtering Policy tab for the rule named Block QUIC
    • Verify the configuration: Action: Block/Reset, Criteria: Network Services = QUIC
    • Verify there are no preceding rules that would allow QUIC traffic to pass
  3. Click the View icon for the Block QUIC rule to see the full configuration and confirm:
    • Services tab: Network Services = QUIC
    • Action: Block/Reset
  4. Click Close.

Why block QUIC? QUIC is UDP-based (HTTP/3). When allowed, browsers bypass the Zscaler SSL inspection tunnel. Block/Reset forces the browser’s QUIC failsafe to engage, and the browser retries over HTTP/2 (TCP) — which Zscaler can inspect. Adding an explicit Block QUIC rule is recommended even when the Default rule blocks it, to ensure QUIC is always blocked regardless of future Default rule changes.

MENTAL NOTE
Six security layers: (1) Malware Protection — real-time AV scanning; (2) Advanced Threat Protection — C2, XSS, phishing, Page Risk; (3) Sandbox — file detonation with AI Quarantine; (4) Browser Isolation — render risky apps in a remote browser; (5) Firewall — Default Block/Drop catches all non-web protocols, Block QUIC forces HTTP/2 fallback for SSL inspection; (6) DNS Control — blocks at the DNS resolution level. QUIC must be explicitly blocked via Firewall (Block/Reset on UDP 443) to ensure SSL inspection can operate on all web traffic.

LAB 5 PROGRESS
QUICK REFERENCE ▲
Sandbox: Quarantine vs Allow-and-Scan
Quarantine holds file before delivery (safest). Allow and Scan delivers immediately and blocks on rescan if malicious (better UX, brief risk window).
Browser Isolation
Renders web content in a remote cloud browser. User sees a pixel stream — no code executes locally. Triggered by Cloud App Control or URL Filtering policies with Isolate action.
Block QUIC (UDP 443)
Forces browsers from HTTP/3 (UDP, uninspectable) to HTTP/2 (TCP, SSL-inspectable). Use Block/Reset action on the QUIC network service. Critical for complete SSL inspection coverage.
Default Firewall Block/Drop
Z-Tunnel 2.0 sends all traffic to the Zscaler Firewall. The catch-all Default rule ensures nothing passes unless explicitly allowed by a preceding rule.
DNS Control
Blocks/redirects DNS queries by category (Gambling, Malicious, etc.). Requires Z-Tunnel 2.0 for full DNS visibility. Logs visible in Analytics → Insights → DNS Insights.
Page Risk Index
Real-time page risk score in Advanced Threat Protection. Low Risk = block anything suspicious. High Risk = high tolerance. Setting at 33 means moderate risk sites are blocked.
