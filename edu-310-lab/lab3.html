LAB 3

Scenario: Barry Bogdown (Device: Barry-PC) reports periodic slowdowns when accessing SaaS apps. The issue comes and goes. Service Desk has escalated for immediate investigation.

Time: ~30 minutes
Environment: ZDX Admin Portal · User: Barry Bogdown · Device: Barry-PC · Time Range: Last 48 Hours
💡 THINK FIRST · DEVICE CPU ROOT CAUSE

Not every ZDX score drop is a network problem. Device health data — CPU, memory, disk I/O — is captured alongside network probes so you can distinguish between endpoint and infrastructure issues without visiting the device.

  • The ZDX Score for Barry Bogdown shows a repeating cycle of Poor, then Good, then Poor again. What is different about this pattern compared to a user whose score is always Poor?
  • Y-Engine reports Device CPU High at 100% confidence. Could a CPU-bound device affect cloud application performance even if the network path is healthy? Explain why.
  • Adobe Premiere Pro.exe is using 97% CPU. The process appears periodically. What remediation steps would you recommend to Barry’s manager, and how would you use ZDX to confirm the fix worked?
Q1 — MODEL ANSWER
A repeating cycle (Poor → Good → Poor) suggests an intermittent process that starts, peaks, and finishes periodically — a scheduled task, a rendering job, or a background sync. Always-poor scores more likely indicate a persistent condition: wrong DNS server, misconfigured Zscaler policy, or failing hardware.
Q2 — MODEL ANSWER
Yes. When the CPU is fully saturated, the operating system cannot schedule the ZCC client, browser processes, or application threads promptly. This causes Page Fetch Time to spike not because the network is slow, but because the local machine cannot process the response data fast enough. The Web Probe’s PFT measurement captures this end-to-end — it is not network-only.
Q3 — MODEL ANSWER
Recommend: (1) Close Adobe Premiere Pro when not actively editing, or schedule renders outside business hours. (2) Add the process to a blocklist or low-priority CPU scheduler policy via Intune. (3) After the fix is applied, check ZDX over the next 48 hours: the CPU Usage Device Health graph should no longer show 100% spikes, and the ZDX Score should remain consistently Good.

Task 3.2: Analyze Poor ZDX Score Interval

2 Identify the Root Cause with Y-Engine

  1. In the ZDX Score Over Time graph, zoom in on a period where the score first drops to Poor.
  2. Click on the point where the score drops to Poor and click Analyze Score.
  3. Review the factors table. Note the top factor and its confidence level.

Result: The top factor is Device CPU High — 100% confidence. User experience was impacted due to high CPU utilisation (99% or more sustained utilisation detected).

Y-Engine Analyze Score result: Device CPU High at 100% confidence, confirming sustained CPU utilisation as the root cause.

3 Confirm Impact via Web Probe Data

  1. Scroll down to the Web Probe Metrics section.
