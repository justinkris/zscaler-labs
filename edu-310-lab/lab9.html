Monitoring Self Service Notifications

ZDX can notify end users directly when their device has performance issues, guiding them to self-resolve without a helpdesk ticket. The Self Service Dashboard lets you track notification volume, user feedback, and drill into the health data behind each notification.

💡 THINK FIRST · SELF SERVICE NOTIFICATIONS

Self Service shifts the resolution model: instead of waiting for a user to file a ticket, ZDX notifies the user in real time via the Zscaler Client Connector tray app. The administrator role is to configure thresholds and monitor adoption, not to handle every case reactively.

  • ZDX sends CPU and Wi-Fi notifications directly to users via the Zscaler Client Connector tray app. What are two situations where this self-service approach would not work and you would still need helpdesk involvement?
  • The Self Service Dashboard shows "Users Found Notifications Helpful = 0" even with 187 notifications sent. What does this tell you about the notification configuration or user experience?
  • The Notification Type for one row is sel_pengine_download_fail. Without looking it up, what would you infer this notification type means, and what would you do to investigate?
Q1 — MODEL ANSWER
Self-service would not work when: (1) The user is not technical enough to follow the recommended actions (e.g. does not know how to close a high-CPU process). (2) The root cause requires administrative privileges that the user does not have (e.g. uninstalling a misbehaving application, updating a driver, or reconfiguring a network adapter). In these cases, the notification is a trigger for a helpdesk action, not a user self-resolution.
Q2 — MODEL ANSWER
Zero helpful feedback with 187 notifications sent suggests one of: (1) Users are not clicking "Found it Helpful" — the feedback mechanism is present but users are not engaging with it. (2) The notifications are not relevant or actionable to the users — they see the notification but don’t benefit from it. (3) The notification threshold is too sensitive, firing on transient spikes that users experience as false alarms. Review the Notification Type breakdown and consider adjusting thresholds.
Q3 — MODEL ANSWER
sel_pengine_download_fail suggests a failure in the ZDX probe engine (sel = Zscaler endpoint agent, pengine = probe engine, download_fail = download failure). To investigate: search for the affected user in the ZDX portal, check their Zscaler Client Connector version and connectivity status, and look for ZCC update or probe configuration failures in the Device Health or Zscaler row of User Device Events.

Lab 9: Monitoring Self Service Notifications

LAB 9

Scenario: Your organisation has enabled ZDX Self Service Notifications to proactively notify users of CPU and Wi-Fi issues. Review the Self Service Dashboard to understand notification volume, user feedback, and device health data behind notifications.

Time: ~20 minutes
Environment: ZDX Admin Portal · Dashboard › Self Service Dashboard

Task 9.1: View End User Self Service Demonstration

1 Watch the Self Service Animation

  1. Click the link to view the End User Self Service Demonstration animation (opens in a new browser tab).
  2. Note the following activities shown in the animation:
    • Notification of High Resource Consumption appears via the Zscaler application in the Windows tray.
    • User opens Zscaler Client Connector and views the Notifications page.
    • User clicks the Learn More link to see details and recommended actions.
    • User provides feedback on whether the recommendations were helpful.
  3. Close the tab and return to this lab guide.

Note: The animation is a GIF that loops. After watching, close the tab to continue.

Task 9.2: View Self Service Dashboard

2 Review Notification Statistics

  1. In the ZDX Admin Portal, click Dashboard › Self Service Dashboard.
  2. View the Total Notifications Sent count in the selected time range.
  3. Review the Notifications By Types (Wi-Fi, CPU, Other).
  4. Scroll through the Notifications Sent table to see individual user notifications.

Note: If no notifications are shown, select a longer time interval (e.g. All Hours).

Task 9.3: View User Feedback Summary

3 Review Feedback Statistics

  1. Review the Users Found Notifications Helpful count and percentage.

  3. Review Active Users With Self Service Enabled and Users Who Disabled Notifications counters.

Task 9.4: View User Device Health for a Notification

4 Drill Into a Notification Event

  1. In the Notifications Sent table, click the Username for a row where the Notification Type is System High CPU.
  2. Scroll up to view the Device Health section for that user.
  3. View the Top Processes by CPU Incidents chart.
  4. Click on the application with the highest indicated CPU usage to see a trend graph for that application.

Result: Clicking the notification event switches the view to a user report for the time period of the notification. The CPU Usage and Top Processes data confirms what triggered the notification.
