💭 THINK FIRST · ZDX ENVIRONMENT

The ZDX lab uses a shared tenant with pre-populated historical data — you are not generating live events but rather interpreting data that already exists. Understanding this distinction shapes how you read every graph and score in the labs.

  • All 14 labs use a single ZDX Admin Portal with pre-populated data and a Read-Only role. What are the advantages and limitations of using shared historical data rather than a live isolated tenant for a troubleshooting course?
    Q1 — MODEL ANSWER
    Advantages: Pre-populated data guarantees interesting incidents, score dips, Y-Engine findings, and Call Quality issues exist — no waiting for live events to occur. It also means all students see the same data, so lab instructions can point to specific users or timestamps with confidence. Limitation: you cannot trigger new ZCC enrollments, configure probes, or test policy changes — the Read-Only role prevents any writes. This mirrors a Tier 1 analyst role but limits hands-on configuration practice.
  • Lab 1 asks you to navigate all major ZDX dashboards before troubleshooting. Why does a structured orientation to the portal reduce mean-time-to-diagnose when a real user reports a problem?
    Q2 — MODEL ANSWER
    Orientation builds a mental map of where data lives before the pressure of a live incident. When a user reports slow performance, a ZDX operator who has already explored every dashboard can immediately decide: Is this fleet-wide (Incidents Overview) or user-specific (Users → drill-down)? Is it app-related (Applications) or device-related (Device Health)? Without prior orientation, analysts waste diagnostic time navigating unfamiliar menus under pressure.

Lab Description and Scenario

You are a ZDX Helpdesk Administrator at a Zero Trust Exchange demonstration organisation. Users across departments and geolocations report a variety of experience issues — poor Wi-Fi, CPU spikes, slow DNS, bad Teams call quality, and device startup problems. Your job is to investigate each scenario using ZDX data, identify the root cause, and document your findings using the same workflow a real ZDX operator would follow.

Labs 2–7 each focus on a specific user and problem type. Labs 8–10 cover proactive monitoring and sharing. Labs 11–14 introduce advanced capabilities including ZDX Copilot, Hosted Monitoring probes, Data Explorer, and the Wi-Fi Dashboard.

Lab Environment

This lab is entirely portal-based. You access the ZDX Admin Portal at https://admin.zdxcloud.net/ directly from your local browser. The tenant is pre-populated with historical data from real device telemetry — ZDX Score time series, Web Probe metrics, Cloud Path hops, User Device Events, Deep Tracing sessions, and Active Alerts are all available to explore.

Access Points

  • ZDX Admin Portal — https://admin.zdxcloud.net/. Your primary workspace for all 14 labs. Login: ptcb76-ADMIN@thezerotrustexchange.com with Session Password.
  • Access role: Read-Only Administrator. You can view all dashboards, drill into user and device data, run Y-Engine analysis, and explore all report sections — but cannot save policy changes, create probes, or configure alerts.
  • No Skytap VM required: Unlike EDU-200 and EDU-230, there is no Corp: Client PC VM in this lab. All tasks are performed in the ZDX Admin Portal from your laptop browser.

QUICK REFERENCE ▲
ZDX Portal URL
https://admin.zdxcloud.net/ — your primary workspace for all 14 labs.
ZDX Score
0–100 composite metric from Web Probe + Cloud Path data, normalised to per-user baseline. <50 = Poor, 50–79 = Okay, 80+ = Good.
Y-Engine
ZDX AI/ML engine — Analyze Score or Analyze Range on a score point to get ranked root-cause factors with confidence percentages.
Web Probe
Synthetic probe measuring Page Fetch Time, Server Response Time, DNS Time, and Availability from the user's device to the application.
Cloud Path
ICMP traceroute-style hop-by-hop latency from the client to the application server. Hop View shows per-hop detail; Command Line View shows traceroute output.
Deep Tracing
On-demand diagnostic session capturing detailed device health and network path data for a specific user and time window.
Incidents
ZDX-detected events affecting multiple users simultaneously. Distinguished from single-user performance issues by scope and epicenter mapping.
ZDX Snapshot
Shareable URL capturing a point-in-time view of a user's ZDX report for escalation or cross-team collaboration.
