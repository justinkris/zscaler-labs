Time: ~30 minutes
Environment: ZDX Admin Portal · User: Quincy Martin · Device: QUINCY-PC · Time Range: Last 48 Hours
💡 THINK FIRST · NETWORK HOP LATENCY

When both DNS Resolve Time and Server Response Time increase simultaneously, the bottleneck is not DNS or the application server — it’s somewhere on the shared network path. Cloud Path + Hop View turns that insight into a specific IP address.

  • In Lab 4, only DNS Resolve Time increased. In this lab, both DNS and Server Response Time increase. What does this difference in the PFT breakdown tell you about where the problem is?
  • The Hop View shows 496ms between 10.1.2.1 (gateway) and 10.0.0.254 (router). What are two possible explanations for high latency on this single hop?
  • ZDX Score Comparison confirms the issue. If you were writing a ticket to the network team, what three pieces of data from ZDX would you include to make the ticket actionable?
Q1 — MODEL ANSWER
If only DNS increases, the DNS server itself (or the path to it) is slow. If both DNS and Server Response Time increase proportionally, the shared upstream network path is introducing latency that affects all TCP connections equally — because both DNS queries and HTTPS requests must traverse the same bottleneck. This means a router, firewall, or ISP segment, not a specific service.
Q2 — MODEL ANSWER
Two explanations: (1) Congestion — the router at 10.0.0.254 is saturated (high utilisation) and packets are being queued, causing queuing delay. (2) Hardware or configuration fault — the router interface or the link between 10.1.2.1 and 10.0.0.254 has a hardware problem (damaged cable, half-duplex mismatch, failing NIC) that introduces processing delay or retransmissions.
Q3 — MODEL ANSWER
Include: (1) The specific hop IP addresses: gateway 10.1.2.1 to router 10.0.0.254. (2) The latency measured: 496ms at peak vs. 1ms at baseline — a 496× increase. (3) The time range when the issue was observed (from the ZDX timestamp). Optionally attach the ZDX Score Comparison screenshot showing the Before vs. After hop latency side by side.

Task 5.3: Identify High Latency Cloud Path Link

3 Locate the Bottleneck Link

  1. Scroll down to the Cloud Path section.
  2. Select a point on the graph when End-End latency is high.
  3. Examine the latency metrics for each leg:
    • Client → Egress
    • ZIA Public Service Edge → Application
    • End-End total

Result: Client-Egress latency is 496ms out of a total End-End latency of 551ms — the local network path from device to internet egress is responsible for 90% of the total delay.

Cloud Path latency breakdown: Client-Egress = 496ms, ZIA PSE-App = 30ms, End-End = 551ms. The local leg is the bottleneck.

Task 5.4: Analyze High Hop Latency

4 Pinpoint the Specific Router Hop

  1. Scroll to the Hop View below the Cloud Path graph.
  2. Locate the high-latency link (shown in orange).
  3. Click the magnifying glass icon on the high-latency hop to expand the view.
  4. Note the IP addresses of the two endpoints of the high-latency link.

Result: The 496ms latency is isolated to the hop between the Client Gateway at 10.1.2.1 and the next router at 10.0.0.254. This is a specific network segment, not the device itself.

Hop View showing the expanded Client-Egress segment. The 496ms latency is between gateway 10.1.2.1 and router 10.0.0.254.
