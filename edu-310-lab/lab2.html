Environment: ZDX Admin Portal · User: Curtis Hardwick · Device: PC-Curtis-WiFi · Time Range: Last 48 Hours
💡 THINK FIRST · WIFI TROUBLESHOOTING

Poor Wifi is the most common ZDX root cause. The challenge is that users experience it as "slow internet" without realising their device has silently roamed to a weaker access point.

  • The ZDX Score fluctuates between Good and Poor every few hours. What does this oscillation pattern suggest about the root cause compared to a score that is consistently Poor?
    Q1 — MODEL ANSWER
    Oscillation between Good and Poor suggests a transient, repeating condition — most likely the device roaming between two access points. A consistently Poor score suggests a persistent underlying problem: a misconfigured DNS server, a failing network component, or a broken Zscaler tunnel.
  • The Y-Engine lists Suboptimal Wifi at 8.33% confidence alongside Low Wifi Signal at 91.67%. Should you act on a low-confidence factor? Under what circumstances?
    Q2 — MODEL ANSWER
    Low-confidence factors should be noted, not acted on immediately. In this case, Suboptimal Wifi names the specific SSID and channel — that detail is actionable. Cross-reference it with the User Device Events SSID Change at the same timestamp. If they align, the low-confidence factor is corroborated.
  • If the Client-Egress latency is 319ms but the Wifi signal strength looks fine in the Device Events data, what else would you check to explain the latency?
    Q3 — MODEL ANSWER
    Check the Hop View Command Line View for packet loss between the client gateway and the next hop. Also check the Wi-Fi Channel field in User Device Events — channel congestion (multiple devices on the same channel) increases latency even when signal strength is adequate. Additionally, confirm via Cloud Path whether the latency spike appears only on the Client-Egress leg or propagates further.

Task 2.2: Analyze Poor ZDX Score Interval

2 Use Y-Engine to Identify Root Cause Factors

  1. In the ZDX Score Over Time graph, click and drag to zoom in on a period including a Poor score.
  2. Click on a data point with a Poor ZDX Score.
  3. Click Analyze Score and wait a few seconds for the analysis to complete.
  4. Review the The following factors that might have impacted the ZDX score table. Note the top factor and its Confidence Level.

Result: In this example, the top factor is Low Wifi Signal (91.67% confidence) followed by Suboptimal Wifi (SSID: 3Com_Wifi, channel 1, type 802.11n).

ZDX Score Over Time zoomed in showing the Poor score interval. The Analyze Score button appears when a point is selected.
Y-Engine analysis results: Low Wifi Signal at 91.67% confidence and Suboptimal Wifi at 8.33% confidence.
Troubleshoot — Y-Engine shows no root cause or "Insufficient data"

Y-Engine analysis requires at least one complete probe cycle in the selected interval. If you see "Insufficient data" or an empty factors table: (1) Try zooming in on a longer Poor-score period — a single data point may not have enough probe history. (2) Select a different application — some applications may have fewer probes configured. (3) Verify the time range includes at least 15–30 minutes of Poor score (not just a single point). Since this is a shared read-only environment, data is pre-populated — your view may differ slightly from the screenshots.

Task 2.4: Isolate High Latency Hop

4 Locate the High Latency Link in the Cloud Path

  1. Click the ← Compare ZDX Scores link to return to the user report.
  2. Zoom in on a period containing a Poor ZDX Score in the graph.
  3. Scroll down to view the Cloud Path section and note which leg has the highest latency.
  4. Examine the Hop View below the graph. The highest-latency link is highlighted in orange (differential).

Result: In this example, the Client-Egress latency is 319ms — the hop between the client device and its local gateway accounts for most of the end-to-end delay, confirming a local Wifi problem.

Cloud Path Hop View showing high Client-Egress latency of 319ms between the device and its local gateway.

Task 2.5: View User Device Events Associated With ZDX Score Changes

5 Correlate SSID Change to Score Transition

  1. Scroll to the bottom of the user report to view the User Device Events timeline plots.
  2. Mouse-over the dot on the Network timeline at the time when the ZDX Score transitioned to Poor.
  3. Scroll through the Network Interface data in the popup to find the event with a value in the New Value field.
  4. Mouse-over the dot corresponding to when the score transitioned back to Good and note the SSID change.

Result: The SSID Change from Google_Nest to 3Com_Wifi is associated with the Poor score. Switching back to Google_Nest restores Good performance.

User Device Events Network timeline popup showing SSID Change from Google_Nest to 3Com_Wifi at the Poor score transition.
User Device Events showing the return SSID Change from 3Com_Wifi back to Google_Nest, corresponding to the Good score transition.

QUICK REFERENCE ▲
Analyze Score
Y-Engine function that lists the most likely factors behind a ZDX score drop at a selected time point, with confidence percentages.
Compare to › Last Known Good
Comparison tool that shows side-by-side ZDX data for a Poor score point vs. the most recent Good score, highlighting Key Differences.
Client-Egress Latency
Cloud Path leg measuring latency from the client device to the internet egress point. High values indicate a local network or Wifi problem.
User Device Events
Timeline of network, hardware, software, and Zscaler events captured on the endpoint. SSID Change events appear on the Network row.
SSID Change
User Device Event recorded when the device connects to a different Wifi access point. The New Value field shows the new SSID.
