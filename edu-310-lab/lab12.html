Lab 12: ZDX Hosted Monitoring

LAB 12

Scenario: Your organisation wants to monitor key SaaS applications from Zscaler data centers worldwide, independent of end-user devices. Configure and review Hosted Monitoring probe data.

Time: ~20 minutes
Environment: ZDX Admin Portal · Configuration › Zscaler Hosted Probes → Analytics › Zscaler Hosted Monitoring
💡 THINK FIRST · HOSTED PROBE MONITORING

End-user ZDX data tells you what users experience. Hosted probes tell you what the application looks like from the cloud — without any user involved. If Hosted probes show a problem but end-user ZDX scores are good, the issue is specific to that user’s location or device.

  • End-user ZDX scores are based on measurements from the Zscaler Client Connector on a user’s device. Hosted Monitoring probes measure from Zscaler data centers. If Hosted Monitoring shows an application is slow but end-user ZDX scores are fine, what does this tell you about where the problem is?
  • A Hosted Probe can be either a Web Probe or a Cloud Path Probe. What is the key difference in what they measure, and when would you choose each type?
  • The Probe Run Details page shows Redirect Time, DNS, TCP, SSL Handshake, Server Response, and Page Fetch Time in milliseconds. If Server Response Time is 1142ms but DNS is only 100ms, where would you focus your investigation?
Q1 — MODEL ANSWER
If Hosted Monitoring shows slowness but end-user ZDX scores are fine, the issue is infrastructure-level but not affecting users at that moment — possibly a Zscaler data center-specific performance issue that users are routing around, or a probe-specific configuration problem (wrong target URL, wrong expected response code). Conversely, if end-user scores are Poor but Hosted Monitoring looks fine, the problem is between the user’s device and the nearest Zscaler data center, not in the Zscaler or application infrastructure.
Q2 — MODEL ANSWER
Web Probe measures HTTP/HTTPS layer metrics: Page Fetch Time, Server Response Time, DNS Time, Availability, SSL Handshake time. Use it for web applications where you care about full-page load time. Cloud Path Probe measures network-layer hop-by-hop ICMP latency: End-to-End Latency, Packet Loss, Jitter. Use it for non-HTTP services, network infrastructure monitoring, or when you want to identify a specific router or ISP segment causing latency.
Q3 — MODEL ANSWER
Focus on the application server or upstream infrastructure. Server Response Time of 1142ms means the server took over 1 second to return the first byte after the TCP connection was established and the HTTP request was sent — this is an application performance problem, not a network problem. DNS at 100ms is somewhat elevated but not the primary concern. Check the application server logs, CDN configuration, or backend database performance.

Task 12.1: View the Configuration of Hosted Monitoring

1 Browse Hosted Probe Collections

  1. Navigate to Configuration › Zscaler Hosted Probes.
  2. Review the Collections in the left panel: Network, Websites, Apps, API, SLA.
  3. Click on each collection to see the probes it contains (Type, Name, Host, Status, Protocol, Locations, Alert Rules, Frequency).
  4. Click the view icon on any probe to open its configuration wizard.
  5. Navigate through the wizard steps: Select a Probe Type (Web or Cloud Path), Configure, Additional Parameters (Protocol, Run Frequency, Hop Count, Interval, Zscaler Hosted Locations), Add Alerts.
  6. Click Cancel to close without saving.

Task 12.2: View the Hosted Monitoring Reports

2 Analyse Hosted Probe Metrics

  1. Navigate to Analytics › Zscaler Hosted Monitoring.
  2. In the left panel, browse the probe list by collection (Network, SLA, Apps, API, Websites).
  3. Select a Web Probe (e.g. Box (Web) from the SLA collection) and view the Availability and Page Fetch Time graphs (30 Days).
  4. Click and drag on the Availability graph to zoom in on a time range.
  5. Use the dropdown on the PFT graph to add additional metrics (e.g. DNS).
  6. Switch to the scatter graph view. Click a data point and click View Details to see the Probe Run Details page with key metrics (Redirect Time, DNS, TCP, SSL Handshake, Server Response, Page Fetch Time).
  7. Use the Zscaler Hosted Locations dropdown on the Probe Run Details page to compare results from different data centers.
  8. Navigate back to Hosted Monitoring and select a Network Probe (e.g. Box (Network)). View the End-to-End Latency and add Packet Loss from the dropdown.

MENTAL NOTE
Hosted Probes = synthetic tests from Zscaler DCs, not user devices. Collections: Network, SLA, Apps, API, Websites. Probe types: Web (PFT, DNS, SSL, availability) or Cloud Path (E2E latency, packet loss). Probe Run Details: Redirect + DNS + TCP + SSL + Server Response + PFT in ms. Key insight: Hosted poor + end-user fine = Zscaler DC/routing issue. Hosted fine + end-user poor = user-side problem.

QUICK REFERENCE ▲
Zscaler Hosted Probes
Synthetic monitoring probes run from Zscaler data centers. Measure application performance independently of end-user devices.
Collections
Logical groups of Hosted Probes: Network, Websites, Apps, API, SLA. Each collection can contain multiple Web or Cloud Path probes.
Web Probe
HTTP/HTTPS probe measuring Page Fetch Time, Server Response Time, DNS Time, Availability. Used for web application monitoring.
Cloud Path Probe
ICMP probe measuring End-to-End Latency, Packet Loss, and Jitter. Used for network infrastructure and non-HTTP service monitoring.
Probe Run Details
Per-data-point breakdown page showing Redirect Time, DNS, TCP, SSL Handshake, Server Response, and Page Fetch Time with per-location comparison.
Zscaler Hosted Locations
Zscaler data centers from which probes are run: Washington DC, San Jose, Amsterdam, Zurich, Frankfurt, Chicago, and others.
