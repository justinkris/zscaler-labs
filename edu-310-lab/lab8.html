Lab 8: Incident Monitoring on Incident Dashboard

LAB 8

Scenario: Operations staff need to proactively monitor for incidents affecting multiple users. Use the ZDX Incidents Dashboard to identify and drill into a Wi-Fi incident.

Time: ~15 minutes
Environment: ZDX Admin Portal · Dashboard › Incidents Dashboard
💡 THINK FIRST · INCIDENT DETECTION

Individual user complaints are reactive. The Incidents Dashboard is proactive: ZDX AI/ML correlates signals from many devices to detect an incident’s epicenter — a single Wi-Fi access point, an ISP segment, or a Zscaler data center — before the issue has been widely reported.

  • The Incidents Dashboard shows incidents across 7 key areas: Device, DNS, Wi-Fi, Last Mile ISP, Intermediate SP, Zscaler, and Application. For each area, give one example of a real-world cause that would trigger an incident in that category.
  • An Incident has a ZDX Score Drop metric and an Average Wi-Fi Access Point Latency metric. Why are both needed to characterise a Wi-Fi incident? What does each one tell you that the other doesn’t?
  • An incident shows 4 Impacted Users from 1 Geolocation. The SSID is 3com_wifi. What would you tell the on-site IT team to investigate, and what information from the Incident Details page would you include in your message?
Q1 — MODEL ANSWER
Device: a RAM failure or overheating causing CPU throttling. DNS: a corporate DNS server misconfiguration or overload. Wi-Fi: a failing access point or a rogue SSID with the same name as a corporate AP. Last Mile ISP: an ISP outage or BGP route withdrawal. Intermediate SP: a backbone carrier peering issue. Zscaler: a Zscaler data center degradation affecting egress traffic. Application: a SaaS service outage (e.g. a Salesforce API endpoint going down).
Q2 — MODEL ANSWER
ZDX Score Drop shows the user experience impact — whether users were actually affected and how severely. AP Latency shows the technical measurement — the actual roundtrip time between client and access point. You need both because high AP latency may not always cause a ZDX score drop (e.g. if the probe retry masks it), and a ZDX score drop may have multiple contributing factors. Together they confirm both the cause and the user experience impact.
Q3 — MODEL ANSWER
Tell the on-site team: "There is a Wi-Fi incident on SSID 3com_wifi affecting 4 users in [Geolocation]. The incident started at [Start Time from Incident Details] and has lasted [Duration]. Key Metrics show a ZDX Score Drop and elevated Wi-Fi Access Point Latency. Please check the AP associated with SSID 3com_wifi for hardware faults, channel interference, or client association issues."

Task 8.1: View Incidents Dashboard

1 Open the Incidents Dashboard

  1. In the ZDX Admin Portal, click on Dashboard › Incidents Dashboard.
  2. Note the number of Wi-Fi Incidents reported in the selected time range (14 Days).
  3. Review the Incidents Across Key Areas icons (Device, DNS, Wi-Fi, Last Mile ISP, Intermediate SP, Zscaler, Application).

Task 8.2: Identify Incidents Epicenters

2 View the Incidents by Epicenters Map

  1. Scroll down on the Incidents Dashboard to view the Incidents by Epicenters map.
  2. Note the location of the incidents markers showing epicenters and the number of incidents indicated.
  3. Click on an incidents marker to see an expanded view of the types of incidents at that epicenter.
  4. Click View Incident Details to open the incident detail page.

Task 8.3: View Incident Details

3 Drill Into the Incident

  1. On the Incident Details page, review:

Key Metric showing the average and maximum roundtrip latency measured between client devices and the Wi-Fi Access Point.
